# Synchronicity / Asynchrony

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> **Почему это важно для аналитика:** Выбор между синхронным и асинхронным взаимодействием — это фундаментальное архитектурное решение. Понимание trade-offs между двумя подходами критично при проектировании систем и выборе технологий интеграции.

## Основные определения

### Синхронное взаимодействие (Synchronous)
Отправитель **ждет** завершения операции перед тем, как продолжить свою работу.

```
Отправитель → [ждет] → Получатель обрабатывает → Ответ → Отправитель продолжает
                        ↑                                    ↑
                    Заблокирован                        Разблокирован
```

### Асинхронное взаимодействие (Asynchronous)
Отправитель **не ждет** завершения операции и продолжает свою работу.

```
Отправитель → Отправляет сообщение → Продолжает работу
                    ↓
            Message Broker / Queue
                    ↓
            Получатель обрабатывает позже
```

## Синхронное взаимодействие

### Характеристики

| Аспект | Описание |
|--------|---------|
| **Блокировка** | Отправитель блокируется до получения ответа |
| **Временная связь** | Отправитель и получатель должны быть активны одновременно |
| **Простота** | Легко понять и отладить (как обычный вызов функции) |
| **Задержка** | Зависит от скорости получателя |

### Примеры синхронного взаимодействия

```
1. HTTP запрос к REST API
   GET /api/users/123 → [ждет ответа] → JSON { user }

2. Вызов функции/метода
   result = getUserData(123) → [ждет] → return { user }

3. Синхронный RPC (Remote Procedure Call)
   response = callRemoteFunction(args) → [ждет] → return result

4. Синхронный вызов базы данных
   user = db.query("SELECT * FROM users WHERE id=123")
```

### Когда использовать синхронное взаимодействие?

✅ **Используйте синхронное когда:**
- Нужен немедленный результат
- Операция быстрая (< 1 сек)
- Отправитель не может продолжить без результата
- Требуется гарантирование успеха операции
- Простота важнее масштабируемости

❌ **Избегайте синхронного когда:**
- Операция может быть долгой (> 10 сек)
- Хотите масштабировать систему
- Нужна отказоустойчивость к отказам получателя

### Проблемы синхронного взаимодействия

```
Пример: Заказ на e-commerce

1. Пользователь нажимает "Купить"
2. API блокируется и:
   - Проверяет платеж (может занять 5 сек)
   - Обновляет склад (может занять 3 сек)
   - Отправляет email (может занять 2 сек)
   - Создает уведомление (может занять 1 сек)
   Итого: 11 секунд ожидания!

3. Если какой-то сервис падает → вся операция падает

Масштабируемость:
- 100 одновременных пользователей = 100 блокированных потоков
- 1000 пользователей = 1000 потоков (исчерпание ресурсов)
```

## Асинхронное взаимодействие

### Характеристики

| Аспект | Описание |
|--------|---------|
| **Блокировка** | Отправитель не блокируется |
| **Временная связь** | Отправитель и получатель независимы |
| **Сложность** | Сложнее в реализации и отладке |
| **Масштабируемость** | Высокая |

### Примеры асинхронного взаимодействия

```
1. Асинхронный HTTP (Callback)
   request('GET /api/users/123', (err, data) => {
     console.log(data); // Обработка ответа
   });
   // Продолжаем работу, не ждем ответа

2. Promise / Async-Await
   const user = await fetchUser(123);
   // Выглядит синхронным, но асинхронно

3. Message Queue / Event System
   producer.send('create_order', order);
   // Не ждем результата
   // Consumer обрабатывает позже

4. WebSocket / Real-time Events
   socket.on('message', (data) => {
     console.log('New message:', data);
   });
   // Реагируем на события когда они приходят
```

### Когда использовать асинхронное взаимодействие?

✅ **Используйте асинхронное когда:**
- Операция может быть долгой
- Результат не нужен немедленно
- Нужна высокая масштабируемость
- Требуется отказоустойчивость
- Есть очереди или Event Brokers

❌ **Избегайте асинхронного когда:**
- Нужен результат немедленно
- Простота важнее масштабируемости
- Сложность не оправдана

### Решение проблемы синхронного взаимодействия

```
До (синхронно):
Пользователь → API → [ждет 11 сек] → Платеж → Склад → Email → Уведомление

После (асинхронно):
Пользователь → API → (200 OK за 100ms)
                          ↓
                    Message Queue
                     ↙  ↓  ↘  ↘
                Платеж Склад Email Уведомление (параллельно, позже)

Результат:
- Пользователь видит ответ за 100ms
- Все операции выполняются параллельно в фоне
- Если Email сервис упал, остальное работает
- 10000 пользователей = нет исчерпания потоков
```

## Сравнение Синхронного и Асинхронного

| Критерий | Синхронное | Асинхронное |
|----------|-----------|------------|
| **Задержка для пользователя** | Высокая | Низкая |
| **Простота** | Простое | Сложное |
| **Масштабируемость** | Плохая | Отличная |
| **Отказоустойчивость** | Хрупкое | Надежное |
| **Отладка** | Легко | Сложно |
| **Использование ресурсов** | Неэффективно | Эффективно |
| **Гарантия результата** | Высокая | Требует обработки ошибок |

## Практический пример: E-commerce

### Синхронный подход (проблемный)
```
POST /api/orders
{
  "user_id": 123,
  "items": [...]
}

Сервер выполняет:
1. Проверка платежа → 3сек
2. Обновление склада → 2сек
3. Отправка email → 5сек (медленный сервис!)
4. Создание уведомления → 1сек

Результат: Пользователь ждет 11 секунд
Проблема: Если email падает, весь заказ падает
```

### Асинхронный подход (правильный)
```
POST /api/orders
{
  "user_id": 123,
  "items": [...]
}

Сервер:
1. Сохраняет заказ в БД → 100ms
2. Отправляет сообщение "order_created" в RabbitMQ → 50ms
3. Возвращает 202 Accepted

Параллельно (в фоне):
- Worker1 обрабатывает платеж
- Worker2 обновляет склад
- Worker3 отправляет email
- Worker4 создает уведомление

Результат: Пользователь видит ответ за 150ms
Преимущества:
- Все операции выполняются параллельно
- Если email упал, заказ все равно создан
- Легко масштабировать workers
```

## Паттерны асинхронного взаимодействия

### 1. Fire-and-Forget
Отправить сообщение и забыть о нем (не важен результат):

```
producer.send('event', data);
// Не проверяем, успешно ли обработано
```

### 2. Request-Reply с timeout
Отправить запрос, ждать ответ с таймаутом:

```
future = producer.send('request', data);
try {
  result = future.get(timeout=5000);
} catch (TimeoutException) {
  // Обработать таймаут
}
```

### 3. Callback
Передать функцию обратного вызова:

```
request('GET /api/data', (error, data) => {
  if (error) handleError(error);
  else processData(data);
});
```

### 4. Event-Driven
Слушать события и реагировать:

```
eventBus.on('order_created', (order) => {
  processOrder(order);
});
```

## Best Practices

1. **Гибридный подход** — используйте оба подхода в зависимости от ситуации
2. **Timeout** — всегда устанавливайте timeout для синхронных операций
3. **Error handling** — асинхронное требует обработки ошибок в фоне
4. **Monitoring** — отслеживайте асинхронные операции в фоне
5. **Idempotency** — асинхронные операции должны быть идемпотентны (безопасны при повторе)
