# Authentication

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> **Почему это важно для аналитика:** Аутентификация и авторизация — это критические компоненты безопасности каждой системы. Понимание различных методов необходимо при проектировании API, анализе требований безопасности и интеграции сторонних сервисов.

## Основные понятия

Важно различать три связанных, но разных процесса:

### 1. Идентификация
**Определение "кто ты есть"** — предоставление идентификационных данных (логин, email, ID).

Пример: пользователь вводит логин "john_doe"

### 2. Аутентификация
**Подтверждение подлинности** — проверка того, что ты действительно тот, кем себя выдаешь.

Пример: проверка пароля, отпечатка пальца, одноразового кода

### 3. Авторизация
**Определение "что ты можешь делать"** — предоставление прав доступа к определенным ресурсам.

Пример: администратор может удалять пользователей, обычный пользователь — нет

## Процесс аутентификации

```
1. Идентификация → 2. Аутентификация → 3. Сеанс → 4. Авторизация
   (вводит логин)   (вводит пароль)   (получает токен) (проверка прав)
```

## Основные методы аутентификации

### 1. Basic Authentication
Отправка логина/пароля в каждом запросе в заголовке:

```
GET /api/users HTTP/1.1
Authorization: Basic dXNlcm5hbWU6cGFzc3dvcmQ=
```

**Плюсы:** Простой, встроенный в HTTP
**Минусы:** Небезопасен без HTTPS, пароль в каждом запросе

### 2. Session-based Authentication
Сервер создает сеанс (session) и отправляет ID в cookies:

```
1. POST /login → { username, password }
2. Сервер создает session и отправляет в cookies
3. Клиент автоматически отправляет cookies в каждом запросе
4. Сервер проверяет session ID по каждому запросу
```

**Плюсы:** Знаком большинству разработчиков
**Минусы:** Требует хранилища сеансов, сложнее для микросервисов

### 3. Token-based Authentication (JWT)
Сервер выдает токен, клиент отправляет его в заголовке:

```
1. POST /login → { username, password }
2. Сервер создает JWT токен и возвращает его
3. Клиент отправляет токен в заголовке:
   GET /api/users
   Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

4. Сервер проверяет подпись токена (не требует БД)
```

Структура JWT:
```
header.payload.signature

header:    {"alg": "HS256", "typ": "JWT"}
payload:   {"user_id": 123, "username": "john", "exp": 1234567890}
signature: HMACSHA256(header.payload, secret)
```

**Плюсы:** Stateless, хорошо для микросервисов, масштабируем
**Минусы:** Сложнее отозвать, нужна защита от XSS

### 4. OAuth 2.0
Делегированная аутентификация через третью сторону (Google, GitHub и т.д.):

```
1. Пользователь нажимает "Sign in with Google"
2. Редирект на Google OAuth сервер
3. Пользователь вводит пароль в Google
4. Google возвращает authorization code
5. Ваше приложение обменивает code на access token
6. Приложение получает информацию пользователя
7. Создается локальный токен/сеанс
```

**Плюсы:** Безопаснее (пароль не у вас), удобно для пользователя
**Минусы:** Сложнее реализовать, зависит от провайдера

### 5. Single Sign-On (SSO)
Одна аутентификация для множества приложений:

```
Пользователь:
App1 (требует auth) → SSO сервер → (вводит пароль один раз)
App2 (требует auth) → SSO сервер → (уже authenticated)
App3 (требует auth) → SSO сервер → (уже authenticated)
```

**Примеры:** Active Directory, Okta, Auth0
**Плюсы:** Удобно для корпоративных сред
**Минусы:** Сложно настраивать, требует центрального сервера

## Сравнение методов

| Метод | Безопасность | Сложность | Масштабируемость | Использование |
|-------|--------------|-----------|------------------|---------------|
| **Basic** | Низкая | Очень низкая | Плохая | Внутренние API |
| **Session** | Средняя | Низкая | Средняя | Веб-приложения |
| **JWT** | Высокая | Средняя | Отличная | REST API, микросервисы |
| **OAuth 2.0** | Высокая | Высокая | Отличная | Публичные API, социальные |
| **SSO** | Очень высокая | Очень высокая | Хорошая | Корпоративные системы |

## Практический пример: JWT Authentication

```javascript
// Клиент
POST /api/login { username: "john", password: "secret" }

// Сервер возвращает
{
  "access_token": "eyJhbGciOiJIUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 3600
}

// Клиент сохраняет токен и отправляет в каждом запросе
GET /api/users
Authorization: Bearer eyJhbGciOiJIUzI1NiIs...

// Сервер проверяет токен
const decoded = jwt.verify(token, SECRET);
// Если валиден → обрабатывает запрос
// Если невалиден → возвращает 401 Unauthorized
```

## Когда использовать какой метод?

| Сценарий | Рекомендуется | Причина |
|----------|--------------|---------|
| Внутренние API, микросервисы | JWT | Stateless, масштабируемо |
| Публичное веб-API | JWT или OAuth 2.0 | Безопасность, гибкость |
| Веб-приложение (SPA) | JWT или сессии | Эффективно для браузера |
| Мобильное приложение | JWT или OAuth 2.0 | Работает offline |
| Интеграция с Google/GitHub | OAuth 2.0 | Встроенная поддержка |
| Корпоративная система | SSO (LDAP/AD) | Централизованное управление |

## Best Practices

1. **Всегда используйте HTTPS** — защитите передачу токенов
2. **Храните токены безопасно** — localStorage для браузера (не идеально), secure cookies
3. **Устанавливайте expiration** — токены должны истекать (1 час)
4. **Используйте refresh tokens** — для обновления доступа без повторной аутентификации
5. **Логируйте попытки входа** — отслеживайте подозрительную активность
6. **Двухфакторная аутентификация (2FA)** — для критических операций
7. **Защита от CSRF** — если используете cookies

## Инструменты и сервисы

- **Auth0** — полноценный сервис аутентификации
- **Firebase Auth** — облачная аутентификация от Google
- **Okta** — корпоративное управление идентификацией
- **Passport.js** — Node.js библиотека для аутентификации
- **json-web-token (jwt)** — библиотеки для работы с JWT
