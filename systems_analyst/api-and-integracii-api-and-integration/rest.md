> **Почему это важно для аналитика:** REST — это де-факто стандарт для веб-API. Понимание его принципов необходимо для проектирования интеграций, документирования требований и коммуникации с разработчиками.

## Что такое REST?

**REST** (Representational State Transfer) — это архитектурный стиль, определяющий принципы построения распределенных систем и веб-сервисов через HTTP. REST не является протоколом, а набором идей о том, как должна устроена коммуникация в веб.

## Основные принципы REST

### 1. Клиент-серверная архитектура
Четкое разделение между клиентом (инициирует запросы) и сервером (предоставляет ресурсы). Это позволяет независимо развивать обе части.

### 2. Без состояния (Stateless)
Каждый запрос содержит всю необходимую информацию. Сервер не хранит контекст клиента между запросами, что упрощает масштабирование.

### 3. Ресурсная модель
Данные представлены как ресурсы, каждый с уникальным идентификатором (URL). Например:
- `/api/users` — коллекция пользователей
- `/api/users/123` — конкретный пользователь
- `/api/users/123/orders` — заказы пользователя 123

### 4. Методы HTTP
- **GET** — получение данных (безопасная, идемпотентная)
- **POST** — создание новых ресурсов
- **PUT** — полное обновление ресурса
- **PATCH** — частичное обновление ресурса
- **DELETE** — удаление ресурса

### 5. Кэширование
REST поддерживает кэширование через HTTP-заголовки, улучшая производительность и снижая нагрузку.

### 6. Единообразный интерфейс
Стандартные способы работы с ресурсами упрощают понимание API.

## Пример REST API

```
GET /api/v1/users/123 HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer token_123

Ответ:
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 123,
  "name": "John Doe",
  "email": "john@example.com",
  "created_at": "2024-01-15T10:30:00Z"
}
```

## REST vs SOAP

| Аспект | REST | SOAP |
|--------|------|------|
| **Тип** | Архитектурный стиль | Протокол |
| **Формат данных** | JSON, XML | XML |
| **Сложность** | Простой, легкий | Сложный, формализованный |
| **Производительность** | Лучше (меньше данных) | Хуже (больше overhead) |
| **Протоколы** | HTTP | HTTP, SMTP, TCP |
| **Состояние** | Stateless | Может быть stateful |
| **Использование** | Современные API | Legacy-системы |

## Когда использовать REST?

- Публичные веб-API
- Мобильные приложения
- Интеграция веб-сервисов
- Микросервисная архитектура
- CRUD операции над ресурсами

## Когда выбрать альтернативу?

- **gRPC** — для высоконагруженных микросервисов
- **GraphQL** — для сложных запросов с множественными связями
- **WebSocket** — для real-time коммуникации
- **SOAP** — для интеграции с legacy-системами (редко)

## Best Practices

1. **Версионирование API** (`/api/v1/`, `/api/v2/`)
2. **Логичная структура URL** (существительные, не глаголы)
3. **Правильные HTTP коды** (200 OK, 201 Created, 400 Bad Request и т.д.)
4. **Документирование** через OpenAPI/Swagger
5. **Пагинация** для коллекций (`?limit=10&offset=20`)
6. **Фильтрация и сортировка** (`?status=active&sort_by=created_at`)
