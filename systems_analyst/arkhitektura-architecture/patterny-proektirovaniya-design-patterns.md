> Паттерны проектирования - это повторяемые, проверенные архитектурные решения, которые решают часто встречающиеся проблемы при разработке систем.

## Что такое паттерны проектирования?

Паттерны проектирования - это описание проблемы и решения, которое может быть применено в различных сценариях. Паттерн включает:
- **Название** и описание проблемы
- **Контекст**, в котором применяется паттерн
- **Решение** с описанием структуры и взаимодействия компонентов
- **Последствия** применения паттерна (плюсы и минусы)
- **Примеры** из реальных систем

Паттерны обеспечивают общий язык между разработчиками и позволяют передавать идеи без детального объяснения.

## Три категории паттернов

### 1. Порождающие паттерны (Creational Patterns)

Решают проблемы создания объектов гибким способом.

**Singleton** - гарантирует только один экземпляр класса в приложении.
- Пример: logger, database connection pool, configuration manager
- Используется для: глобального доступа к ресурсам

**Factory** - создает объекты без указания их конкретных классов.
- Пример: DatabaseFactory может создать MySQL, PostgreSQL или MongoDB драйвер
- Используется для: скрытия сложности создания объектов

**Abstract Factory** - создает семейства связанных объектов.
- Пример: UI Theme Factory (Light и Dark режимы)
- Используется для: создание объектов по темам или конфигурациям

**Builder** - конструирует сложные объекты пошагово.
- Пример: StringBuilder для создания строк, Query Builder для SQL
- Используется для: создание объектов с множеством параметров

**Prototype** - создает новые объекты путем копирования существующего объекта.
- Пример: clone() метод
- Используется для: быстрого создания похожих объектов

### 2. Структурные паттерны (Structural Patterns)

Решают проблемы組компоновки объектов в структуры.

**Adapter** - делает несовместимый интерфейс совместимым.
- Пример: адаптер для подключения старого кода к новому API
- Используется для: интеграция разных компонентов

**Decorator** - добавляет функциональность к объектам динамически.
- Пример: добавление логирования, кэширования к функции
- Используется для: расширение функциональности без изменения исходного кода

**Facade** - предоставляет упрощенный интерфейс к сложной системе.
- Пример: платежный шлюз обеспечивает один API для разных платежных систем
- Используется для: упрощение использования сложных компонентов

**Proxy** - контролирует доступ к объекту, представляя его.
- Пример: lazy loading (загрузить данные только когда они нужны)
- Используется для: контроль доступа, кэширование, логирование

**Composite** - рассматривает иерархию объектов единообразно.
- Пример: файловая система (файлы и папки)
- Используется для: иерархические структуры

### 3. Поведенческие паттерны (Behavioral Patterns)

Решают проблемы взаимодействия между объектами и распределения ответственности.

**Observer** - уведомляет множество объектов об изменении состояния.
- Пример: event listeners в браузере, publish-subscribe в кафке
- Используется для: реактивные системы, event-driven архитектуры

**Strategy** - инкапсулирует алгоритмы в отдельные классы.
- Пример: разные алгоритмы сортировки, способы оплаты
- Используется для: выбор алгоритма во время выполнения

**Command** - инкапсулирует запрос как объект.
- Пример: Undo/Redo функциональность
- Используется для: отложенное выполнение, очереди команд

**State** - позволяет объекту изменять поведение в зависимости от состояния.
- Пример: заказ (новый, оплачен, отправлен, доставлен)
- Используется для: конечные автоматы

**Iterator** - доступ к элементам коллекции без раскрытия структуры.
- Пример: цикл for-each по массиву
- Используется для: обход коллекций

**Template Method** - определяет скелет алгоритма в базовом классе.
- Пример: класс для обработки платежа определяет шаги, подклассы их реализуют
- Используется для: повторное использование алгоритма

## Таблица паттернов

| Паттерн | Категория | Проблема | Решение |
|---------|-----------|----------|---------|
| Singleton | Creational | Множество экземпляров | Один экземпляр на приложение |
| Factory | Creational | Сложное создание объектов | Фабрика создает объекты |
| Adapter | Structural | Несовместимые интерфейсы | Адаптер делает совместимыми |
| Observer | Behavioral | Слабая связь между объектами | Подписка на события |
| Strategy | Behavioral | Множество алгоритмов | Инкапсуляция алгоритма |

## Архитектурные паттерны (Architectural Patterns)

На уровне архитектуры системы выделяют:

**MVC (Model-View-Controller)**: разделение на данные, представление и логику взаимодействия.

**MVVM (Model-View-ViewModel)**: используется в современных UI фреймворках (Angular, React).

**Clean Architecture**: разделение по уровням ответственности.

**Microservices Pattern**: архитектура из независимых сервисов.

**Event Sourcing**: сохранение всех событий вместо текущего состояния.

## Пример для аналитика

При проектировании системы заказов встречаются паттерны:

1. **Strategy Pattern**: разные способы расчета стоимости доставки (курьер, почта, самовывоз)
2. **Observer Pattern**: при изменении статуса заказа уведомляются клиент, склад, доставщик
3. **State Pattern**: заказ переходит между состояниями (новый → подтвержден → собран → отправлен → доставлен)
4. **Factory Pattern**: создание разных типов уведомлений (Email, SMS, Push)
5. **Facade Pattern**: один API для работы со сложной системой заказов

## Когда использовать паттерны?

Паттерны полезны, но не всегда нужны. Помните:
- Не переусложняйте код, используя паттерны где они не нужны
- Используйте паттерны когда вы видите проблему, которую они решают
- Паттерны должны упростить код, а не сделать его сложнее
- Сначала напишите рабочий код, потом рефакторьте с паттернами

## Антипаттерны

Избегайте антипаттернов:
- **God Object** - объект, который делает слишком много
- **Code Duplication** - повторение кода (Dry principle)
- **Magic Numbers** - необъяснимые числа в коде (используйте константы)
- **Long Methods** - методы, которые делают слишком много
