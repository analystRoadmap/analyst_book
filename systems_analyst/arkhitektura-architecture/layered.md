# Слоистая архитектура (Layered Architecture)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ Systems Analyst |

> Слоистая архитектура разделяет систему на горизонтальные слои, каждый из которых выполняет определенную функцию и взаимодействует со слоями ниже.

## Что такое слоистая архитектура?

Слоистая (или многоуровневая) архитектура разделяет приложение на несколько слоев, стоящих друг над другом. Каждый слой имеет определенную ответственность:
- Один слой отвечает за взаимодействие с пользователем
- Другой содержит бизнес-логику
- Третий управляет доступом к данным

Слои изолированы, что позволяет изменять один слой без влияния на другие.

## Классическая трехслойная архитектура

**Presentation Layer (Слой представления)**:
- Отвечает за отображение интерфейса пользователю
- Обрабатывает пользовательский ввод
- Не содержит бизнес-логики
- Примеры: веб-интерфейс, мобильное приложение, API контроллеры

**Business Logic Layer (Слой бизнес-логики)**:
- Содержит основную логику приложения
- Обрабатывает правила бизнеса
- Выполняет вычисления и валидацию
- Не знает о деталях БД или интерфейса пользователя
- Примеры: Services, Controllers с логикой

**Data Access Layer (Слой доступа к данным)**:
- Обрабатывает взаимодействие с хранилищем данных
- Выполняет CRUD операции (Create, Read, Update, Delete)
- Содержит SQL запросы или ORM код
- Не должен содержать бизнес-логику
- Примеры: Repositories, DAOs, Database Models

## Схема взаимодействия

```
┌─────────────────────────────────────────┐
│  Presentation Layer (UI, Controllers)   │
├─────────────────────────────────────────┤
│  Business Logic Layer (Services)        │
├─────────────────────────────────────────┤
│  Data Access Layer (Repositories)       │
├─────────────────────────────────────────┤
│  Database / File System                 │
└─────────────────────────────────────────┘
```

Запрос идет вверх: UI → Service → Repository → БД
Результат возвращается вниз: БД → Repository → Service → UI

## Основные правила слоистой архитектуры

**Изоляция слоев**: слой может обращаться только к слою непосредственно ниже, не к слоям из середины или выше.

**Разделение ответственности**: каждый слой отвечает только за свою задачу.

**Независимость реализации**: можно менять реализацию одного слоя, если интерфейс остается прежним.

**Горизонтальное разделение**: слои разделяют приложение горизонтально (по функциям), не вертикально (по компонентам).

## Преимущества слоистой архитектуры

| Преимущество | Описание |
|---|---|
| **Простота** | Интуитивна и легка для понимания, даже новичками |
| **Разделение ответственности** | Каждый слой отвечает за одно, легче тестировать и поддерживать |
| **Модульность** | Слои разделены, можно заменить один слой без влияния на другие |
| **Переиспользование** | Сервисы (бизнес-логика) могут использоваться из разных интерфейсов |
| **Тестируемость** | Слои тестируются отдельно, можно мокировать зависимости |
| **Масштабируемость инструментов** | Каждый слой использует подходящие инструменты (React для UI, Spring для бизнес-логики) |

## Ограничения и вызовы

**Производительность**: каждый вызов проходит через несколько слоев, что добавляет задержку. Требуется кэширование.

**Проблема "Thick Middle Layer"**: бизнес-логика может разбухнуть и стать сложной для поддержки. Требуется дополнительное разделение на подслои.

**Не подходит для сложных систем**: при наличии множества независимо масштабируемых компонентов лучше использовать микросервисы.

**Распределение данных**: сложнее при масштабировании БД, когда нужны шarding, replication.

**Вертикальное разделение**: приложение разделяется горизонтально, что может быть неестественно для некоторых доменов (например, разные бизнес-сущности обрабатываются одинаково).

## Практический пример для аналитика

Система управления заказами:

```
┌─────────────────────────────────────────┐
│  API Controller (/api/orders)           │  ← Presentation
│  Web UI (HTML, CSS, JavaScript)         │
├─────────────────────────────────────────┤
│  OrderService                           │  ← Business Logic
│  - validateOrder()                      │
│  - calculateTotal()                     │
│  - applyDiscount()                      │
├─────────────────────────────────────────┤
│  OrderRepository                        │  ← Data Access
│  - findOrderById()                      │
│  - saveOrder()                          │
│  - deleteOrder()                        │
├─────────────────────────────────────────┤
│  MySQL Database                         │
└─────────────────────────────────────────┘
```

При добавлении мобильного приложения просто добавляем новый Presentation слой, используя тот же OrderService.

## Расширенные слои

Часто добавляются дополнительные слои:

**Model Layer**: POJO/DTO объекты, которые передаются между слоями.

**API Gateway Layer**: маршрутизация, аутентификация, rate limiting.

**Cache Layer**: Redis, Memcached для кэширования данных.

**Queue/Message Layer**: асинхронная обработка задач, интеграция систем.

## Когда использовать слоистую архитектуру?

Слоистая архитектура подходит для:
- Стандартных веб-приложений (CRUD операции)
- Систем с четкой разделением UI, логики и данных
- Команд до 10-15 человек
- Приложений, которые будут развиваться по одному направлению

Избегайте слоистой архитектуры для:
- Высоконагруженных систем с разной нагрузкой по компонентам
- Систем, требующих независимого развертывания компонентов
- Очень сложных доменов с множеством взаимосвязей
