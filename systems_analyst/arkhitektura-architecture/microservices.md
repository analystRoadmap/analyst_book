# Микросервисная архитектура (Microservices)

| Грейд | Роль |
|-------|------|
| 🟠 Senior | 🅰️ Systems Analyst |

> Микросервисы - это архитектурный стиль, где приложение разбивается на набор небольших, независимых сервисов, каждый из которых выполняет одну бизнес-функцию.

## Что такое микросервисная архитектура?

Микросервисная архитектура - это подход разделения крупного приложения на набор небольших, слабо связанных сервисов, которые:
- Разрабатываются независимо
- Развертываются отдельно
- Общаются между собой через сетевые протоколы (HTTP, gRPC, очереди сообщений)
- Имеют собственные базы данных (часто)
- Могут использовать различные технологии

Примеры микросервисов в e-commerce системе:
- Сервис каталога продуктов
- Сервис корзины покупок
- Сервис оплаты
- Сервис доставки
- Сервис уведомлений

## Основные характеристики

**Автономность**: каждый микросервис разрабатывается, тестируется и развертывается независимо. Команда может работать параллельно, не дожидаясь других команд.

**Разделение ответственности**: каждый сервис отвечает за одну бизнес-функцию. Если появляется новая функция, добавляется новый сервис.

**Распределенная коммуникация**: сервисы взаимодействуют через HTTP REST, gRPC, очереди сообщений или event-driven подходы.

**Собственные хранилища**: каждый сервис может иметь собственную БД, что обеспечивает независимость и горизонтальное масштабирование.

## Преимущества

| Преимущество | Описание |
|---|---|
| **Независимое развертывание** | Развернуть изменения в одном сервисе без влияния на остальные |
| **Технологическая гибкость** | Разные сервисы могут использовать разные технологии и языки |
| **Масштабируемость** | Масштабировать отдельно нагруженные сервисы, а не всё приложение |
| **Отказоустойчивость** | Если один сервис упал, остальная система работает, может быть graceful degradation |
| **Команды разработчиков** | Команда может полностью владеть одним микросервисом от разработки до поддержки |

## Вызовы и сложности

**Сетевая коммуникация**: увеличение задержек, возможность потери пакетов, необходимость обработки сетевых ошибок и таймаутов.

**Распределенная транзакция**: при необходимости обновить данные в нескольких сервисах возникают проблемы консистентности. Используют паттерны Saga, двухфазная коммитация.

**Мониторинг и отладка**: сложнее отследить запрос, проходящий через несколько сервисов, требуется распределенная трассировка.

**Управление зависимостями**: версионирование API, мониторинг breaking changes, понимание влияния изменений в одном сервисе на другие.

**Операционная сложность**: нужна система управления контейнерами (Kubernetes), логирование, метрики, алерты для всех сервисов.

**Тестирование**: интеграционное тестирование требует подъема всей экосистемы, контрактные тесты, chaos engineering.

## Архитектурные паттерны

**API Gateway**: единая точка входа для всех запросов, маршрутизирует на необходимые микросервисы, обрабатывает аутентификацию.

**Service Discovery**: автоматическое обнаружение и регистрация сервисов, критично в облачных окружениях где IP может меняться.

**Event-Driven Communication**: сервисы публикуют события, другие на них подписываются. Ослабляет связь между сервисами.

**Saga Pattern**: координация транзакций между микросервисами, состоит из последовательности локальных транзакций.

## Матрица выбора: когда использовать микросервисы?

| Фактор | Монолит подходит | Микросервисы подходят |
|--------|------------------|----------------------|
| **Размер команды** | <10 человек | >10 человек |
| **Частота развертывания** | Несколько раз в месяц | Несколько раз в день |
| **Требования к масштабированию** | Равномерная нагрузка | Разная нагрузка по компонентам |
| **Технологическое разнообразие** | Один стек технологий | Разные стеки по сервисам |
| **Независимость команд** | Низкая | Высокая |
| **Инфраструктура** | Традиционная | Cloud-native, Kubernetes |

## Пример для аналитика

Система управления проектами может быть разделена на:

1. **Auth Service** - аутентификация и авторизация
2. **User Service** - профили пользователей, роли
3. **Project Service** - создание, обновление проектов
4. **Task Service** - управление задачами в проектах
5. **Notification Service** - уведомления по email/push
6. **File Service** - загрузка и хранение файлов

Каждый сервис имеет свою БД, собственный CI/CD pipeline, может быть развернут на разных серверах с разной нагрузкой.
