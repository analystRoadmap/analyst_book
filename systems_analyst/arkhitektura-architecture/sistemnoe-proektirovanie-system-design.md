| Грейд | Роль |
|-------|------|

> Системное проектирование - это процесс определения архитектуры, компонентов и взаимодействия для создания системы, которая удовлетворяет функциональным и нефункциональным требованиям.

## Что такое системное проектирование?

Системное проектирование связывает функциональные требования (что должна делать система) с техническими ограничениями (как это реализовать). Для аналитика это критично при проектировании масштабируемых, надежных и производительных систем.

Процесс системного проектирования включает:
1. Определение функциональных требований
2. Анализ нефункциональных требований (SLA, масштабируемость, отказоустойчивость)
3. Выбор архитектурных паттернов
4. Определение границ контекстов и сервисов
5. Планирование масштабирования и отказоустойчивости
6. Документирование решений

## Функциональные и нефункциональные требования

**Функциональные требования**: что система должна делать
- Пользователь может создать заказ
- Система отправляет уведомление при обновлении статуса
- API возвращает список товаров с фильтрацией

**Нефункциональные требования**: как система должна это делать
- **Масштабируемость**: обработка 10,000 запросов в секунду
- **Доступность**: 99.9% uptime (4.3 часа downtime в год)
- **Производительность**: ответ за <200ms на 95% запросов
- **Безопасность**: шифрование данных, аутентификация, rate limiting
- **Надежность**: восстановление после сбоев, backup, disaster recovery
- **Стоимость**: минимизировать затраты на инфраструктуру

## Архитектурные решения и компромиссы

При проектировании системы всегда есть компромиссы:

| Решение | Плюсы | Минусы |
|---------|-------|--------|
| **Монолит** | Простота, производительность | Сложность масштабирования, развертывания |
| **Микросервисы** | Независимость, масштабируемость | Сложность операций, сетевые задержки |
| **SQL БД** | ACID, транзакции | Масштабируемость, гибкость схемы |
| **NoSQL БД** | Масштабируемость, гибкость | Нет ACID, сложность консистентности |
| **Кэш** | Производительность | Стоимость, консистентность |
| **CDN** | Скорость доставки | Стоимость, задержка обновления |

## Ключевые метрики и SLA

**Request Per Second (RPS)**: количество запросов в секунду
- Низкая нагрузка: <100 RPS
- Средняя: 100-10,000 RPS
- Высокая: 10,000-1,000,000 RPS

**Latency**: время ответа
- P50 (медиана): 50% запросов отвечают быстрее
- P95, P99: 95% и 99% запросов отвечают быстрее
- Обычная цель: P99 < 200ms

**Availability**: процент времени, когда система работает
- 99.9% (три девятки) = 8.76 часов downtime в год
- 99.99% (четыре девятки) = 52.56 минут downtime в год

**Data Consistency**: актуальность данных
- Strong consistency: при записи все читают новые данные
- Eventual consistency: данные становятся актуальными со временем

## Паттерны масштабирования

**Vertical Scaling** (масштабирование вверх): добавить мощности (CPU, RAM) одному серверу
- Плюсы: просто, нет сложности координации
- Минусы: есть потолок, очень дорого, single point of failure

**Horizontal Scaling** (масштабирование вширь): добавить больше серверов
- Плюсы: нет потолка, распределение нагрузки
- Минусы: сложность синхронизации, консистентность данных

**Database Sharding**: разбиение данных по нескольким БД
- Плюсы: масштабируемость для больших объемов
- Минусы: сложность запросов, cross-shard joins

**Caching**: хранение часто используемых данных в памяти
- Плюсы: значительное ускорение
- Минусы: управление консистентностью, память

## Отказоустойчивость и восстановление

**Replication**: копирование данных на несколько серверов
- Улучшает availability (если один сервер упал, другой работает)
- Требует синхронизации

**Load Balancer**: распределение нагрузки между серверами
- Пользователь подключается к балансировщику
- Балансировщик маршрутизирует на здоровые сервера

**Circuit Breaker**: предотвращает каскадные сбои
- Если сервис неработающий, не отправляем запросы
- Через время снова пробуем подключиться

**Graceful Degradation**: деградация функциональности при сбое
- Вместо полного отказа, показываем кэшированные данные
- Отключаем второстепенные функции, оставляем основное

## Пример системного дизайна: YouTube-подобная система

**Функциональные требования**:
- Загрузить видео
- Просмотреть видео
- Поиск видео
- Рекомендации

**Нефункциональные требования**:
- 100 миллионов пользователей
- 10 миллионов загрузок видео в день
- 5 миллиардов просмотров в день
- Latency < 200ms
- Availability 99.9%

**Архитектура**:
```
[CDN] ← видео контент (глобально распределено)
  ↑
[Web Server + Load Balancer] ← обработка запросов
  ↓
[Video Service] ← обработка загрузок, кодирование
[Search Service] ← индексирование и поиск (Elasticsearch)
[Recommendation Service] ← ML модель для рекомендаций
  ↓
[Primary DB] ← метаданные видео
[Replica DB] ← реплика для чтения
[Cache] ← горячие данные (Redis)
[Message Queue] ← асинхронные задачи (кодирование видео)
```

## Процесс системного дизайна

1. **Clarify Requirements**: понять требования, задать вопросы
2. **Back of Envelope Calculation**: примерная оценка нагрузки
3. **High Level Design**: определить основные компоненты
4. **Deep Dive**: детализировать каждый компонент
5. **Address Bottlenecks**: найти узкие места, решить

## Антипаттерны системного дизайна

- **Недооценка нагрузки**: спроектировать систему на 100 RPS, потом нужна для 10,000 RPS
- **Переусложнение**: использовать микросервисы для простого CRUD приложения
- **Игнорирование SLA**: забыть про отказоустойчивость и backup
- **Отсутствие мониторинга**: не знать, что происходит в production
