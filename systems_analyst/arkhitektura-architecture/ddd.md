# Domain-Driven Design (DDD)

| Грейд | Роль |
|-------|------|
| 🟠 Senior | 🅰️ Systems Analyst |

> Domain-Driven Design - методология разработки, где архитектура и код отражают структуру и правила бизнеса (домена).

## Что такое DDD?

Domain-Driven Design - это методология и набор практик для проектирования сложных систем. Главная идея: архитектура должна отражать структуру и правила бизнеса, а не техническую реализацию.

Вместо того, чтобы спроектировать систему с нуля технически, DDD начинает с глубокого понимания бизнес-домена:
- Какие сущности существуют в домене?
- Какие правила управляют взаимодействием этих сущностей?
- Как язык бизнеса (терминология) должен отразиться в коде?

## Ключевые концепции DDD

### Ubiquitous Language (Общий язык)

Разработчики и эксперты домена должны использовать один и тот же язык для обсуждения системы.

Вместо технических терминов используем термины домена:
- Плохо: "создать запись в таблице orders"
- Хорошо: "создать заказ покупателя"

Этот язык должен отражаться в коде:
- Классы именуются по терминам домена (Order, Customer, Payment)
- Методы описывают бизнес-действия (createOrder(), cancelOrder())

### Bounded Context (Ограниченный контекст)

Большие системы разбиваются на ограниченные контексты - подсистемы с четкими границами.

Пример: система e-commerce имеет контексты:
- **Sales Context**: заказы, клиенты, корзина покупок
- **Shipping Context**: доставка, отслеживание, адреса
- **Billing Context**: счета, оплата, налоги
- **Inventory Context**: товары, складской учет, поставщики

Каждый контекст:
- Имеет свой набор сущностей и правил
- Может использовать разные БД и технологии
- Взаимодействует с другими контекстами через четко определенные интерфейсы
- Может разрабатываться отдельной командой

### Aggregates (Агрегаты)

Группы связанных объектов, которые обновляются вместе и сохраняют консистентность.

Пример: Order Aggregate
```
Order (корневая сущность)
├── OrderItems (список товаров)
│   ├── Product ID
│   ├── Quantity
│   └── Price
├── Customer ID
├── Status
└── Total Amount
```

Правила:
- Обновляется только через корневую сущность (Order)
- Все правила консистентности для товаров в заказе проверяются в Order
- Если удалить заказ, удаляются все товары в нем

### Entities (Сущности)

Объекты с идентичностью, которые существуют в течение времени.

Пример: Customer имеет ID (уникальный идентификатор) и существует как долгоживущий объект. Два Customer с одинаковыми именами - разные сущности (разные ID).

### Value Objects (Объекты значения)

Объекты, определяемые их значением, а не идентичностью.

Пример: Address (адрес). Два Address с одинаковыми значениями (улица, город, индекс) считаются одинаковыми, даже если это разные объекты в памяти.

### Repositories (Хранилища)

Обеспечивают доступ к агрегатам. Предоставляют интерфейс для сохранения и поиска.

```
OrderRepository.save(order)  // Сохранить заказ
order = OrderRepository.findById(id)  // Найти заказ по ID
orders = OrderRepository.findByCustomer(customer)  // Найти заказы клиента
```

### Services (Сервисы)

Содержат бизнес-логику, которая не принадлежит отдельной сущности.

Пример: OrderService.createOrder() может:
- Создать новый Order
- Проверить доступность товаров
- Рассчитать стоимость доставки
- Отправить уведомление клиенту

## Слои архитектуры DDD

```
┌─────────────────────────────────────────┐
│  Presentation Layer                     │  ← UI, API контроллеры
├─────────────────────────────────────────┤
│  Application Layer                      │  ← Application Services
│  (Orchestration, DTOs, Mapping)         │
├─────────────────────────────────────────┤
│  Domain Layer                           │  ← Entities, Aggregates,
│  (Business Rules, Ubiquitous Language)  │     Value Objects, Services
├─────────────────────────────────────────┤
│  Infrastructure Layer                   │  ← Repositories, Database,
│  (Persistence, External Services)       │     External APIs
└─────────────────────────────────────────┘
```

## Преимущества DDD

| Преимущество | Описание |
|---|---|
| **Выравнивание с бизнесом** | Код отражает структуру бизнеса, легче обсуждать изменения |
| **Понимаемость** | Код написан на языке домена, понятен не только разработчикам |
| **Гибкость** | Правила бизнеса локализованы, легче вносить изменения |
| **Разделение ответственности** | Clear separation between business logic and infrastructure |
| **Масштабируемость разработки** | Независимые контексты = независимые команды |

## Вызовы и сложности

**Кривая обучения**: требуется время для понимания DDD, особенно Bounded Context и Aggregates.

**Переусложнение**: не все системы нуждаются в полном DDD. CRUD-приложения могут быть проще.

**Координация между контекстами**: требуется обозначить интеграционные точки (API, events).

**Миграция легаси**: переход на DDD в старых системах требует больших изменений.

## Практический пример для аналитика

Система управления заказами в e-commerce:

**Sales Context**:
- Entities: Order, Customer
- Aggregates: Order (Order + OrderItems)
- Value Objects: Money (сумма в валюте), Address
- Services: OrderService (создание, отмена заказа)

**Shipping Context**:
- Entities: Shipment, Carrier
- Aggregates: Shipment (Shipment + TrackingHistory)
- Services: ShippingService (отправка, отслеживание)

**Binding** между контекстами:
- Когда заказ оплачен (Sales), создается отправка (Shipping)
- Когда отправка доставлена (Shipping), уведомляется заказ (Sales)

Каждый контекст может иметь свою БД, свой код, свою команду.

## Когда использовать DDD?

Используйте DDD для:
- Сложных бизнес-доменов с множеством правил
- Долгоживущих систем, которые будут развиваться
- Систем, требующих сотрудничества разработчиков и экспертов домена
- Микросервисных архитектур (каждый сервис = Bounded Context)

Избегайте полного DDD для:
- CRUD-приложений с простой логикой
- Прототипов и MVP
- Систем с четким техническим дизайном без сложной бизнес-логики
