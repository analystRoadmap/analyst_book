## Что такое реляционная база данных

Реляционная база данных (РБД) хранит данные в виде таблиц, связанных между собой через ключи. Каждая таблица представляет сущность (например, «Клиент», «Заказ»), столбцы — атрибуты сущности, строки — конкретные записи.

**Ключевые понятия:**

- **Первичный ключ (PK)** — уникальный идентификатор каждой записи в таблице (обычно `id`).
- **Внешний ключ (FK)** — ссылка на первичный ключ другой таблицы, создающая связь между ними.
- **Индексы** — структуры данных для ускорения поиска (как оглавление в книге).
- **Constraints** — ограничения целостности: NOT NULL, UNIQUE, CHECK, FOREIGN KEY.

## ACID — гарантии целостности данных

Реляционные БД обеспечивают четыре ключевых свойства транзакций:

| Свойство | Что означает | Пример |
|----------|-------------|--------|
| **Atomicity** (атомарность) | Транзакция выполняется целиком или не выполняется совсем | Перевод денег: либо списание И зачисление, либо ничего |
| **Consistency** (согласованность) | После транзакции БД остаётся в валидном состоянии | Баланс не может стать отрицательным, если есть constraint |
| **Isolation** (изолированность) | Параллельные транзакции не мешают друг другу | Два кассира не продадут один и тот же последний товар |
| **Durability** (долговечность) | Завершённая транзакция сохраняется даже при сбое | После подтверждения заказа он не потеряется при перезагрузке |

## Проектирование: пошаговый подход для аналитика

1. **Выделите сущности** из бизнес-требований. Пример: для интернет-магазина — Customer, Order, Product, OrderItem, Payment.
2. **Определите атрибуты** каждой сущности и типы данных.
3. **Назначьте первичные ключи** — обычно суррогатный `id` (SERIAL / UUID).
4. **Определите связи и кардинальность:**
   - 1:1 — Customer ↔ CustomerProfile
   - 1:N — Customer → Order (один клиент, много заказов)
   - M:N — Order ↔ Product (через промежуточную OrderItem)
5. **Нормализуйте до 3NF** — устраните дублирование данных.
6. **Добавьте constraints** — NOT NULL для обязательных полей, UNIQUE для email, CHECK для статусов.
7. **Подумайте об индексах** — на полях, по которым часто ищут или фильтруют.

## Нормализация: три главные формы

| Форма | Правило | Пример нарушения |
|-------|---------|-----------------|
| **1NF** | Каждая ячейка содержит одно атомарное значение | Поле `phones` = «+7999..., +7888...» — два значения в одной ячейке |
| **2NF** | Все неключевые атрибуты зависят от всего первичного ключа | В таблице OrderItem(order_id, product_id, product_name) — product_name зависит только от product_id |
| **3NF** | Нет транзитивных зависимостей между неключевыми атрибутами | В таблице Employee(id, department_id, department_name) — department_name зависит от department_id, а не от id |

## ERD-диаграммы: язык общения аналитика и разработчика

ERD (Entity-Relationship Diagram) — графическое представление модели данных. Наиболее распространённая нотация — **Crow's Foot**, где:

- Прямоугольники = сущности (таблицы)
- Линии = связи
- «Лапки вороны» (три линии) = «много»
- Одна линия = «один»
- Кружок = «ноль» (опционально)

## Популярные реляционные СУБД

| СУБД | Особенности | Когда выбирать |
|------|------------|----------------|
| **PostgreSQL** | Расширяемая, поддержка JSON, полнотекстового поиска | Универсальный выбор, особенно для новых проектов |
| **MySQL / MariaDB** | Простая, быстрая на чтение | Веб-приложения, CMS (WordPress, Drupal) |
| **MS SQL Server** | Интеграция с экосистемой Microsoft | Корпоративные .NET-решения |
| **Oracle** | Мощная, для enterprise | Банки, телеком, госсектор |
| **SQLite** | Встроенная, без сервера | Мобильные приложения, прототипы |
