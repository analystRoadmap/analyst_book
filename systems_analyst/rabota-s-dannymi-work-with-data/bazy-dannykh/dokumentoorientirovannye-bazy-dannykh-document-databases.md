# Документоориентированные базы данных (Document Databases)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> Документные БД — самый популярный тип NoSQL. Аналитику важно понимать, когда гибкая схема документа выигрывает у жёсткой табличной структуры, и уметь проектировать модели данных в документной парадигме.

## Что такое документная база данных

Документная БД хранит данные в виде документов — обычно в формате JSON или BSON. Каждый документ — самодостаточная единица данных, содержащая все нужные поля. Документы группируются в **коллекции** (аналог таблиц в SQL).

Главное отличие от реляционной БД: у каждого документа в коллекции может быть своя структура. Не нужно заранее определять схему.

## Пример: как выглядят данные

**В реляционной БД** — три таблицы: customers, addresses, preferences.

**В документной БД** — один документ:
```json
{
  "_id": "cust_001",
  "name": "Иван Петров",
  "email": "ivan@example.com",
  "addresses": [
    {"type": "home", "city": "Москва", "street": "Ленина 1"},
    {"type": "work", "city": "Москва", "street": "Тверская 10"}
  ],
  "preferences": {
    "language": "ru",
    "notifications": true,
    "theme": "dark"
  }
}
```

Все данные о клиенте хранятся вместе — не нужен JOIN. Это ускоряет чтение и упрощает API.

## Когда выбирать документную БД

**Подходит:**
- Каталоги товаров с разными атрибутами по категориям
- Профили пользователей с переменным набором полей
- CMS и контент-менеджмент (статьи, посты, комментарии)
- Прототипы, где схема ещё не финализирована
- Микросервисы, где каждый сервис владеет своими данными

**Не подходит:**
- Финансовые транзакции, требующие ACID (лучше SQL)
- Данные с множеством связей между сущностями (лучше граф или SQL + JOIN)
- Сложная аналитика с агрегациями по разным срезам (лучше колоночная или SQL)

## Встраивание vs ссылки (Embedding vs Referencing)

Ключевое решение при проектировании документной модели:

| Подход | Когда использовать | Пример |
|--------|-------------------|--------|
| **Embedding** (вложение) | Данные читаются вместе, обновляются редко | Адреса внутри документа клиента |
| **Referencing** (ссылка) | Данные общие для нескольких документов, часто обновляются | `"category_id": "cat_electronics"` — ссылка на отдельную коллекцию |

**Правило:** встраивайте, если данные «принадлежат» документу и не используются отдельно. Ссылайтесь, если данные общие и меняются независимо.

## Популярные документные СУБД

| СУБД | Особенности | Когда выбирать |
|------|------------|----------------|
| **MongoDB** | Лидер рынка, Atlas (облако), агрегации | Большинство сценариев, хорошая экосистема |
| **CouchDB** | HTTP API, встроенная репликация | Offline-first приложения |
| **Firebase Firestore** | Realtime, мобильные приложения | Быстрые прототипы, мобильные app |
| **Amazon DocumentDB** | MongoDB-совместимый в AWS | Проекты в экосистеме AWS |
