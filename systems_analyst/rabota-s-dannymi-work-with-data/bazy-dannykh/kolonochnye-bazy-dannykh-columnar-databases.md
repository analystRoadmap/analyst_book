# Колоночные базы данных (Columnar Databases)

| Грейд | Роль |
|-------|------|
| 🟠 Senior | 🅰️ System Analyst |

> Колоночные БД — основа современной аналитики больших данных. Если аналитик сталкивается с задачей «построить быстрый отчёт по миллионам записей», понимание колоночного хранения — обязательно.

## Что такое колоночная база данных

В традиционной (строковой) БД данные одной записи хранятся рядом на диске: `[id=1, name="Иван", city="Москва", amount=500]`. Чтобы посчитать сумму по полю `amount`, БД вынуждена прочитать все поля всех строк — включая ненужные `name` и `city`.

В колоночной БД данные хранятся по столбцам: все значения `amount` лежат рядом, все `city` — рядом. Чтобы посчитать сумму, БД читает только колонку `amount` — это в разы быстрее.

## Строковое vs колоночное хранение: наглядно

**Строковое (row-based):**
```
Row 1: [id=1, name="Иван",  city="Москва",  amount=500]
Row 2: [id=2, name="Мария", city="Питер",   amount=300]
Row 3: [id=3, name="Пётр",  city="Москва",  amount=700]
```

**Колоночное (column-based):**
```
id:     [1, 2, 3]
name:   ["Иван", "Мария", "Пётр"]
city:   ["Москва", "Питер", "Москва"]
amount: [500, 300, 700]
```

Запрос `SELECT SUM(amount) FROM orders` читает только последнюю колонку — в 4 раза меньше данных.

## Когда выбирать колоночную БД

**Подходит (OLAP-сценарии):**
- Аналитические отчёты: «сумма продаж по месяцам», «средний чек по регионам»
- Data Warehouse и BI-системы
- Логи и события (миллиарды записей)
- Метрики и мониторинг (временные ряды)

**Не подходит (OLTP-сценарии):**
- Частые вставки/обновления отдельных строк
- Транзакционные системы (корзина, оплата)
- Запросы, которые возвращают целые строки по ID

## Почему колоночные БД быстрые

1. **Чтение только нужных колонок** — не тратим I/O на ненужные данные
2. **Сжатие** — значения одного типа в одной колонке отлично сжимаются (повторяющиеся города, даты)
3. **Vectorized execution** — процессор обрабатывает массив однотипных значений эффективнее
4. **Пропуск данных** — партиции и zone maps позволяют пропускать целые блоки данных

## Популярные колоночные СУБД

| СУБД | Особенности | Когда выбирать |
|------|------------|----------------|
| **ClickHouse** | Очень быстрый, Open Source, от Яндекса | Аналитика в реальном времени, логи |
| **Amazon Redshift** | Управляемый в AWS, SQL-совместимый | DWH в экосистеме AWS |
| **Google BigQuery** | Serverless, масштабируемый | Аналитика в GCP, ad-hoc запросы |
| **Apache Cassandra** | Распределённая, высокая доступность | IoT, временные ряды, geo-distributed |
| **Apache HBase** | На базе Hadoop, для больших данных | Проекты с Hadoop-стеком |
| **Vertica** | Enterprise, мощная аналитика | Корпоративные DWH |

## Пример из практики аналитика

**Задача:** бизнес хочет дашборд «продажи по регионам за последний год, с фильтрами по категориям товаров». Таблица фактов — 500 млн записей.

**На PostgreSQL:** запрос выполняется 45 секунд (строковое хранение, читает все колонки каждой записи).

**На ClickHouse:** тот же запрос — 0.3 секунды (читает только нужные колонки, сжатие, vectorized execution).

Роль аналитика: при проектировании системы предложить колоночное хранилище для аналитической подсистемы, оставив PostgreSQL для транзакционной.
