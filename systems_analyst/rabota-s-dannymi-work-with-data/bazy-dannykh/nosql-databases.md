# NoSQL databases

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> NoSQL — это не «анти-SQL», а целое семейство баз данных с разными моделями хранения. Аналитику важно понимать, когда реляционная БД не подходит и какой тип NoSQL решит задачу лучше.

## Что такое NoSQL

NoSQL (Not Only SQL) — общее название для баз данных, которые не используют табличную реляционную модель. Вместо жёсткой схемы со столбцами и строками NoSQL предлагает гибкие модели данных, оптимизированные под конкретные сценарии.

## Четыре основные модели NoSQL

| Модель | Как хранит данные | Пример СУБД | Типичный сценарий |
|--------|-------------------|-------------|-------------------|
| **Key-Value** | Пары «ключ → значение» | Redis, DynamoDB, Memcached | Кэширование, сессии пользователей, корзины |
| **Document** | JSON/BSON документы в коллекциях | MongoDB, CouchDB, Firestore | Каталоги, CMS, профили пользователей |
| **Graph** | Узлы и рёбра (связи) | Neo4j, Amazon Neptune | Социальные сети, рекомендации, fraud detection |
| **Column-family** | Данные по колонкам | Cassandra, HBase, Bigtable | Логи, IoT, временные ряды |

## SQL vs NoSQL: когда что выбирать

| Критерий | SQL (реляционная) | NoSQL |
|----------|-------------------|-------|
| Схема данных | Фиксированная (ALTER TABLE для изменения) | Гибкая (можно добавить поле в любой документ) |
| Связи между данными | Сильные (JOIN, FK) | Слабые или денормализованные |
| Транзакции | Полный ACID | Обычно eventual consistency (BASE) |
| Масштабирование | Вертикальное (мощнее сервер) | Горизонтальное (больше серверов) |
| Запросы | Мощный SQL | Зависит от СУБД, часто API-ориентированные |

## BASE vs ACID

Если реляционные БД дают гарантии ACID, то многие NoSQL-решения следуют модели BASE:

- **Basically Available** — система почти всегда доступна
- **Soft state** — состояние может меняться со временем даже без новых записей
- **Eventually consistent** — данные станут согласованными... когда-нибудь

**Пример из жизни:** когда вы публикуете пост в соцсети, не все ваши друзья видят его мгновенно. Через несколько секунд данные «растекаются» по серверам. Это eventual consistency — и для этого сценария она приемлема.

## Практический пример: когда выбрать NoSQL

**Задача:** Разрабатывается маркетплейс. У каждого продавца карточки товаров имеют разные атрибуты: у одежды — размер и цвет, у электроники — вес и напряжение, у еды — срок годности и состав.

**Проблема с SQL:** в реляционной БД придётся либо создать 50 таблиц под каждую категорию, либо сделать таблицу-«монстра» с 200 nullable-колонками, либо использовать EAV-паттерн (entity-attribute-value), который неудобен для запросов.

**Решение с Document DB (MongoDB):** каждый товар — JSON-документ со своим набором полей. Одежда хранит `{size, color}`, электроника — `{weight, voltage}`. Гибкая схема решает задачу естественно.

## Что важно помнить аналитику

1. **NoSQL — не замена SQL**, а дополнение. Большинство проектов используют оба типа.
2. **Денормализация — осознанный выбор.** В NoSQL данные часто дублируются ради скорости чтения.
3. **Отсутствие JOIN** в большинстве NoSQL БД означает, что связи нужно продумывать на этапе проектирования.
4. **Выбор NoSQL влияет на архитектуру** — это решение не «давайте попробуем», а архитектурный выбор с последствиями.
