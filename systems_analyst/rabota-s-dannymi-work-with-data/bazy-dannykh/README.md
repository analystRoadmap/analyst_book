# Базы данных

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> Выбор типа базы данных — одно из ключевых архитектурных решений в проекте. Аналитик должен понимать, чем отличаются типы БД и когда какой применять, чтобы предложить адекватное решение, а не навязать единственный знакомый инструмент.

## Что такое база данных

База данных (БД) — организованная коллекция данных, к которой можно обращаться, управлять и обновлять. Система управления базой данных (СУБД) — программа, которая обеспечивает доступ к данным: PostgreSQL, MongoDB, Redis и другие.

## Типы баз данных: когда что выбирать

| Тип | Модель данных | Когда выбирать | Примеры |
|-----|---------------|----------------|---------|
| [Реляционные](relyacionnye-bazy-dannykh-relational-databases/) | Таблицы со строками и столбцами | Структурированные данные, сложные связи, ACID-транзакции | PostgreSQL, MySQL, Oracle, MS SQL |
| [NoSQL (Key-Value)](nosql-databases.md) | Пары «ключ — значение» | Кэширование, сессии, простые lookup-операции | Redis, DynamoDB, Memcached |
| [Документные](dokumentoorientirovannye-bazy-dannykh-document-databases.md) | JSON/BSON документы в коллекциях | Гибкая схема, каталоги товаров, CMS | MongoDB, CouchDB, Firestore |
| [Графовые](grafovye-bazy-dannykh-graph-databases.md) | Узлы и рёбра (связи) | Социальные сети, рекомендации, fraud detection | Neo4j, Amazon Neptune |
| [Колоночные](kolonochnye-bazy-dannykh-columnar-databases.md) | Данные хранятся по колонкам | Аналитика, OLAP, агрегации по большим объёмам | ClickHouse, Apache Cassandra, Redshift |

## OLTP vs OLAP — два мира баз данных

Это ключевое различие, которое аналитик должен понимать при проектировании:

**OLTP (Online Transaction Processing)** — обрабатывает множество коротких транзакций: создать заказ, обновить баланс, зарегистрировать пользователя. Главное — скорость записи и целостность. Примеры: PostgreSQL, MySQL.

**OLAP (Online Analytical Processing)** — выполняет сложные аналитические запросы по большим объёмам данных: «сумма продаж за квартал по регионам». Главное — скорость чтения и агрегаций. Примеры: ClickHouse, Redshift, BigQuery.

**Практический пример для аналитика:** когда бизнес просит «добавить дашборд с аналитикой продаж», не нужно строить сложные SQL-запросы к OLTP-базе продакшена. Правильный подход — предложить отдельное аналитическое хранилище (OLAP), куда данные поступают через ETL.

## CAP-теорема: простое объяснение

При проектировании распределённых систем невозможно одновременно гарантировать все три свойства:

- **Consistency** (согласованность) — все узлы видят одинаковые данные
- **Availability** (доступность) — система всегда отвечает на запросы
- **Partition tolerance** (устойчивость к разделению) — система работает, даже если часть узлов недоступна

Приходится выбирать два из трёх. Реляционные БД обычно выбирают CP (согласованность + устойчивость), многие NoSQL-решения — AP (доступность + устойчивость).

## Чек-лист: как аналитик выбирает тип БД

1. **Данные структурированы и связаны?** → Реляционная БД
2. **Схема меняется часто, данные разнородные?** → Документная БД
3. **Нужна аналитика по миллионам записей?** → Колоночная БД или OLAP
4. **Важны связи между объектами (граф)?** → Графовая БД
5. **Нужно быстрое кэширование?** → Key-Value (Redis)
6. **Нужны ACID-транзакции?** → Реляционная БД
7. **Будет горизонтальное масштабирование?** → NoSQL
