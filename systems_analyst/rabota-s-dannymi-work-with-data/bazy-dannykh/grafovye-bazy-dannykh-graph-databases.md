# Графовые базы данных (Graph Databases)

| Грейд | Роль |
|-------|------|
| 🟠 Senior | 🅰️ System Analyst |

> Графовые БД — специализированный инструмент для задач, где важны связи между объектами. Аналитику важно распознать, когда данные по своей природе являются графом, и предложить соответствующее решение.

## Что такое графовая база данных

Графовая БД хранит данные в виде графа: **узлы** (nodes) представляют сущности, а **рёбра** (edges) — связи между ними. У узлов и рёбер могут быть свойства (properties).

В отличие от реляционных БД, где для получения связей нужно делать JOIN по нескольким таблицам, графовая БД хранит связи как первоклассные объекты. Это делает обход связей очень быстрым, независимо от размера базы.

## Когда графовая БД лучше реляционной

**Правило:** если запросы к данным формулируются как «найди связанные объекты на N уровнях глубины» — это кандидат для графовой БД.

| Задача | SQL подход | Графовый подход |
|--------|-----------|-----------------|
| «Друзья друзей» в соцсети | JOIN users 3 раза — медленно при миллионах записей | Обход графа на 2 шага — миллисекунды |
| Fraud detection | Сложные подзапросы с множеством JOIN | Поиск подозрительных паттернов связей |
| Рекомендации | «Люди, купившие X, также купили Y» — тяжело на SQL | Естественный обход графа покупок |
| Оргструктура компании | Рекурсивные CTE | Нативный обход иерархии |

## Пример: модель данных социальной сети

```
(Alice)-[:FOLLOWS]->(Bob)
(Bob)-[:FOLLOWS]->(Charlie)
(Alice)-[:LIKES]->(Post{title:"GraphDB intro"})
(Charlie)-[:AUTHORED]->(Post{title:"GraphDB intro"})
```

**Запрос на Cypher (язык Neo4j):** «Найти авторов постов, которые лайкнула Alice»
```cypher
MATCH (alice:User {name: "Alice"})-[:LIKES]->(post)<-[:AUTHORED]-(author)
RETURN author.name, post.title
```

Тот же запрос на SQL потребовал бы 3–4 JOIN и был бы менее читаемым.

## Популярные графовые СУБД

| СУБД | Особенности | Лицензия |
|------|------------|----------|
| **Neo4j** | Лидер рынка, язык Cypher, богатая экосистема | Community (бесплатно) / Enterprise |
| **Amazon Neptune** | Управляемый сервис в AWS, поддержка Gremlin и SPARQL | Облачный (pay-per-use) |
| **ArangoDB** | Мульти-модельная: граф + документы + key-value | Open Source |
| **JanusGraph** | Распределённая, масштабируемая | Open Source |

## Что важно помнить аналитику

1. **Графовая БД — не серебряная пуля.** Если данные хорошо ложатся в таблицы и связи простые (1:N) — реляционная БД будет проще и эффективнее.
2. **Язык запросов другой.** Вместо SQL используется Cypher (Neo4j), Gremlin (TinkerPop) или SPARQL (RDF). Аналитику полезно понимать базовый синтаксис.
3. **Моделирование «от связей».** При проектировании графовой модели начинайте с вопросов, которые нужно задать данным: «Кто связан с кем?», «Какой кратчайший путь?»
