# Файловое хранилище (File Storage)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ System Analyst |

> Не все данные живут в базах данных. Файлы, изображения, документы, логи, бэкапы — всё это хранится в файловых и объектных хранилищах. Аналитику важно понимать типы хранилищ, чтобы правильно проектировать системы.

## Три типа хранилищ

| Тип | Как работает | Для чего | Примеры |
|-----|-------------|----------|---------|
| **Файловое (File)** | Иерархия папок и файлов, как на компьютере | Общие документы, NAS в офисе | NFS, SMB, EFS |
| **Блочное (Block)** | Данные разбиты на блоки, как жёсткий диск | Диски виртуальных машин, БД | AWS EBS, Azure Disk |
| **Объектное (Object)** | Плоское пространство: объект + метаданные + ключ | Картинки, видео, бэкапы, Data Lake | AWS S3, GCS, MinIO |

## Объектное хранилище — стандарт для современных систем

Amazon S3 (Simple Storage Service) стал де-факто стандартом. Большинство облачных и on-premise решений поддерживают S3-совместимый API.

**Ключевые концепции:**
- **Bucket** — контейнер верхнего уровня (как корневая папка)
- **Object** — файл + метаданные + уникальный ключ
- **Key** — путь к объекту: `orders/2025/03/report.csv`
- **Версионирование** — хранение истории изменений файла
- **Lifecycle policies** — автоматическое перемещение старых данных в дешёвое хранилище

**Пример из практики:** пользователи загружают фотографии профиля. Куда их хранить?

Не в базу данных (BLOB-поля раздувают БД и замедляют бэкапы). Правильный подход: загрузить в S3, сохранить в БД только URL: `https://cdn.example.com/avatars/user_123.jpg`.

## Классы хранения (Storage Classes)

Облачные провайдеры предлагают разные классы по цене и скорости доступа:

| Класс (AWS S3) | Доступ | Цена | Для чего |
|----------------|--------|------|----------|
| **Standard** | Мгновенный | $$$ | Активные данные |
| **Infrequent Access** | Мгновенный, но дороже за чтение | $$ | Данные, нужные раз в месяц |
| **Glacier** | Минуты-часы | $ | Архивы, compliance |
| **Glacier Deep Archive** | До 12 часов | ¢ | Долгосрочное хранение (7+ лет) |

**Задача аналитика:** определить политику жизненного цикла данных. Например: «Логи за последний месяц — Standard, за последний год — IA, старше года — Glacier».

## Data Lake: хранилище для всего

Data Lake — концепция, где все данные организации хранятся в одном месте (обычно в объектном хранилище) в исходном формате.

Структура Data Lake обычно включает зоны:
- **Raw (Landing)** — данные как есть из источников
- **Cleaned (Staging)** — очищенные и валидированные
- **Curated (Analytics)** — подготовленные для аналитики

Аналитик участвует в проектировании структуры зон и определении правил перемещения данных между ними.
