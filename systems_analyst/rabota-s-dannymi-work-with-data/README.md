| Грейд | Роль |

> Данные — центральный объект работы системного аналитика. Умение понимать, как данные хранятся, передаются и преобразуются, отличает сильного SA от «рисователя схем».

## Зачем аналитику разбираться в данных

Системный аналитик не пишет SQL-запросы в продакшен и не настраивает ETL-пайплайны. Но он принимает решения, которые определяют, как данные будут жить в системе: какую модель выбрать, в каком формате передавать между сервисами, где хранить и как визуализировать для бизнеса.

Типичные ситуации, в которых аналитику нужны знания о данных:

- **Проектирование новой фичи** — нужно определить, какие сущности появятся в БД, какие связи между ними и какие атрибуты хранить.
- **Интеграция систем** — выбрать формат обмена (JSON, XML, CSV), понять ограничения источника данных, спроектировать маппинг полей.
- **Аналитические задачи** — помочь бизнесу сформулировать, какие данные нужны для отчёта, и объяснить команде разработки, откуда их взять.
- **Оптимизация производительности** — понять, почему запрос работает медленно, и предложить индексы или денормализацию.
- **Data Quality** — выявить проблемы с качеством данных: дубликаты, пустые поля, несогласованные форматы.

## Карта компетенций по грейдам

**🟢 Junior** — знает базовый SQL (SELECT, JOIN, WHERE, GROUP BY), различает JSON и XML, понимает что такое таблица и первичный ключ.

**🔵 Middle** — проектирует ER-модели из 10–20 таблиц, пишет сложные SQL-запросы с подзапросами и оконными функциями, понимает разницу между OLTP и OLAP, знает когда использовать NoSQL вместо реляционной БД.

**🟠 Senior** — проектирует модели данных для сложных доменов, понимает нормализацию/денормализацию как осознанный выбор, работает с ETL-архитектурой, разбирается в Data Warehouse и Data Lake.

**🔴 Senior+** — определяет стратегию данных для продукта, проектирует Data Mesh / Data Governance подходы, участвует в выборе технологического стека для работы с данными.
