> Apache Airflow — стандартный инструмент оркестрации данных. Аналитику не нужно писать DAG-и на Python, но важно понимать ключевые концепции, чтобы грамотно формулировать требования к пайплайнам и обсуждать их с командой.

## Что такое Apache Airflow

Airflow — это платформа для создания, планирования и мониторинга рабочих процессов (workflows). Создан в Airbnb, сейчас — проект Apache Software Foundation. Основная идея: описать последовательность задач как код (Python) и запускать их по расписанию.

## Ключевые концепции

| Концепция | Что это | Аналогия |
|-----------|---------|----------|
| **DAG** (Directed Acyclic Graph) | Граф задач без циклов — описывает «что за чем выполнять» | Рецепт: сначала нарезать, потом обжарить, потом подать |
| **Task** | Отдельная единица работы в DAG | Один шаг рецепта |
| **Operator** | Тип задачи: SQL-запрос, Python-скрипт, HTTP-вызов | Инструмент для шага: нож, сковорода |
| **Schedule** | Расписание запуска DAG (cron-выражение) | «Каждый день в 3:00» |
| **Sensor** | Задача, которая ждёт внешнее событие | «Подожди, пока файл появится в S3» |
| **XCom** | Механизм передачи данных между задачами | Передать результат одного шага на вход другому |
| **Connection** | Настроенное подключение к внешней системе (БД, API, S3) | Сохранённые реквизиты доступа |

## Пример DAG глазами аналитика

```
[Extract from CRM] → [Extract from billing] → [Transform & join] → [Load to DWH] → [Update dashboard]
```

Аналитик описывает это в требованиях:
- **Источники:** CRM (PostgreSQL), Billing (API)
- **Расписание:** ежедневно в 02:00 MSK
- **Зависимости:** трансформация начинается только после успешного извлечения из обоих источников
- **SLA:** данные должны быть в DWH к 06:00
- **Алерт:** если DAG не завершился до 05:30 — уведомление в Slack

## Что аналитик должен уметь в контексте Airflow

| Навык | Зачем |
|-------|-------|
| Читать DAG в UI Airflow | Понимать статус пайплайна: что выполнилось, что упало |
| Описать зависимости задач | «Сначала загрузи справочники, потом факты» |
| Определить расписание | Понимать cron-выражения: `0 2 * * *` = каждый день в 2:00 |
| Сформулировать SLA | «Данные за вчера доступны к 8:00 для дашборда» |
| Описать retry-стратегию | «При ошибке — 3 повтора с интервалом 10 минут, потом алерт» |
| Сформулировать проверки качества | «После загрузки проверить: count > 0, нет NULL в ключевых полях» |

## Airflow vs альтернативы

| Инструмент | Особенности | Когда выбирать |
|------------|------------|----------------|
| **Apache Airflow** | Python, гибкий, стандарт де-факто | Большинство команд с инженерами |
| **Prefect** | Современный Python-оркестратор | Команды, ищущие альтернативу Airflow |
| **Dagster** | Data-aware, встроенные assets | Фокус на data quality |
| **dbt** | SQL-трансформации, не оркестратор | ELT в DWH, часто работает поверх Airflow |
