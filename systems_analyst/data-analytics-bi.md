# Data Analytics и BI (Business Intelligence)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️🅰️ BA+SA |

> Data Analytics и BI преобразуют данные в знания для принятия решений. Аналитику важно понимать типы анализа, метрики и архитектуру BI-систем.

## Разница между Data Analytics и BI

### Business Intelligence (BI)

**BI** — это процесс сбора, хранения, анализа и визуализации данных для поддержки бизнес-решений.

**Компоненты:**
- Data Warehouse — централизованный репозиторий данных
- ETL процессы — загрузка и трансформация данных
- Дашборды и отчеты — визуализация для пользователей

**Для аналитика:** BI отвечает на вопрос "Какое текущее состояние бизнеса?"

### Data Analytics

**Analytics** — это наука анализа данных для выявления закономерностей, тенденций и взаимосвязей.

**Направления:**
- **Descriptive:** Что произошло? (описание прошлого)
- **Diagnostic:** Почему это произошло? (анализ причин)
- **Predictive:** Что произойдет? (прогнозирование)
- **Prescriptive:** Что нам делать? (рекомендации)

**Для аналитика:** Analytics отвечает на вопрос "Почему это произошло и что делать?"

## Типичная архитектура BI

```
Источники данных
  ├─ Веб-приложение (логи, события)
  ├─ CRM система (контакты, сделки)
  ├─ ERP система (заказы, инвентарь)
  └─ Платежная система (транзакции)
    ↓ (ETL процесс)
Data Warehouse (PostgreSQL, BigQuery)
  ├─ Таблица: events
  ├─ Таблица: customers
  ├─ Таблица: transactions
  └─ Таблица: products
    ↓ (запросы SQL / аналитика)
BI инструмент (Tableau, Power BI, Metabase)
    ↓ (визуализация)
Дашборды и отчеты для менеджментов
```

## Ключевые метрики и KPI

### Метрики

**Метрика** — это измеримый показатель деятельности.

**Примеры:**
- Revenue (выручка)
- Active users (активные пользователи)
- Conversion rate (процент конверсии)
- Customer lifetime value (стоимость клиента за время жизни)

### KPI (Key Performance Indicators)

**KPI** — это метрика, критичная для достижения стратегических целей.

**Критерии хорошего KPI (SMART):**
- **Specific** — четко определен
- **Measurable** — есть числовое выражение
- **Achievable** — реалистичен
- **Relevant** — важен для бизнеса
- **Time-bound** — ограничен по времени

**Пример плохого KPI:**
```
"Система должна быть быстрой"
```

**Пример хорошего KPI:**
```
"P95 response time < 200ms"
"Uptime 99.9% per month"
"Page load time < 2 seconds"
```

## Типы дашбордов

### Executive Dashboard

Для руководства — высокоуровневые метрики.

```
Метрики:
- Месячная выручка: $250,000
- Количество активных пользователей: 15,000
- Среднее значение заказа: $45
- Чистая прибыль: $50,000
```

### Operational Dashboard

Для команд — текущие операционные метрики.

```
Метрики:
- Заказов сегодня: 250
- Средний time-to-delivery: 2 часа
- % заказов с ошибками: 0.5%
- Аптайм сервиса: 99.98%
```

### Analytical Dashboard

Для аналитиков — детальный анализ.

```
Анализ:
- Когортный анализ (когда пришли пользователи, когда ушли)
- Фанел-анализ (откуда выпадают пользователи)
- A/B тесты результаты
- Корреляции между событиями
```

## Data Governance и качество

### Data Lineage

**Data Lineage** — отслеживание происхождения данных: откуда пришли, как трансформировались, где используются.

**Пример:**
```
CRM система → API → ETL процесс → Data Warehouse
                      ↓
                   Трансформация: очистка, нормализация
                      ↓
                   Таблица: customers (100000 строк)
                      ↓ (запросы)
                   Dashboard: Customer Segmentation
```

**Для аналитика:** отслеживание обеспечивает понимание качества данных и причин ошибок.

### Data Quality Dimensions

| Измерение | Описание | Пример проблемы |
|-----------|---------|-----------------|
| **Accuracy** | Соответствие действительности | Email с опечатками |
| **Completeness** | Наличие всех значений | Пустое поле "phone" |
| **Consistency** | Единообразность формата | "John Doe" vs "john doe" |
| **Timeliness** | Актуальность | Данные из вчера в системе сегодня |
| **Validity** | Соответствие правилам | Email без @ |
| **Uniqueness** | Отсутствие дублей | Один пользователь в системе 3 раза |

## Real-time Analytics

### Streaming vs Batch

**Batch (периодическая обработка):**
```
Каждый день в 2:00 ночи:
1. Загрузить все данные за день
2. Обработать (агрегировать, очистить)
3. Загрузить в BI систему
Результат: данные свежие раз в день
```

**Streaming (обработка в реальном времени):**
```
При каждом событии (пользователь кликнул):
1. Событие отправляется в систему обработки
2. Обработка в реальном времени (< 1 сек)
3. Обновление дашборда с задержкой < 1 сек
Результат: данные свежие всегда
```

**Когда использовать:**
- Batch: для большинства случаев (дешевле, проще)
- Streaming: для метрик реального времени (мониторинг, live дашборды)

## Практические требования для аналитика

### Требование 1: Дашборд для руководства

```
Дашборд: Monthly Sales Dashboard

Метрики:
1. Total Revenue
   - Текущий месяц vs прошлый месяц (% изменение)
   - Цель месяца (progress bar)

2. Orders Count
   - Количество заказов по дням (линейный график)
   - Среднее значение заказа

3. Top Products
   - 5 топ продуктов по продажам (bar chart)

4. Customer Segments
   - Распределение по типам клиентов (pie chart)

Обновление: каждый час
Пользователи: CEO, CFO, Sales Director
```

### Требование 2: A/B Test аналитика

```
A/B тест: Checkout page redesign

Гипотеза: Новый дизайн улучшит конверсию на 5%

Метрики для отслеживания:
- Conversion rate (% завершивших покупку)
- Average order value (средний чек)
- Cart abandonment rate (% покинувших тележку)
- Time to checkout (среднее время оформления)

Статистика:
- Размер выборки: 5000 пользователей в каждой группе
- Длительность: 2 недели
- Уровень значимости: 95% confidence

Результат интерпретация:
- Если p-value < 0.05 → результат значимый
- Если конверсия группа B > группа A на 5% → запустить новый дизайн
```

## ETL процессы

**ETL** = Extract, Transform, Load

```
Шаг 1: Extract (извлечение)
- Загрузить данные из CRM API

Шаг 2: Transform (трансформация)
- Очистить: удалить дубли, заполнить пустые значения
- Нормализовать: привести к единому формату
- Обогатить: добавить новые поля из других источников

Шаг 3: Load (загрузка)
- Загрузить в Data Warehouse
```

**Требование для аналитика:**
```
ETL процесс: CRM to Data Warehouse

Frequency: Ежедневно в 3:00 ночи
Источник: CRM API
Целевая таблица: DW.crm_customers

Логика:
1. Загрузить все контакты из CRM, обновленные за последние 24 часа
2. Трансформация:
   - Дублеты по email: оставить самый новый
   - Пустые phone: заполнить на основе других контактов
   - Нормализация: lowercase email, remove spaces
3. Загрузить в DW
4. Логирование: количество загруженных, дубликатов, ошибок

SLA:
- Процесс должен завершиться за < 1 часа
- Если ошибка → отправить email инженеру
```

## Рекомендации для аналитика

1. **Определить бизнес-вопросы** — какие решения нужно принимать на основе данных?
2. **Выбрать метрики** — какие показатели будут отслеживать успех?
3. **Спроектировать дашборды** — какая информация нужна, в каком формате?
4. **Спланировать данные** — где они будут храниться, как свежие?
5. **Документировать** — как рассчитываются метрики, какие их ограничения?
