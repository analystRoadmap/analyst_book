## Что такое Git и почему аналитику это нужно

**Git** — это распределенная система контроля версий, которая отслеживает изменения в файлах, позволяет возвращаться к предыдущим версиям и одновременно работать нескольким людям без конфликтов.

Для разработчиков Git — инструмент управления кодом. Для аналитика это инструмент управления знаниями, требованиями и документацией.

## Использование Git для аналитика

### Управление версиями документации

При работе над требованиями создается много артефактов: спецификации, диаграммы, макеты, примеры данных. Git позволяет:

- **Отслеживать изменения** — видеть, кто, когда и почему изменил требование
- **Вернуться к старой версии** — если новая версия оказалась хуже
- **Сравнивать версии** — понять разницу между редакциями
- **Добавлять комментарии** — через pull requests обсуждать изменения

Пример: требование к API менялось трижды. Git покажет всю историю и контекст каждого изменения.

### Коллаборация в команде

Git позволяет нескольким аналитикам работать над одним проектом параллельно:

- **Ветвление** — каждый аналитик работает в своей ветке
- **Слияние** — изменения объединяются без потери данных
- **Разрешение конфликтов** — если два аналитика изменили один файл, можно выбрать правильный вариант

Пример: один аналитик работает над требованиями к аутентификации, другой — к платежам. Обе ветки развиваются параллельно и потом сливаются.

### Изоляция и откат

Ветвление в Git позволяет тестировать новые идеи без риска повредить основную версию:

- **Feature branch** — создаем ветку для экспериментов
- **Тестирование** — проверяем идею
- **Откат** — если не подошло, удаляем ветку, основная версия не затронута

Пример: при анализе требований к новой функции создаем ветку, уточняем спецификацию с пользователем. Если требование отклонено, ветка удаляется.

### Аудит и анализ истории

Git сохраняет полную историю изменений с автором и описанием:

```
commit abc123: Добавлен функционал фильтрации по дате
Автор: Иван Петров
Дата: 2026-04-09
```

Аналитик может:
- Понять, почему было принято то или иное решение
- Найти источник конкретного требования
- Проанализировать эволюцию функции

### Резервное копирование и восстановление

Git репозиторий можно выложить на удаленный сервер (GitHub, GitLab, Gitea), что обеспечивает:

- **Резервное копирование** — копия документации всегда есть
- **Восстановление** — в случае потери локальной копии
- **Доступность** — документация доступна из любой точки

## Практические сценарии использования для аналитика

### Сценарий 1: Управление требованиями

```
main (основная версия требований)
├── feature/authentication (требования к логину)
├── feature/payments (требования к платежам)
└── feature/reports (требования к отчетам)
```

Каждая ветка содержит требования к отдельной функции. Когда требования согласованы, ветка сливается в main.

### Сценарий 2: Документирование процессов

Аналитик документирует бизнес-процессы в виде файлов Markdown в репозитории:

```
docs/
├── business-processes/
│   ├── order-processing.md
│   ├── customer-onboarding.md
│   └── invoice-management.md
├── requirements/
│   ├── functional-requirements.md
│   └── non-functional-requirements.md
└── architecture/
    └── system-design.md
```

Каждый файл имеет историю изменений и может быть просмотрен в любой версии.

### Сценарий 3: Code Review для требований

Вместо отправки документа по email:

1. Аналитик создает Pull Request с требованиями
2. Команда оставляет комментарии и предложения
3. Аналитик учитывает замечания и обновляет документ
4. После согласия требования одобряются (merge)

## Инструменты и сервисы

### GitHub

Облачный хостинг Git репозиториев с веб-интерфейсом. Удобен для сотрудничества и имеет встроенные инструменты для управления проектами.

**Для аналитика:**
- Веб-интерфейс для просмотра файлов
- Pull requests для code review требований
- Issues для отслеживания задач
- Wiki для документации

### GitLab

Аналог GitHub с поддержкой самохостинга. Часто используется в корпоративных окружениях.

**Для аналитика:**
- Более мощный управляемый проектами модуль
- Встроенная CI/CD для документации
- Лучшая поддержка больших организаций

### Gitbook

Специализированная платформа для создания документации на основе Git репозитория. Автоматически конвертирует файлы Markdown в красивую веб-книгу.

**Для аналитика:**
- Красивый веб-интерфейс для документации
- Поиск по всей документации
- Версионирование документов
- Удобное чтение на мобильных устройствах

## Минимальные Git навыки для аналитика

| Навык | Применение |
|-------|-----------|
| `git clone` | Загрузить репозиторий с требованиями |
| `git status` | Узнать, какие файлы изменились |
| `git add` / `git commit` | Сохранить изменения требований |
| `git push` | Загрузить изменения на сервер |
| `git pull` | Загрузить изменения от коллег |
| `git checkout -b` | Создать новую ветку для работы |
| `git log` | Просмотреть историю изменений |

Не нужно быть экспертом в Git, но эти 8 команд позволяют эффективно работать с документацией.

## Рекомендации

1. **Попросите код** — если ваш проект в Git, попросите доступ и начните использовать
2. **Документируйте в Markdown** — простой текстовый формат, удобный для Git
3. **Пишите понятные commit messages** — описывайте, что вы изменили и почему
4. **Используйте ветки** — не работайте напрямую в main, создавайте feature branches
5. **Запросите code review** — пусть коллеги проверят ваши требования перед merge
6. **Изучайте историю** — смотрите git log, чтобы понять, как эволюционировали требования
