# Backend (серверная часть)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ SA |

> Backend — это сердце приложения. Аналитику важно понимать архитектуру, ограничения и возможности серверной части при формулировании требований.

## Что такое Backend

**Backend** (серверная часть) — это невидимая для пользователя часть приложения, которая отвечает за обработку данных, бизнес-логику, хранилище информации и взаимодействие с внешними системами.

Если frontend — это витрина, то backend — это склад и цех производства.

## Основные функции Backend

### Управление данными

Backend отвечает за:
- **Сохранение данных** в базах данных
- **Чтение данных** по запросам
- **Обновление данных** при изменениях
- **Удаление устаревших данных**

Пример: когда пользователь создает заказ, frontend отправляет данные на backend, а backend сохраняет их в базу.

### Бизнес-логика

Backend реализует правила бизнеса:

```
Когда пользователь делает заказ на сумму > 10000 рублей:
→ Проверить, есть ли скидка за оптовый заказ
→ Применить скидку
→ Проверить наличие товара на складе
→ Зарезервировать товар
→ Создать заказ в системе
→ Отправить уведомление в логистику
```

Аналитик должен четко описать эти правила, а backend их реализует.

### API (Application Programming Interface)

Backend предоставляет API — набор правил, по которым frontend и внешние системы общаются с backend.

Типичный REST API выглядит так:

```
GET /api/orders/123          → получить заказ 123
POST /api/orders             → создать новый заказ
PUT /api/orders/123          → обновить заказ 123
DELETE /api/orders/123       → удалить заказ 123
```

Аналитик должен определить, какие операции нужны, какие параметры они принимают и какие ошибки могут вернуть.

### Аутентификация и безопасность

Backend отвечает за:
- **Проверку пароля** и подтверждение личности пользователя
- **Контроль доступа** — что может делать каждый пользователь
- **Защиту от атак** (SQL injection, XSS и т.д.)
- **Шифрование** конфиденциальных данных

### Интеграция с внешними сервисами

Backend может взаимодействовать с:
- **Платежными системами** — обработка платежей
- **Email сервисами** — отправка уведомлений
- **SMS провайдерами** — отправка смс
- **API других компаний** — получение данных курсов валют, погоды и т.д.

### Производительность и масштабируемость

Backend должен быть оптимизирован:
- **Кэширование** — сохранять часто используемые данные в памяти
- **Индексы в БД** — ускорять поиск по большим таблицам
- **Масштабирование** — обработка растущей нагрузки

## Backend для аналитика: практический взгляд

### Пример 1: Требование к API эндпоинту

Неправильно:
```
Требование: Система должна позволить получить информацию о пользователе
```

Правильно:
```
Эндпоинт: GET /api/v2/users/{user_id}
Входные параметры: user_id (целое число, обязательно)
Выходные данные:
- id (целое число)
- name (строка, макс 255 символов)
- email (email)
- created_at (дата в ISO 8601)
- status (enum: active/inactive/suspended)
Ошибки:
- 401: пользователь не авторизован
- 403: доступ запрещен
- 404: пользователь не найден
Время ответа (P95): < 200ms
Rate limit: 100 запросов в минуту на пользователя
```

### Пример 2: Требование к интеграции

Вместо: "Система должна интегрироваться с платежной системой"

Напишите:
```
Интеграция: Payment Gateway Stripe
Операции:
1. Создать платеж (amount, currency, customer_id)
   → Stripe API returns transaction_id
   → Сохранить в БД
   → Отправить email-подтверждение
2. Отменить платеж (transaction_id)
   → Отправить refund в Stripe
   → Обновить статус заказа
   → Вернуть деньги клиенту
Обработка ошибок:
- Если платеж отклонен → показать ошибку пользователю
- Если timeout (>30 сек) → повторить (не более 3 раз)
- Если успешно → отправить квитанцию
Требования:
- Все данные платежа должны быть зашифрованы
- Хранить только токен, не номер карты
```

## Сравнение Backend фреймворков

| Framework | Язык | Когда использовать | Особенности |
|-----------|------|-------------------|------------|
| **Express.js** | JavaScript | Веб-приложения, API | Простой, гибкий, большое сообщество |
| **Spring Boot** | Java | Корпоративные системы | Надежный, много готовых компонентов |
| **Django** | Python | Быстрая разработка | Все встроено, минимум кода |
| **Go (Gin/Echo)** | Go | Высоконагруженные API | Простой, быстрый, параллелизм |
| **.NET Core** | C# | Enterprise, Microsoft-экосистема | Мощный, интегрирован с Azure |

Для аналитика важно знать, на чем написан backend вашего проекта и его основные ограничения.

## Типичные проблемы Backend и как их предотвратить

### Проблема 1: N+1 queries

При получении списка заказов вместо одного запроса к БД выполняются сотни:

```
Получить 100 заказов = 1 запрос
Для каждого заказа получить товары = 100 запросов
Итого: 101 запрос вместо 2-3
Время ответа: 5 сек вместо 50ms
```

**Решение:** аналитик должен указать в требованиях, какие данные нужны, а backend оптимизирует запросы.

### Проблема 2: Отсутствие кэширования

Если одни и те же данные (каталог товаров, конфиги) запрашиваются тысячи раз в день из БД, это замораживает БД.

**Решение:** аналитик указывает, какие данные изменяются редко и могут кэшироваться.

### Проблема 3: Слабая обработка ошибок

Если сервис платежей недоступен, что произойдет? Рушится ли система?

**Решение:** аналитик описывает сценарии отказа и как система должна себя вести.

## Нефункциональные требования для Backend

| Требование | Пример | Важность |
|-----------|--------|----------|
| **Производительность** | Ответ < 200ms (P95) | Критична |
| **Доступность** | Uptime 99.9% | Критична |
| **Масштабируемость** | Обработка 10000 req/sec | Высока |
| **Надежность** | Retry, graceful degradation | Высока |
| **Безопасность** | HTTPS, аутентификация, rate limiting | Критична |
| **Логирование** | Все операции должны логироваться | Высока |
| **Мониторинг** | Алерты на ошибки | Высока |

## Взаимодействие аналитика с Backend-разработчиком

**Аналитик должен знать:**
- Текущая архитектура (монолит, микросервисы)
- Основные технологии (какой язык, фреймворк, БД)
- Ограничения (макс размер запроса, timeout, rate limits)
- Сроки разработки для разной сложности функций

**Разработчик должен уметь объяснить:**
- Почему невозможна та или иная функция
- Какие технические альтернативы есть
- Как архитектурное решение влияет на требования

## Рекомендации для аналитика

1. **Поговорите с backend-архитектором** — поймите текущую архитектуру
2. **Посетите technical standup** — слушайте, как разработчики обсуждают проблемы
3. **Изучите текущее API** — посмотрите, как устроены существующие эндпоинты
4. **Будьте конкретны в требованиях** — не "система должна быть быстрой", а "P95 < 200ms"
5. **Обсуждайте trade-offs** — быстрое ≠ надежное, масштабируемое ≠ простое
