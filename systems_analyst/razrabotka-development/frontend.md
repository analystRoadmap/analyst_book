| Грейд | Роль |
|-------|------|

> Frontend создает впечатление о приложении. Аналитику важно понимать возможности и ограничения веб- и мобильной разработки при определении требований к интерфейсу.

## Что такое Frontend

**Frontend** (клиентская часть) — это всё, что видит пользователь: интерфейс, кнопки, текст, анимации, отзывчивость на действия. Frontend работает в браузере (веб) или мобильном приложении.

Если backend — это мозг, то frontend — это лицо приложения.

## Основные компоненты Frontend

### HTML (разметка)

HTML определяет структуру страницы: где находится заголовок, список, форма, кнопка.

```html
<h1>Каталог товаров</h1>
<div class="product-list">
  <div class="product">
    <h2>Товар 1</h2>
    <p>Цена: 100 рублей</p>
    <button>Купить</button>
  </div>
</div>
```

Аналитик должен понимать, что HTML — это просто структура, без стилей и интерактивности это выглядит неприглядно.

### CSS (оформление)

CSS определяет внешний вид: цвета, размеры, расположение, шрифты, анимации.

```css
.product {
  border: 1px solid #ccc;
  padding: 20px;
  border-radius: 8px;
}
.product h2 {
  color: #333;
  font-size: 18px;
}
button {
  background-color: #007bff;
  color: white;
  padding: 10px 20px;
}
```

Красивый CSS превращает скучный HTML в привлекательный интерфейс.

### JavaScript (интерактивность)

JavaScript добавляет поведение: реакция на клики, валидация форм, анимации, загрузка данных с сервера.

```javascript
document.querySelector('button').addEventListener('click', function() {
  // Отправить запрос на сервер
  fetch('/api/cart/add', {
    method: 'POST',
    body: JSON.stringify({ product_id: 123 })
  })
  .then(response => response.json())
  .then(data => {
    alert('Товар добавлен в корзину!');
  });
});
```

## Frontend в разных контекстах

### Веб-приложение (браузер)

**Стеки:**
- **React** (Facebook) — самый популярный, очень гибкий
- **Vue** (China) — простой и элегантный
- **Angular** (Google) — мощный, для больших приложений

**Ограничения:**
- Работает только в браузере, зависит от скорости интернета
- Разные браузеры могут отличаться
- Проблемы с SEO (поиск может не индексировать)

**Когда использовать:**
- Веб-сайты, веб-приложения
- Когда нужна кроссплатформенность

### Мобильное приложение (iOS/Android)

**Подходы:**
- **Native** (Swift для iOS, Kotlin для Android) — быстро и надежно, но дорого
- **Cross-platform** (React Native, Flutter) — один код для обеих платформ

**Ограничения:**
- Мобильные процессоры слабее, батарея ограничена
- Экран маленький, сложнее с информацией
- Нужны разные версии для iOS и Android

**Когда использовать:**
- Когда нужна работа без интернета
- Когда нужна высокая производительность
- Когда нужны уведомления в реальном времени

### Прогрессивное веб-приложение (PWA)

Гибрид веб и мобильного: работает в браузере, но как приложение.

## Frontend для аналитика: практический взгляд

### Пример 1: Требование к интерфейсу

Неправильно:
```
Требование: Красивый каталог товаров
```

Правильно:
```
Компонент: Каталог товаров
Функциональность:
1. Отобразить список товаров с фото, названием, ценой
2. Фильтры: по категории, цене, рейтингу
3. Сортировка: по цене (↑/↓), популярности, новизне
4. Пагинация: 20 товаров на странице
5. При клике на товар → переход на страницу товара

UI требования:
- Сетка 4 колонки на desktop, 2 колонки на tablet, 1 на mobile
- Фото товара 200x200px, квадратное
- Быстрая загрузка: < 2 сек даже на 3G
- Поддержка мобильного (touch-friendly кнопки, минимум 44px высоты)

Тестирование:
- Работает в Chrome, Safari, Firefox
- Адаптивность на экранах от 320px до 2560px
- Клавиатурная навигация (Tab, Enter)
- Экранный читатель (для людей с нарушениями зрения)
```

### Пример 2: Требование к производительности

Вместо: "Приложение должно быть быстрым"

Напишите:
```
Требования к производительности:
- Начальная загрузка: < 3 сек (на 4G, первый визит)
- Повторная загрузка: < 1 сек (с кэшем)
- Время отклика на клик: < 100ms
- FCP (First Contentful Paint): < 1 сек
- LCP (Largest Contentful Paint): < 2.5 сек

Оптимизация:
- Изображения: WebP формат, сжатие, lazy loading
- Критичный CSS: встроить в HTML
- Некритичный CSS и JS: отложить загрузку
- Минификация и bundling
```

### Пример 3: Требование к доступности

```
Доступность (Accessibility):
- WCAG 2.1 Level AA compliance
- Контрастность текста: минимум 4.5:1
- Все изображения должны иметь alt text
- Форма заполняется только с клавиатуры (без мыши)
- Экранный читатель должен понимать структуру
- При ошибке в форме → сообщение об ошибке
- Автофокус на первое поле (для мобильного)
```

## Сравнение Frontend стеков

| Framework | Сложность | Производительность | Когда использовать |
|-----------|-----------|-------------------|------------------|
| **HTML + CSS + Vanilla JS** | Низкая | Отличная | Простые сайты, статичный контент |
| **jQuery + Bootstrap** | Средняя | Хорошая | Относительно простые приложения |
| **React** | Высокая | Отличная | Сложные интерфейсы, высокая интерактивность |
| **Vue** | Средняя | Отличная | Быстрая разработка, легко обучить |
| **Angular** | Очень высокая | Хорошая | Большие корпоративные приложения |

## Ограничения Frontend разработки

### Браузерная совместимость

Не все браузеры поддерживают новые возможности. Если нужна поддержка IE11, возможности сильно ограничены.

**Совет:** аналитик должен уточнить, какие браузеры нужно поддерживать.

### Производительность мобильных

Мобильные процессоры в 5-10 раз слабее настольных. Анимация, которая плавная на desktop, может тормозить на мобильном.

**Совет:** тестируйте требования на реальных мобильных устройствах.

### Сетевые ограничения

Мобильные пользователи могут быть на 3G или медленном 4G. 10MB приложение загружается > 30 сек.

**Совет:** требуйте оптимизацию размера и кэширование данных.

### Touch vs Mouse

На мобильном нет мыши, есть касание пальцем. Это требует разных подходов к интерфейсу.

**Совет:** при проектировании UI учитывайте touch-friendly размеры кнопок (минимум 44x44px).

## Типичные проблемы Frontend

| Проблема | Симптом | Как избежать |
|----------|---------|------------|
| **Плохая производительность** | Приложение зависает | Требуйте performance бюджеты |
| **Несовместимость браузеров** | На Safari работает, на Chrome нет | Тестируйте во всех браузерах |
| **Плохая адаптивность** | На мобильном невозможно пользоваться | Требуйте mobile-first подход |
| **Недоступность** | Слепой человек не может пользоваться | Требуйте WCAG 2.1 AA compliance |
| **Утечки памяти** | Приложение становится медленнее со временем | Требуйте тестирование на утечки |

## Взаимодействие аналитика с Frontend-разработчиком

**Вопросы, которые стоит задать:**
- Какой фреймворк использует проект?
- Какие браузеры нужно поддерживать?
- Какие мобильные устройства приоритетные?
- Каковы ограничения по размеру бандла?
- Как организуется состояние (Redux, MobX, Context)?

**Рекомендации при постановке требований:**
1. **Будьте конкретны** — не "красивый", а описывайте, как должно выглядеть
2. **Тестируйте на мобильном** — даже если требование работает на desktop
3. **Учитывайте доступность** — это нужно не только юридически, это помогает всем
4. **Обсуждайте производительность** — высокая интерактивность требует оптимизации
5. **Взаимодействуйте с UX/UI дизайнером** — макеты должны быть реализуемы

## Рекомендации для аналитика

1. **Изучите базовый HTML/CSS** — понимание структуры поможет в работе
2. **Используйте браузерные инструменты** — Inspector, Network tab
3. **Тестируйте на реальных устройствах** — не только браузер на компьютере
4. **Читайте про Web Performance** — производительность критична
5. **Следите за стандартами** — WCAG, SEO, PWA лучшие практики
