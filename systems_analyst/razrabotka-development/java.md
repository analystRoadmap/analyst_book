## Когда аналитику нужны знания Java

Java широко распространена в:
- **Банках и финансовых организациях** — надежность и масштабируемость критичны
- **Крупных корпоративных системах** — ERP, CRM, системы управления
- **Микросервисной архитектуре** — Spring Boot позволяет писать микросервисы

Если ваш проект использует Java, важно понимать её особенности.

## Основные концепции Java для аналитика

### Spring Framework

**Spring** — это фреймворк, который делает разработку на Java проще. Spring Boot — это quickstart для Spring, который позволяет за несколько минут поднять микросервис.

**Что должен знать аналитик:**
- Spring позволяет разработчикам быстро создавать REST API
- Концепция **dependency injection** — компоненты автоматически подключаются
- **Аннотации** (@RestController, @Service, @Repository) — разработчики не пишут много boilerplate кода

### REST контроллеры

В Spring REST контроллеры выглядят так:

```java
@RestController
@RequestMapping("/api/products")
public class ProductController {

  @GetMapping("/{id}")
  public ProductDTO getProduct(@PathVariable Long id) {
    // Получить товар из БД
    return productService.findById(id);
  }

  @PostMapping
  public ProductDTO createProduct(@RequestBody CreateProductRequest req) {
    // Создать товар
    return productService.create(req.getName(), req.getPrice());
  }
}
```

Аналитик должен понимать, что это — описание эндпоинтов. При требовании к API нужно указать метод (GET/POST/PUT/DELETE), путь и параметры.

### Сборка проекта

Java проекты собираются с помощью **Maven** или **Gradle**:

```
mvn build     → скомпилировать код
mvn test      → запустить тесты
mvn package   → создать JAR файл
```

**Для аналитика важно:** понимать, что разработчик не может просто внести изменение и запустить. Нужно перекомпилировать, пересобрать, переразвернуть. Поэтому требования должны быть четкими и протестированными, чтобы минимизировать количество переборок.

## Java микросервисы

Типичный микросервис на Spring Boot:
- **Один сервис** = одна бизнес-функция (например, управление товарами)
- **Независимое развертывание** — обновление одного микросервиса не трогает другие
- **REST API** для коммуникации между сервисами
- **Собственная БД** — каждый сервис имеет свою базу данных

**Для аналитика:** при проектировании микросервисов нужно четко определить границы между сервисами и то, как они коммуницируют.

## Типичные требования к Java-сервисам

### Требование 1: CRUD операции для сущности

```
Микросервис: Product Service
Операции:
1. Создать товар
   Эндпоинт: POST /api/v2/products
   Тело: { name, price, description, category_id }
   Возврат: { id, name, price, ... }
   Статусы: 201 (успешно), 400 (ошибка валидации), 409 (товар уже существует)

2. Получить товар
   Эндпоинт: GET /api/v2/products/{product_id}
   Возврат: { id, name, price, description, category_id, created_at, updated_at }
   Статусы: 200 (успешно), 404 (не найден)

3. Обновить товар
   Эндпоинт: PUT /api/v2/products/{product_id}
   Тело: { name, price, description, category_id } (все поля опциональны)
   Возврат: обновленный товар
   Статусы: 200 (успешно), 404 (не найден), 400 (ошибка валидации)

4. Удалить товар
   Эндпоинт: DELETE /api/v2/products/{product_id}
   Возврат: 204 (успешно), 404 (не найден)
```

### Требование 2: Асинхронная обработка

```
Процесс: Генерация отчета
1. POST /api/v2/reports/generate
   Параметры: { start_date, end_date, format }
   Возврат: { report_id, status: "processing" }
   Статус: 202 (Accepted)

2. Сервис запускает фоновую задачу:
   - Загрузить данные за период
   - Скомпилировать отчет
   - Сохранить в хранилище
   - Отправить email пользователю с ссылкой на отчет

3. GET /api/v2/reports/{report_id}/status
   Возврат: { status: "completed", download_url, ... }
```

## Интеграция Java микросервисов

Когда несколько Java микросервисов работают вместе:

```
Order Service (заказы)
    ↓ (вызывает)
Product Service (товары)
    ↓ (вызывает)
Inventory Service (склад)
```

**Требования для аналитика:**
- Описать, какие сервисы взаимодействуют
- Указать, что происходит при отказе одного сервиса (circuit breaker, retry)
- Определить timeout-ы (если Product Service не отвечает 5 сек, что происходит?)
- Обсудить компенсирующие транзакции (если платеж прошел, но товар не зарезервировался)

## Типичные проблемы Java разработчики решают

| Проблема | Как обсуждать с аналитиком |
|----------|---------------------------|
| **Давление памяти при большой нагрузке** | "Сколько одновременных пользователей?", "Какие данные кэшировать?" |
| **Версионирование API** | "Какие старые версии API нужно поддерживать?", "Как отключать старые версии?" |
| **Обработка ошибок** | "Что делать при network timeout?", "Повторять запрос или вернуть ошибку?" |
| **Логирование** | "Какая информация должна логироваться?", "На каком уровне (INFO, WARN, ERROR)?" |

## Инструменты для коммуникации

Когда говорите с Java разработчиком:

- **Будьте конкретны**: не "быстро", а "P95 < 200ms"
- **Используйте OpenAPI/Swagger**: явное описание API вместо слов
- **Спросите про ограничения**: какой максимальный размер запроса? timeout?
- **Обсудите обработку ошибок**: что происходит при отказе внешнего сервиса?
- **Документируйте решения**: почему выбрали эту архитектуру?

## Рекомендации для аналитика

1. **Посмотрите на существующий код** — посетите code review, поймите архитектуру
2. **Изучите основные паттерны** — микросервисы, REST API, circuit breaker
3. **Задавайте вопросы** — "Почему эта операция медленная?", "Какие лимиты на API?"
4. **Документируйте API** — используйте Swagger/OpenAPI для ясности
5. **Обсуждайте trade-offs** — быстро ≠ надежно, простота ≠ масштабируемость
