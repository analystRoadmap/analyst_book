> Модели сетевых протоколов описывают, как данные передаются между устройствами через несколько уровней абстракции.

## Модель OSI (Open Systems Interconnection)

Модель OSI - это семислойная концептуальная модель, разработанная Международной организацией по стандартизации (ISO) в 1984 году. Она описывает стандартный процесс передачи данных в компьютерных сетях и служит справочным материалом для понимания протоколов.

### Семь уровней OSI (от нижнего к верхнему)

| Уровень | Название | Функция | Примеры |
|---------|----------|---------|---------|
| 7 | Application | Интерфейс для приложений | HTTP, HTTPS, FTP, SMTP, SSH, DNS |
| 6 | Presentation | Кодирование, сжатие, шифрование | SSL/TLS, JPEG, GIF, PNG |
| 5 | Session | Установка и управление сеансами | RPC, NetBIOS |
| 4 | Transport | Доставка данных между приложениями | TCP, UDP, SCTP |
| 3 | Network | Маршрутизация пакетов | IP (IPv4, IPv6), ICMP, IGMP |
| 2 | Data Link | Передача между соседними узлами | Ethernet, PPP, Wi-Fi (802.11), MAC адреса |
| 1 | Physical | Физическая передача данных | Кабели, Wi-Fi волны, оптические волокна |

### Описание каждого уровня

**Физический уровень (Layer 1)**:
- Отвечает за физическую передачу битов через среду связи
- Определяет характеристики электрических сигналов (напряжение, частоту)
- Примеры: коммутационные кабели, RJ45 разъемы, оптические волокна

**Канальный уровень (Layer 2)**:
- Обеспечивает надежную передачу между соседними устройствами в одной сети (LAN)
- Использует MAC адреса (например, 00:1A:2B:3C:4D:5E)
- Примеры: Ethernet, Wi-Fi, переключатели (switches)

**Сетевой уровень (Layer 3)**:
- Маршрутизирует пакеты между сетями
- Использует IP адреса (например, 192.168.1.1)
- Примеры: маршрутизаторы, IP протокол, ICMP (ping)

**Транспортный уровень (Layer 4)**:
- Обеспечивает надежную или быструю доставку между приложениями
- TCP гарантирует доставку, UDP быстрый
- Использует номера портов (например, порт 80 для HTTP)

**Уровень сеанса (Layer 5)**:
- Устанавливает, поддерживает и завершает сеансы связи
- Управляет диалогом между приложениями

**Уровень представления (Layer 6)**:
- Переводит, сжимает и шифрует данные
- Пример: JPEG сжатие изображений, SSL/TLS шифрование

**Прикладной уровень (Layer 7)**:
- Предоставляет интерфейс для приложений
- Примеры: браузер (HTTP), почтовый клиент (SMTP), SSH клиент

## Модель TCP/IP

Модель TCP/IP - это четырехслойная модель, которая описывает реальную архитектуру интернета. Она более практична и широко используется чем OSI.

### Четыре уровня TCP/IP

| Уровень | Название | Функция | Примеры |
|---------|----------|---------|---------|
| 4 | Application | Приложения и сервисы | HTTP, HTTPS, FTP, SMTP, DNS, SSH |
| 3 | Transport | Доставка между приложениями | TCP, UDP |
| 2 | Internet | Маршрутизация | IP (IPv4, IPv6), ICMP |
| 1 | Network Access | Физическая передача | Ethernet, Wi-Fi, PPP |

## TCP vs UDP (Транспортный уровень)

| Характеристика | TCP | UDP |
|---|---|---|
| **Доставка** | Гарантированная | Не гарантированная |
| **Порядок** | Сохраняется | Не сохраняется |
| **Скорость** | Медленнее | Быстрее |
| **Контроль потока** | Да | Нет |
| **Обнаружение ошибок** | Да | Нет |
| **Используется для** | HTTP, HTTPS, SMTP, FTP | DNS, видеопотоки, онлайн игры |

**TCP (Transmission Control Protocol)**:
- Установка соединения перед передачей (Three-way handshake)
- Проверка целостности данных
- Переотправка потерянных пакетов
- Идеален для приложений, требующих надежности

**UDP (User Datagram Protocol)**:
- Отправка данных без установления соединения
- Быстрая доставка, не ждет подтверждения
- Идеален для приложений, требующих низкой задержки

## IP адресация

**IPv4**: 32-битный адрес, написаны как четыре числа (0-255), разделенные точками
- Пример: 192.168.1.1
- Максимум 4.3 миллиарда адресов
- Сейчас почти полностью исчерпаны

**IPv6**: 128-битный адрес, написаны как восемь групп шестнадцатеричных цифр
- Пример: 2001:0db8:85a3::8a2e:0370:7334
- Максимум 3.4 × 10^38 адресов
- Будущее интернета

## Практический пример для аналитика

При анализе сетевой проблемы:

**Проблема**: "Приложение не может подключиться к внешнему API"

Проверяем по уровням:
1. **Physical (Layer 1)**: есть ли интернет соединение? (ping 8.8.8.8)
2. **Network (Layer 3)**: правильный ли IP-адрес сервера? (nslookup api.example.com)
3. **Transport (Layer 4)**: открыт ли порт? (telnet api.example.com 443)
4. **Application (Layer 7)**: правильная ли версия API? (curl -v https://api.example.com)

## Инструменты для диагностики

**ping**: проверка доступности хоста на уровне IP (Layer 3)
```
ping google.com  → проверить задержку (latency) до google.com
```

**traceroute**: показывает маршрут пакета через интернет
```
traceroute google.com  → какие маршрутизаторы пакет проходит
```

**nslookup/dig**: разрешение доменного имени в IP (Layer 7/3)
```
nslookup google.com  → какой IP-адрес у google.com
```

**telnet**: проверка открытого порта (Layer 4)
```
telnet example.com 80  → есть ли доступ к http порту
```

**netstat**: показывает активные соединения
```
netstat -an  → список всех TCP/UDP соединений
```
