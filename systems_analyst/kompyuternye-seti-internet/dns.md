# DNS (Domain Name System)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅰️ Systems Analyst |

> DNS - это система, которая преобразует доменные имена (example.com) в IP-адреса (192.168.1.1), позволяя людям использовать легкие для запоминания адреса вместо чисел.

## Почему нужен DNS?

Компьютеры в интернете идентифицируются по IP-адресам (192.168.1.1), но людям сложно запомнить числа. DNS решает эту проблему - запомнить "google.com" намного проще, чем "142.250.185.46".

DNS работает как справочник:
```
Пользователь: "Какой IP у google.com?"
DNS: "142.250.185.46"
```

## Как работает DNS?

### Процесс разрешения DNS (DNS Resolution)

1. **Браузер запрашивает**:
   ```
   Пользователь вводит: google.com
   Браузер отправляет запрос: "Какой IP у google.com?"
   ```

2. **Проверка локального кэша**:
   - Браузер проверяет свой локальный кэш DNS
   - Если найдено и не истекло (TTL), используется этот результат
   - Если нет, идет дальше

3. **Рекурсивный резолвер** (обычно ваш ISP):
   - Браузер отправляет запрос на рекурсивный резолвер
   - Резолвер не знает ответ, обращается к корневому серверу

4. **Корневой сервер** (root nameserver):
   - Знает, где находятся серверы верхнеуровневых доменов (.com, .org, .ru)
   - Отвечает: "Ищите на этом сервере"

5. **TLD сервер** (Top-Level Domain server):
   - Знает, где находятся авторитетные серверы домена google.com
   - Отвечает: "Ищите на этом сервере"

6. **Авторитетный сервер** (authoritative nameserver):
   - Знает точный IP-адрес google.com
   - Отвечает: "IP адрес: 142.250.185.46"

7. **Результат возвращается**:
   - Ответ приходит браузеру
   - Браузер кэширует результат (TTL указывает, как долго кэшировать)
   - Браузер подключается к серверу по IP-адресу

### Визуализация

```
    Браузер/Клиент
         ↓ (запрос)
    Локальный кэш (если есть, ответ)
         ↓
    Рекурсивный резолвер (ISP DNS)
         ↓
    Корневой сервер (.com, .org)
         ↓
    TLD сервер (google.com)
         ↓
    Авторитетный сервер (ns1.google.com)
         ↑
    [IP-адрес 142.250.185.46]
```

## Компоненты DNS

**DNS клиент**: браузер, приложение, которое отправляет запрос

**Рекурсивный резолвер**: обычно ваш ISP, иногда Google (8.8.8.8) или Cloudflare (1.1.1.1)
- Выполняет всю работу по разрешению имени
- Кэширует результаты

**Корневой сервер**: 13 корневых серверов по всему миру
- Знает о всех TLD (Top-Level Domain) серверах
- Отвечает за .com, .org, .net, .ru и другие

**TLD сервер**: сервер верхнеуровневого домена
- Для .com - VeriSign глобально управляет
- Знает авторитетные серверы всех .com доменов

**Авторитетный сервер**: сервер владельца домена
- Хранит записи DNS для конкретного домена (google.com)
- Владелец домена указывает эти серверы при регистрации

## DNS Записи

Основные типы DNS записей:

| Тип | Назначение | Пример |
|-----|-----------|--------|
| **A** | IPv4 адрес | google.com → 142.250.185.46 |
| **AAAA** | IPv6 адрес | google.com → 2607:f8b0:4004:813::200e |
| **CNAME** | Alias (альтернативное имя) | www.google.com → google.com |
| **MX** | Почтовый сервер | google.com → aspmx.l.google.com |
| **TXT** | Текстовая запись | Для верификации домена, SPF |
| **NS** | Nameserver | Авторитетный сервер домена |
| **SOA** | Start of Authority | Информация об авторитетном сервере |

## TTL (Time To Live)

TTL определяет, как долго DNS запись может быть кэширована:

```
google.com. 300 IN A 142.250.185.46
             ↑
          TTL=300 секунд (5 минут)
```

- **Низкий TTL** (300 секунд): частые обновления, больше запросов к DNS
- **Высокий TTL** (86400 секунд/день): меньше запросов, но изменения долго распространяются

## Кэширование DNS

**Браузерный кэш**: 1-5 минут (уменьшает нагрузку)

**Операционная система**: 1-10 минут

**ISP резолвер**: часов

**CDN резолвер** (Google, Cloudflare): часов

Каждый уровень кэширует результаты, уменьшая нагрузку на систему.

## DNS и производительность

**DNS lookup delay**: обычно 20-50ms
- При загрузке вебсайта браузер тратит 20-50ms только на разрешение имени
- С кэшем: 0ms (мгновенно)

**Оптимизация**:
- Используйте быстрый DNS (Google 8.8.8.8, Cloudflare 1.1.1.1)
- Кэшируйте DNS результаты на клиенте
- Используйте как можно меньше доменов (каждый требует отдельного lookup)

## DNS Security (DNSSEC)

**DNS Spoofing**: злоумышленник может подделать ответ DNS и перенаправить на вредоносный сайт.

**DNSSEC**: добавляет цифровые подписи к DNS ответам, предотвращая подделку.

## Практический пример для аналитика

### Диагностика проблемы "сайт недоступен"

```
nslookup example.com

Server:  8.8.8.8
Address: 8.8.8.8#53

Non-authoritative answer:
Name:   example.com
Address: 93.184.216.34
```

Если DNS разрешился успешно, проблема не в DNS.

### Изменение DNS записей

Когда компания переходит на новый хостинг:
1. Меняют A запись на новый IP-адрес
2. Старая запись имеет TTL=3600 (1 час)
3. На протяжении часа разные пользователи видят старый и новый сайт
4. Через час все видят новый сайт

## DNS для аналитика

При проектировании системы важно учитывать:
- **Глобальный DNS**: для высокой доступности иметь серверы в разных странах
- **Health Checks**: DNS может быть настроен на автоматическое переключение при сбое сервера
- **Geographic Routing**: разные пользователи могут быть направлены на разные серверы
- **Rate Limiting**: DNS резолверы могут быть целью DDoS атак
