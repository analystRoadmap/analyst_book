# Жизненный цикл программного продукта (SDLC)

| Грейд | Роль |
|-------|------|
| 🟢 Junior | 🅱️🅰️ BA+SA |

> **SDLC (Software Development Life Cycle)** — последовательность фаз разработки программного продукта, от идеи до вывода из эксплуатации, каждая из которых имеет четкие цели, задачи и результаты.

## Основные фазы SDLC

```
Идея → Анализ → Проектирование → Разработка → Тестирование →
Развертывание → Поддержка → Оптимизация → Вывод из эксплуатации
```

## Фаза 1: Инициация и Планирование

**Что происходит:**
- Определяется идея и задача
- Ставятся цели и результаты ожидания
- Собирается команда
- Планируются ресурсы и бюджет

**Роль аналитика:**
- Помочь четко сформулировать проблему
- Определить граница проекта (что входит, что не входит)
- Выявить stakeholders

**Документы:**
- Project Charter
- Высокоуровневые требования (HLR)
- Project Plan

---

## Фаза 2: Анализ требований

**Что происходит:**
- Детальное изучение потребностей
- Сбор информации от stakeholders
- Анализ текущих процессов (As-Is)

**Роль аналитика:** ОСНОВНАЯ РОЛЬ

- Проведение интервью
- Изучение текущих процессов
- Документирование требований
- Подготовка моделей и диаграмм

**Документы:**
- Business Requirements Document (BRD)
- Process Models (As-Is)
- Requirements Specification

---

## Фаза 3: Проектирование (Design)

**Что происходит:**
- Архитектор проектирует решение
- Дизайнер создает макеты интерфейса
- Определяется техническая архитектура

**Роль аналитика:**
- Валидировать, что дизайн соответствует требованиям
- Уточнять детали с дизайнером
- Подготавливать к тестированию

**Документы:**
- Design Document
- Wireframes & Mockups
- Technical Specifications
- Architecture Diagram

---

## Фаза 4: Разработка (Development)

**Что происходит:**
- Разработчики пишут код
- Интеграция компонентов
- Unit тестирование

**Роль аналитика:**
- Уточнять вопросы от разработчиков
- Подготавливать документацию для QA
- Отслеживать прогресс

---

## Фаза 5: Тестирование (Testing)

**Что происходит:**
- QA тестирует функциональность
- Проверка соответствия требованиям
- Баг репорты и фиксы

**Роль аналитика:**
- Подготовить критерии приемки
- Помочь QA понять требования
- Участвовать в UAT (User Acceptance Testing)

**Типы тестирования:**
- Unit testing (разработчик тестирует свой код)
- Integration testing (части системы работают вместе)
- System testing (вся система)
- UAT (пользователи тестируют в реальных сценариях)

---

## Фаза 6: Развертывание (Deployment)

**Что происходит:**
- Подготовка к production
- Развертывание в продакшене
- Обучение пользователей

**Роль аналитика:**
- Подготовить документацию для пользователей
- Обучать операционную команду
- Помочь с миграцией данных

---

## Фаза 7: Поддержка и Обслуживание (Support)

**Что происходит:**
- Исправление проблем в production
- Поддержка пользователей
- Мониторинг производительности

**Роль аналитика:**
- Анализировать问题 пользователей
- Определять их причины
- Предлагать решения

---

## Фаза 8: Оптимизация и Развитие (Enhancement)

**Что происходит:**
- На основе feedback улучшается продукт
- Добавляются новые функции
- Оптимизируется производительность

**Цикл повторяется для новых фич**

---

## Фаза 9: Вывод из эксплуатации (Decommissioning)

**Что происходит:**
- Продукт устаревает или заменяется
- Архивируются данные
- Коммуникация с пользователями о переходе

---

## SDLC vs Agile vs Waterfall

### Waterfall (Каскадный)

```
Требования → Дизайн → Разработка → Тестирование → Deploy
    ↓         ↓         ↓            ↓             ↓
  2 недели  1 неделя  4 недели    2 недели    1 неделя

Назад вернуться СЛОЖНО
```

**Когда использовать:**
- ✓ Требования четко определены с самого начала
- ✓ Мало вероятность изменений
- ✓ Проект с жестким deadline
- ❌ Не подходит для innovation projects

---

### Agile/Scrum (Итеративный)

```
Sprint 1: Требования → Дизайн → Разработка → Тестирование → Deploy
           (1-2 дня) (2 дня) (5 дней) (3 дня) (1 день)

Sprint 2: [Новая фича] (повторение)

Sprint 3: [Улучшения] (повторение)
```

**Когда использовать:**
- ✓ Требования могут меняться
- ✓ Нужна регулярная доставка ценности
- ✓ Обратная связь от пользователей важна
- ✓ Инновационные проекты

---

### DevOps / Continuous Deployment

```
Требования → Code → Test → Deploy (несколько раз в день!)
   (часы)    (часы) (часы)  (минуты)
```

**Когда использовать:**
- ✓ Высокая стабильность требуется
- ✓ Много small features
- ✓ Быстрая обратная связь
- ✓ Автоматизация сильна

---

## Роль аналитика на каждой фазе

| Фаза | Основные задачи |
|------|-----------------|
| **Инициация** | Определить scope, составить план анализа |
| **Анализ** | Собрать требования, моделировать процессы |
| **Проектирование** | Валидировать дизайн, подготовить к разработке |
| **Разработка** | Уточнять вопросы, отслеживать прогресс |
| **Тестирование** | Помочь QA, валидировать требования |
| **Развертывание** | Обучать пользователей, документировать |
| **Поддержка** | Анализировать issues, предлагать улучшения |

## Key Milestones

| Milestone | Когда | Что | Кто проверяет |
|-----------|-------|-----|--------------|
| **Requirements Sign-off** | Конец анализа | Требования утверждены | Client, Stakeholders |
| **Design Review** | Конец дизайна | Дизайн готов к разработке | Architect, PM |
| **Development Complete** | Конец разработки | Код готов к тестированию | Tech Lead |
| **Testing Complete** | Конец тестирования | Все баги в production готово | QA Lead |
| **Release Ready** | Перед deploy | Документы, обучение готово | PM, BA |
| **Go-Live** | Deploy day | Продукт в production | Ops, Support |

## Типичные проблемы в SDLC

❌ **Требования меняются в разработке** (нужны более четкие requirements)
❌ **Баги найдены в production** (нужна лучшая валидация требований на тестировании)
❌ **Пользователи не понимают как использовать** (нужна лучшая документация)
❌ **Проект запаздывает** (плохая оценка сложности на этапе анализа)
