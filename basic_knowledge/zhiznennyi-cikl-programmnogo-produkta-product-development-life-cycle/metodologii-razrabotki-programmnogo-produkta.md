> **Методология разработки** — набор процессов, практик и инструментов, которые команда использует для управления разработкой продукта, от планирования до развертывания.

## Основные методологии

## 1. Waterfall (Каскадный подход)

**Главная идея:** Каждая фаза завершается полностью перед началом следующей

```
┌─────────────┐
│ Requirements│
└──────┬──────┘
       ↓
┌─────────────┐
│   Design    │
└──────┬──────┘
       ↓
┌─────────────┐
│  Development│
└──────┬──────┘
       ↓
┌─────────────┐
│   Testing   │
└──────┬──────┘
       ↓
┌─────────────┐
│  Deployment │
└─────────────┘
```

**Сроки:**
- Требования: 2-4 недели
- Дизайн: 2-3 недели
- Разработка: 6-12 недель
- Тестирование: 3-4 недели
- Развертывание: 1 неделя

**Когда использовать:**
✓ Требования четко определены с самого начала
✓ Мало вероятность изменений
✓ Большие корпоративные проекты
✓ Проекты с жесткими deadline и бюджетом

**Плюсы:**
✓ Понятный и предсказуемый процесс
✓ Легко отследить прогресс
✓ Подходит для больших команд
✓ Хороша для регулируемых индустрий (финансы, медицина)

**Минусы:**
❌ Сложно менять требования на позднем этапе
❌ Баги найдены поздно (дорого исправлять)
❌ Пользователи видят результат только в конце (может не понравиться)
❌ Не гибкая к рыночным изменениям

---

## 2. V-модель

**Главная идея:** Каждой фазе разработки соответствует фаза тестирования

```
┌─────────────┐                              ┌─────────────────┐
│ Requirements├─────────────────────────────>│   UAT Testing   │
└─────┬───────┘                              └────────┬────────┘
      │                                                │
      ↓                                                ↑
┌─────────────┐                              ┌────────┴────────┐
│   Design    ├─────────────────────────────>│ System Testing  │
└─────┬───────┘                              └────────┬────────┘
      │                                                │
      ↓                                                ↑
┌─────────────┐                              ┌────────┴────────┐
│  Dev + Unit ├─────────────────────────────>│ Integration Test│
└─────────────┘                              └─────────────────┘
```

**Плюсы:**
✓ Проблемы выявляются раньше (дешевле исправлять)
✓ Лучше качество за счет интенсивного тестирования
✓ Четкая структура

**Минусы:**
❌ Все еще не гибкая
❌ Затратная (много тестирования на ранних этапах)

---

## 3. Agile (Итеративный подход)

**Главная идея:** Разработка в коротких циклах (спринты), с регулярной обратной связью

```
Sprint 1 (2 недели):
  Требования → Дизайн → Разработка → Тестирование → Demo
  [Готовый инкремент продукта]

Sprint 2 (2 недели):
  [Новые требования] → ... → Demo
  [Еще готовый инкремент]

Sprint 3 (2 недели):
  [Улучшения] → ... → Demo
```

### Scrum (самая популярная Agile методология)

**Роли:**
- **Product Owner** — приоритизирует требования
- **Scrum Master** — облегчает процесс
- **Development Team** — разрабатывает

**Артефакты:**
- **Product Backlog** — список всех требований
- **Sprint Backlog** — требования на текущий спринт
- **Increment** — готовый к использованию продукт в конце спринта

**События:**
```
Sprint Planning (2 часа)
    ↓
Daily Standup (15 минут, каждый день)
    ↓
[Работа 5-7 дней]
    ↓
Sprint Review (1 час)
    ↓
Sprint Retrospective (1 час)
    ↓
[Следующий спринт начинается]
```

**Когда использовать:**
✓ Требования могут меняться
✓ Нужна частая доставка ценности
✓ Пользователи хотят рано видеть результаты
✓ Инновационные проекты

**Плюсы:**
✓ Гибкая к изменениям
✓ Пользователи видят результаты каждые 2 недели
✓ Раньше выявляются проблемы
✓ Команда более мотивирована (видны результаты)

**Минусы:**
❌ Сложно планировать долгосрочные ресурсы
❌ Требует активного участия пользователей
❌ Может быть хаотично без хорошего управления

---

### Kanban

**Главная идея:** Непрерывный поток работы, ограничение work-in-progress

```
To Do  │  In Progress  │  In Review  │  Testing  │  Done
────────────────────────────────────────────────────────
  5    │      3        │     2       │    1      │  20
(макс 5)(макс 3)       (макс 2)     (макс 1)
```

**Плюсы:**
✓ Упрощено управление
✓ Легко приоритизировать
✓ Нет фиксированных спринтов (гибче)

**Минусы:**
❌ Нет четкого "конца" цикла
❌ Может быть сложнее предсказывать сроки

---

## 4. DevOps / Continuous Deployment

**Главная идея:** Разработка и операции интегрированы, развертывание несколько раз в день

```
Code → Test → Build → Deploy → Monitor → Feedback
(часы) (часы) (мин) (мин)  (continuous)
```

**Инструменты:**
- **CI/CD Pipeline** (Jenkins, GitHub Actions)
- **Infrastructure as Code** (Terraform)
- **Containerization** (Docker)
- **Monitoring & Logging** (Prometheus, ELK)

**Когда использовать:**
✓ Нужны частые обновления
✓ Высокая стабильность требуется
✓ Автоматизация сильна

---

## 5. Lean

**Главная идея:** Минимизировать потери, максимизировать ценность

**Принципы:**
1. Определить ценность с точки зрения клиента
2. Убрать потери (waste)
3. Создать поток
4. Дать власть команде
5. Совершенствоваться

**Фокус:** Скорость и качество с минимальными ресурсами

---

## 6. Extreme Programming (XP)

**Главная идея:** Инженерные практики выведены на максимум

**Практики:**
- **Pair Programming** — два разработчика на одном компьютере
- **Test-Driven Development** — тесты пишутся перед кодом
- **Continuous Integration** — код интегрируется несколько раз в день
- **Refactoring** — постоянное улучшение кода
- **Collective Code Ownership** — любой может изменять любой код

|----------|-----------|---------|------------|--------|
| **Четкие требования** | ✓ Да | ✓ Да | ○ Частично | ○ Нет |
| **Гибкость требований** | ❌ Нет | ❌ Нет | ✓ Да | ✓ Да |
| **Размер проекта** | ✓ Большие | ✓ Большие | ✓ Любые | ○ Средние-большие |
| **Регулируемые индустрии** | ✓ Да | ✓ Да | ○ Частично | ❌ Нет |
| **Частые релизы** | ❌ Нет | ❌ Нет | ✓ Да | ✓ Да |
| **Бюджет гибкий** | ❌ Нет | ❌ Нет | ✓ Да | ✓ Да |

## Гибридные подходы

Большинство компаний используют **комбинацию методологий:**

```
Planning Phase: Waterfall (полный анализ требований)
  ↓
Development Phase: Agile (итеративная разработка)
  ↓
Deployment Phase: DevOps (автоматизированный deploy)
```

---

## Роль аналитика в разных методологиях

### Waterfall
- **Больше документирования** (полные требования в начале)
- **Меньше контакта с разработчиками** (после анализа работает дизайнер)
- **Более прогнозируемо** (сроки фазы анализа четко определены)

### Agile
- **Continuous involvement** (участие на протяжении всех спринтов)
- **Часто меняющиеся требования** (нужно адаптироваться)
- **Более активное общение** (ежедневные стендапы)

### DevOps
- **Мониторинг продакшена** (анализ ошибок пользователей)
- **Feedback-driven анализ** (требования на основе логов и метрик)
- **Быстрые итерации** (новые требования каждый день)
