# Процесс (Process)

| Грейд | Роль |
|-------|------|
| 🟢 Junior | 🅱️🅰️ BA+SA |

> **Бизнес-процесс** — последовательность взаимосвязанных действий, выполняемых в организации для достижения определенной цели, решения проблемы или создания ценности.

## Почему аналитику нужно понимать процессы?

Аналитик работает с процессами на каждом проекте:
- **Анализирует** текущие (As-Is) бизнес-процессы
- **Проектирует** желаемые (To-Be) процессы
- **Документирует** требования для их автоматизации
- **Оптимизирует** существующие процессы

## Основные характеристики бизнес-процесса

Каждый процесс имеет:

| Элемент | Описание | Пример |
|---------|---------|--------|
| **Цель** | Для чего это нужно | Продать товар клиенту |
| **Вход** | Что запускает процесс | Заказ от клиента |
| **Выход** | Результат процесса | Доставленный товар |
| **Участники** | Кто задействован | Менеджер, Склад, Логистика |
| **Ресурсы** | Что нужно для выполнения | ПО, деньги, время |
| **Метрики** | Как измерить успех | Время выполнения, ошибки |

## Структура процесса

```
Условия начала (триггер)
         ↓
    Начало процесса
         ↓
   Шаг 1 → Шаг 2 → Шаг 3
         ↓
    Принятие решения?
    ├─ Да → Шаг 4
    └─ Нет → Шаг 5
         ↓
   Конец процесса
         ↓
Результат (выход)
```

## Типы бизнес-процессов

### 1. Основные (Core Processes)
Создают ценность для клиента
- Продажа товаров
- Оказание услуг
- Производство продукции

### 2. Поддерживающие (Support Processes)
Обеспечивают работу основных процессов
- HR (найм, зарплата, обучение)
- Финансы (бухгалтерия, платежи)
- IT (поддержка систем)

### 3. Управляющие (Management Processes)
Управляют деятельностью организации
- Планирование
- Контроль качества
- Аудит

## Жизненный цикл работы с процессами

| Фаза | Деятельность | Результат |
|------|--------------|-----------|
| **1. Discovery** | Понять текущий процесс | Диаграмма As-Is |
| **2. Analysis** | Выявить проблемы и возможности | Список улучшений |
| **3. Design** | Разработать новый процесс | Диаграмма To-Be |
| **4. Documentation** | Описать в деталях | Процедуры, инструкции |
| **5. Implementation** | Внедрить изменения | Запуск в production |
| **6. Optimization** | Совершенствовать на основе KPI | Улучшение метрик |

## Как описать процесс?

**Элементы описания:**

1. **Название** — четкое, без сокращений (вместо "ОЗ" писать "Оформление заказа")
2. **Цель** — одно предложение, почему процесс существует
3. **Участники (Роли)** — кто задействован на каждом этапе
4. **Шаги** — нумерованный список действий
5. **Условия** — когда применяются разные варианты
6. **Сроки** — как долго выполняется каждый шаг
7. **Метрики** — как измерить успех
8. **Исключения** — что делать при ошибках

## Типичные ошибки при описании процессов

❌ **Слишком общее описание** — "Делаем работу" вместо конкретных шагов
❌ **Не указаны роли** — непонятно, кто что делает
❌ **Отсутствуют исключения** — только "happy path", забывают об ошибках
❌ **Нет метрик** — непонятно, как оценить эффективность
❌ **Используют сокращения** — непонятны новым сотрудникам

## Инструменты документирования процессов

- **BPMN диаграммы** — стандартная нотация, понятна всем
- **Flowcharts** — простые диаграммы потоков
- **DFD (Data Flow Diagrams)** — потоки данных между операциями
- **Таблицы в Confluence** — структурированное описание
