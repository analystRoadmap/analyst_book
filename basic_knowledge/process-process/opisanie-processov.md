# Описание процессов

| Грейд | Роль |
|-------|------|
| 🟢 Junior | 🅱️🅰️ BA+SA |

> **Описание процесса** — документирование последовательности действий, ролей, ресурсов и метрик, необходимых для выполнения бизнес-процесса и обучения сотрудников.

## Зачем описывать процессы?

Описание процесса — это инструкция для сотрудников:

- **Обучение** — новый сотрудник понимает, как работать
- **Стандартизация** — все делают процесс одинаково
- **Контроль качества** — понятны требования и исключения
- **Соответствие** — доказать аудиторам и регуляторам
- **Основа для автоматизации** — разработчики видят, что нужно программировать

## Структура описания процесса

### 1. Заголовок и основная информация

```
Название: Оформление заказа
Версия: 2.1
Дата последнего обновления: 2024-01-15
Владелец процесса: Менеджер продаж
Статус: Активен
```

### 2. Цель процесса

Одно четкое предложение:
- ✓ "Обработать заказ клиента от момента получения до отправки товара"
- ❌ "Работать с заказами"

### 3. Область действия (Scope)

Где начинается, где кончается:
- Начало: Клиент нажимает кнопку "Купить"
- Конец: Товар передан курьеру

### 4. Входные условия (Preconditions)

Что должно быть верно перед началом:
- Клиент зарегистрирован
- Товар в наличии
- Система платежей работает

### 5. Участники (Roles & Responsibilities)

| Роль | Ответственность | Действия |
|------|-----------------|----------|
| **Клиент** | Оплата | Заполняет данные, подтверждает платеж |
| **Система** | Валидация | Проверяет данные, рассчитывает доставку |
| **Менеджер** | Контроль | Отслеживает выполнение, разрешает исключения |
| **Склад** | Комплектация | Подбирает товар, упаковывает |

### 6. Пошаговое описание

| Шаг | Кто выполняет | Действие | Результат | Сроки |
|-----|---------------|---------|-----------|-------|
| 1 | Клиент | Выбирает товар и нажимает "Купить" | Товар в корзине | - |
| 2 | Клиент | Вводит адрес доставки | Адрес валидирован | 30 сек |
| 3 | Система | Рассчитывает стоимость доставки | Показывает цену | 5 сек |
| 4 | Клиент | Выбирает способ платежа | Переводится на платеж | - |
| 5 | Система | Обрабатывает платеж | Платеж успешен/отклонен | 10 сек |
| 6 | Менеджер | Проверяет заказ | Заказ одобрен | 1 час |
| 7 | Склад | Собирает товар | Товар готов | 2-4 часа |
| 8 | Курьер | Забирает и доставляет | Товар получен | 1-3 дня |

### 7. Условия и исключения

**Что если...**

| Сценарий | Действие |
|----------|----------|
| **Платеж отклонен** | Система запрашивает новый платеж (до 3 попыток) |
| **Товара нет в наличии** | Менеджер предлагает альтернативу или возврат денег |
| **Адрес невалидный** | Система просит уточнить адрес |
| **Клиент отменяет заказ** | Процесс возврата средств (если возможно) |

### 8. Ресурсы и системы

Что нужно для выполнения:
- IT системы: CRM, платежная система, WMS (складская система)
- Люди: менеджер, сотрудник склада, курьер
- Процессы: контроль качества, обработка возвратов

### 9. Метрики и KPI

Как измерить успех:

| Метрика | Целевое значение | Периодичность |
|---------|-----------------|---------------|
| **Время обработки** | < 2 часов | Ежедневно |
| **Процент успешных платежей** | > 95% | Еженедельно |
| **Процент ошибок** | < 1% | Ежемесячно |
| **Удовлетворенность клиента** | > 4.5/5 | Ежемесячно |

### 10. Приложения

- BPMN диаграмма процесса
- Чек-листы для сотрудников
- Примеры email-уведомлений
- Скриншоты интерфейса

## Форматы описания процесса

### 1. Табличный формат (простой)

Подходит для документации и обучения:
```
Шаг | Кто | Действие | Условие | Результат
```

### 2. Текстовый формат

Для подробных документов:
```
Шаг 1: Клиент выбирает товар
Действие: Клиент просматривает каталог и нажимает "Добавить в корзину"
Условие: Товар в наличии
Результат: Товар добавляется в корзину
```

### 3. BPMN диаграмма

Для визуального представления:
- Показывает потоки и ветвления
- Понятна разработчикам
- Лучше для сложных процессов

### 4. Блок-схема (Flowchart)

Простая визуализация:
```
[Начало] → [Условие?] → [Путь 1]
                      → [Путь 2]
          → [Конец]
```

## Лучшие практики при описании процессов

✓ Использовать активный залог ("Менеджер проверяет заказ")
✓ Нумеровать шаги четко (1, 2, 3...)
✓ Одна роль на один шаг
✓ Указывать сроки выполнения
✓ Описывать исключения отдельно
✓ Использовать примеры из реальной жизни
✓ Регулярно обновлять (не реже 1 раза в год)

## Типичные ошибки

❌ "Потом делаем" вместо "Менеджер проверяет"
❌ Слишком много шагов (> 20 для одного процесса)
❌ Описание "как" вместо "что" и "кто"
❌ Забывают про исключения и ошибки
❌ Непонятные сокращения
