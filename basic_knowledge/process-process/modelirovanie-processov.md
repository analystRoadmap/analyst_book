# Моделирование процессов

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️🅰️ BA+SA |

> **Моделирование процессов** — создание графического представления бизнес-процесса, показывающего последовательность действий, участников, данных и условия выполнения.

## Зачем моделировать процессы?

- **Понимание** — визуализация делает процесс очевидным для всех
- **Анализ** — легко выявить узкие места и неэффективность
- **Документирование** — создание reference для сотрудников
- **Автоматизация** — основа для внедрения IT-систем
- **Оптимизация** — данные для улучшения производительности

## BPMN (Business Process Model and Notation)

**BPMN** — стандартная нотация для моделирования бизнес-процессов. Она позволяет описать процесс на языке, понятном как бизнесу, так и разработчикам.

### Основные элементы BPMN

**События (Events)** — что-то происходит

```
Начало процесса:        ○
Конец процесса:         ●
Промежуточное событие:  ◇
```

**Задачи (Tasks)** — действия

```
┌─────────────────┐
│   Заполнить     │
│   анкету        │
└─────────────────┘
```

**Ворота (Gateways)** — условия и разветвления

```
Исключающие ворота (XOR):  ◇  — выбрать один из путей
Параллельные ворота (AND): ◇  — все пути одновременно
```

**Дорожки (Swimlanes)** — разделение по ролям

```
┌─────────────────────────────┐
│ Клиент                      │
├─────────────────────────────┤
│ ○ → Заполнить → Отправить   │
│      анкету    анкету       │
│                 │           │
├─────────────────┼───────────┤
│ Менеджер        │           │
├─────────────────┼───────────┤
│                 ○ → Проверить
│                     → Утвердить → ●
```

### Пример BPMN диаграммы: Процесс оформления заказа

```
Клиент: ○ → Выбрать товар → Добавить в корзину → Перейти к оплате
           │
           └─ Отмена? → ●

Система: → Расчитать доставку → Проверить статус платежа
            │
            ├─ Успех → Отправить уведомление
            └─ Ошибка → Запросить новый платеж

Администратор: → Подготовить отправку → Передать курьеру → ●
```

## Типы диаграмм

### 1. Low-level процесс
Детальное описание каждого шага с условиями и исключениями

### 2. High-level процесс
Общий обзор, не углубляясь в детали (для executives)

### 3. Functional diagram
Показывает, какая функция отвечает за каждый шаг

## Процесс создания BPMN диаграммы

1. **Определить границы** — где начало, где конец процесса
2. **Выявить участников** — создать swimlanes
3. **Описать шаги** — нарисовать задачи в порядке выполнения
4. **Добавить условия** — где ветвятся пути
5. **Указать события** — что запускает процесс
6. **Проверить логику** — все ли пути заканчиваются, нет ли мертвых циклов
7. **Валидировать** — показать бизнесу и разработчикам

## Типичные ошибки в BPMN

❌ **Слишком детальная** — 30+ шагов, сложно восприниматься
❌ **Слишком общая** — невозможно разработать по этому
❌ **Неправильная нотация** — перепутаны ворота и события
❌ **Отсутствуют исключения** — только happy path
❌ **Непонятные названия** — вместо "Проверить наличие" — "Проверка"

## Лучшие практики

✓ Один процесс = одна диаграмма (не пытаться вместить всё)
✓ Горизонтальный поток слева направо
✓ Ясные, глаголные названия задач ("Отправить счет" вместо "Счет")
✓ Одна swimlane = одна роль
✓ Исключения отдельными путями
