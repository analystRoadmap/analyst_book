# Нотации (Notations)

| Грейд | Роль |
|---|---|
| 🟢 Junior | BA + SA |

> **Нотация — это язык для общения** между бизнесом, аналитиками и разработчиками.

## Почему нотации важны?

Требования существуют в головах людей. Нотации помогают:
- Визуализировать сложные процессы
- Избежать недопонимания через текст
- Скоро понять архитектуру системы
- Общаться с людьми на одном языке

```
Без нотации:
"Клиент заказывает товар, система проверяет наличие,
если есть — отправляет, если нет — пишет письмо"
→ Расплывчато, можно интерпретировать по-разному

С нотацией (BPMN):
┌─────────────────────────────────────────────┐
│ Client submits order → System checks stock  │
│                        ↙        ↘           │
│                   Available    Not avail.   │
│                      ↓             ↓         │
│                  Send item    Send email    │
└─────────────────────────────────────────────┘
→ Четко и однозначно
```

## Семь основных нотаций аналитика

### 1️⃣ UML (Unified Modeling Language)

**Для чего:** Визуализировать структуру и взаимодействие компонентов системы.

**Основные диаграммы:**
- Use Case Diagram — актеры и их взаимодействие
- Class Diagram — классы и их связи
- Sequence Diagram — порядок сообщений между объектами
- State Diagram — состояния и переходы

**Кого интересует:** Разработчики, архитекторы, системные аналитики

### 2️⃣ BPMN (Business Process Model & Notation)

**Для чего:** Описывать бизнес-процессы с точки зрения бизнеса.

**Основные элементы:**
- События (Event) — что запускает процесс
- Задачи (Task) — что делает человек или система
- Шлюзы (Gateway) — условия и ветвления
- Потоки (Flow) — направление процесса

**Кого интересует:** Бизнес, процессные аналитики, разработчики

### 3️⃣ ERD (Entity-Relationship Diagram)

**Для чего:** Описывать структуру данных и связи между сущностями.

**Основные элементы:**
- Сущности (таблицы в БД)
- Атрибуты (столбцы в таблице)
- Связи (1:1, 1:N, M:N)

**Кого интересует:** Системные аналитики, DBA, разработчики

### 4️⃣ Flowchart (Блок-схема)

**Для чего:** Описывать алгоритмы и логику работы.

**Основные элементы:**
- Прямоугольник — процесс
- Ромб — решение (условие)
- Параллелограмм — ввод/вывод
- Овал — начало/конец

**Кого интересует:** Разработчики, системные аналитики

### 5️⃣ EPC (Event-driven Process Chain)

**Для чего:** Моделировать процессы ориентированные на события (legacy).

**Основные элементы:**
- События (Event) — то что произойдет
- Функции (Function) — действия
- Логические операторы — AND, OR, XOR

**Кого интересует:** SAP консультанты, старые системы

### 6️⃣ DFD (Data Flow Diagram)

**Для чего:** Показать как данные движутся через систему (legacy).

**Основные элементы:**
- Процессы
- Потоки данных
- Хранилища данных
- Внешние сущности

**Кого интересует:** Системные аналитики, старые системы

### 7️⃣ C4 Model (Context, Container, Component, Code)

**Для чего:** Описать архитектуру системы на разных уровнях абстракции.

**Четыре уровня:**
1. Context — что система, как она взаимодействует с миром
2. Container — основные компоненты (БД, API, UI)
3. Component — структура внутри контейнера
4. Code — классы и функции

**Кого интересует:** Архитекторы, старший разработчики

## Матрица нотаций

| Нотация | Для кого | Сложность | Инструмент | Когда использовать |
|---|---|---|---|---|
| **UML** | Разработчики | Высокая | Enterprise Architect | Архитектура системы |
| **BPMN** | Бизнес | Средняя | draw.io, BPMN.io | Бизнес-процессы |
| **ERD** | Аналитики | Средняя | draw.io, Lucidchart | Модель данных |
| **Flowchart** | Все | Низкая | draw.io | Алгоритмы |
| **EPC** | SAP | Средняя | ARIS | SAP процессы (legacy) |
| **DFD** | Аналитики | Средняя | draw.io | Потоки данных (legacy) |
| **C4** | Архитекторы | Высокая | Structurizr | Архитектура уровни |

## Типичная схема использования в проекте

```
ФАЗА 1 - Выявление требований:
→ BPMN: Описать текущий процесс (AS-IS)
→ Flowchart: Правила и условия

ФАЗА 2 - Дизайн системы:
→ ERD: Модель данных
→ UML Use Case: Взаимодействие с системой
→ C4 Context: Где система в экосистеме

ФАЗА 3 - Разработка:
→ UML Sequence: Как компоненты взаимодействуют
→ UML Class: Структура классов
→ Flowchart: Сложная логика

ФАЗА 4 - Тестирование:
→ BPMN: Описать будущий процесс (TO-BE)
→ Flowchart: Проверка граничных случаев
```

## Лучше один раз увидеть

```
Текст (сложно понять):
"Система получает заказ, проверяет наличие товара,
если товар есть и клиент авторизован, создает заказ,
отправляет письмо и уведомляет склад, иначе
отправляет письмо с объяснением и возвращает деньги"

BPMN диаграмма (ясно):
[Order Received] → [Check Stock?] → Yes → [Create Order] → [Send Email] → [Notify Warehouse]
                       ↓ No
                  [Send Explanation] → [Refund Money]
```
