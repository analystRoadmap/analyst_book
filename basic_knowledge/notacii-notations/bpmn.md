> **BPMN — это язык для описания бизнес-процессов**, понятный и бизнесу, и разработчикам.

## Введение в BPMN

**BPMN (Business Process Model and Notation)** — это стандартный язык для моделирования бизнес-процессов, разработанный Object Management Group (OMG).

```
BPMN показывает:
✓ Кто участвует в процессе (swim lanes)
✓ Какие шаги выполняются (tasks, events)
✓ Как принимаются решения (gateways)
✓ Как данные движутся (data flow)
✓ Где возможны ошибки (exception handling)

BPMN НЕ показывает:
✗ Структуру данных (для этого ERD)
✗ Классы и объекты (для этого UML)
```

## Основные элементы BPMN

### 1. События (Events)

```
○ → Начальное событие
  Что запускает процесс?
  Пример: Customer submits order

○ ◇ → Событие, требующее решения
  Что происходит, и нужно ли действовать?

● → Финальное событие
  Конец процесса
  Пример: Order completed successfully
```

### 2. Задачи (Tasks)

```
┌────┐
│Task│ → Действие или операция
└────┘

Типы задач:
- Manual Task: человек делает вручную
- Service Task: система делает автоматически
- User Task: диалог с пользователем
- Send Task: отправка сообщения
```

### 3. Шлюзы (Gateways)

```
◇ → Exclusive Gateway (XOR)
  Один из вариантов (как if-else)
  Пример: Товар в наличии? Да или Нет

◇+ → Inclusive Gateway (OR)
  Один или несколько вариантов

+ → Parallel Gateway (AND)
  Все варианты одновременно
```

### 4. Потоки (Flows)

```
→ Sequence Flow: нормальный переход
  [условие] → переход только если условие истинно

--→ Message Flow: отправка сообщения между участниками
  Пример: System sends Order to Warehouse

≈ Association: связь с данными или артефактами
```

### 5. Swim Lanes (Дорожки)

```
┌──────────────────────────────────────────────┐
│ Customer                                      │
├──────────────────────────────────────────────┤
│   ○ Submit order                              │
│     └→ [Manual check]                         │
└──────────────────────────────────────────────┘

┌──────────────────────────────────────────────┐
│ System                                        │
├──────────────────────────────────────────────┤
│                ○ Send confirmation            │
│                └→ [Email sent]                │
└──────────────────────────────────────────────┘

Каждая дорожка = один участник процесса
```

## Практический пример: Процесс заказа

### Простой вариант (AS-IS)

```
┌─────────────────────────────────────────────────────────────┐
│ Customer                                                    │
├─────────────────────────────────────────────────────────────┤
│  ○ → [Browse] → [Choose] → ◇ Stock? → [Wait] → ● Done
│                                ↓ No
│                            [Cancel]
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ System                                                      │
├─────────────────────────────────────────────────────────────┤
│                         ← [Update stock]
│                         ← [Send notification]
└─────────────────────────────────────────────────────────────┘
```

### Детальный вариант (TO-BE)

```
┌─────────────────────────────────────────────────────────┐
│ Customer                                                │
├─────────────────────────────────────────────────────────┤
│  ○ → [Browse Products] → [Add to Cart] → [Checkout]
│         │                                    │
│         └────────────────────────────────────→ ◇ Enough?
│                                                │   │
│                                          No ←──┘   │ Yes
│                                          │        │
│                                      [Cancel]     ↓
│                                                 [Confirm]
│                                                   ↓
│                                    ◇ Payment Method?
│                                     │ │ │
│                    ┌────────────────┼─┼─┼────┐
│                    │               Card/PayPal/Bank
│                    ↓                │   │     ↓
│                [Wait for Email]    ↓   ↓   [Bank Redirect]
│                    │               ...      ↓
│                    └───────────────────────[Success] → ●
```

## Обработка исключений (Exception Handling)

```
┌──────────────────────────────────────────┐
│ Main Process                             │
├──────────────────────────────────────────┤
│  ○ → [Process Order] ↘                  │
│                       ◇ Success?        │
│                       │  │              │
│                   No ←┘  │ Yes          │
│                   │      ↓              │
│              [Error] → [Log Error]     │
│                   │           │        │
│                   └─→ ◇ Retry? │       │
│                       │  │     ↓       │
│              Yes ←─────┘  │ [Notify]  │
│              │            │    │      │
│              └→ [Retry] ──┤    │      │
│                           ↓    ↓      │
│                        [End] or ●    │
└──────────────────────────────────────────┘
```

## Данные в BPMN

### Data Objects (объекты данных)

```
◐◐◐ ← Документ/информация
│ ├─ Order Form
│ ├─ Invoice
│ └─ Payment Receipt

Показывает, какие данные используются на каждом этапе
```

### Полный пример с данными

```
[Order Form] ───→ ◐◐ ← используется на этом шаге
   │              │
   └────→ [Create Order] ← Process
                   │
                   ↓
              [Invoice]
                   │
                   ↓
            [Send Invoice]
```

## Типология процессов

### AS-IS (текущее состояние)

```
Текущий процесс:
○ → [Manual check] → [Send email] → [Wait response] → ●
    (3 дня)         (1 день)       (2 дня)

Проблема: Долгий процесс (6 дней)
```

### TO-BE (желаемое состояние)

```
Новый процесс:
○ → [Automated check] → [Auto email] → ●
     (10 сек)         (1 мин)

Улучшение: 6 дней → 1 минута!
```

## Best Practices

```
1. ЯСНОСТЬ:
   ✓ Один процесс = одна диаграмма
   ✓ Max 30 элементов на диаграмме (иначе сложно понять)
   ✗ Не смешивайте несвязанные процессы

2. ПОКРЫТИЕ:
   ✓ Включите исключения (что если ошибка?)
   ✓ Покажите альтернативные пути
   ✗ Не игнорируйте edge cases

3. УЧАСТНИКИ:
   ✓ Четко определите swim lanes
   ✓ Один участник = одна дорожка
   ✗ Не путайте роли и системы

4. ВАЛИДАЦИЯ:
   ✓ Покажите диаграмму реальным пользователям
   ✓ "Это правильно?"
   ✗ Не делайте в изоляции

5. ВЕРСИОНИРОВАНИЕ:
   ✓ BPMN v1.0 = текущий процесс
   ✓ BPMN v2.0 = новый процесс (улучшенный)
   ✗ Не перезаписывайте старую версию
```

## Сравнение BPMN с другими нотациями

| Нотация | Когда | Для кого |
|---|---|---|
| **BPMN** | Бизнес-процессы | Бизнес-аналитики, процессные инженеры |
| **Flowchart** | Алгоритмы, логика | Разработчики |
| **UML Activity** | Технические взаимодействия | Архитекторы, разработчики |
| **EPC** | SAP процессы (legacy) | SAP консультанты |
| **DFD** | Потоки данных (legacy) | Системные аналитики |
