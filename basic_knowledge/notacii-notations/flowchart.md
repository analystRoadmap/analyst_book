> **Flowchart — это самая простая нотация** для описания алгоритмов и логики.

## Определение Flowchart

**Flowchart (блок-схема)** — это диаграмма, показывающая последовательность шагов в процессе или алгоритме с помощью стандартных символов.

```
Когда использовать:
✓ Описание алгоритмов (как что-то работает)
✓ Логика принятия решений (if-else-if)
✓ Простые процессы без много участников
✓ Обучение новичков

Когда НЕ использовать:
✗ Сложные бизнес-процессы (для этого BPMN)
✗ Много участников/дорожек (для этого BPMN или UML)
```

## Основные символы

### 1. Овал (Начало/Конец)

```
    ┌───────┐
    │ START │
    └───────┘
        ↓
   [процесс]
        ↓
    ┌───────┐
    │  END  │
    └───────┘
```

### 2. Прямоугольник (Процесс/Действие)

```
    ┌─────────────────┐
    │ Process Name    │ ← Действие: "Validate input"
    │ или Operation   │
    └─────────────────┘
        ↓
   [следующий шаг]
```

### 3. Ромб (Условие/Решение)

```
         ┌─────────┐
         │ Условие │
         │ верно?  │
         └────┬────┘
          Yes │ No
        ┌─────┴─────┐
        ↓           ↓
    [Action A]  [Action B]
```

### 4. Параллелограмм (Ввод/Вывод)

```
    ┌───────────────────┐
    │  Input / Output   │ ← "Enter username"
    │  (I/O operation)  │    "Print result"
    └───────────────────┘
```

### 5. Скругленный прямоугольник (Подпроцесс)

```
    ┌──────────────┐
    │ Sub-process  │ ← Ссылка на другую блок-схему
    │ (вызов функ) │
    └──────────────┘
```

### 6. Стрелки (Поток)

```
──→  Нормальный переход
 ↓
┌─┴─────────────┐
│   [Условие]   │
└───┬───────┬───┘
    ↓ Yes   ↓ No
  [Path1]  [Path2]
```

## Символы для сложной логики

### Цикл (Loop)

```
    ┌──────────┐
    │ START    │
    └────┬─────┘
         ↓
    ┌──────────┐
    │ i = 0    │
    └────┬─────┘
         ↓
    ┌─────────────┐
    │ i < 10?     │ ← Условие цикла
    └──┬──────┬───┘
       │Yes   │No
    ┌──┴─┐    └────────┐
    ↓    │             ↓
┌──────────┐    ┌─────────────┐
│ Process  │    │ Exit Loop   │
│ Item i   │    └──────┬──────┘
└────┬─────┘           │
     ↓                 │
┌──────────┐           │
│ i = i+1  │           │
└────┬─────┘           │
     │                 │
     └────────┬────────┘
              ↓
         ┌──────────┐
         │   END    │
         └──────────┘
```

### Параллельные пути

```
    ┌──────────┐
    │ START    │
    └────┬─────┘
         ↓
    ┌──────────┐
    │ Split    │
    │ (fork)   │
    └─┬─────┬──┘
      │     │
   ┌──┴──┐ ┌┴─────┐
   ↓     ↓ ↓      ↓
[Process] [Process]
[  1   ]  [  2   ]
   │      │
   └──┬───┘
      ↓
  ┌───────┐
  │ Join  │
  └───┬───┘
      ↓
 ┌──────────┐
 │   END    │
 └──────────┘

(параллельные операции)
```

## Best Practices

```
1. ПРОСТОТА:
   ✓ Max 30-40 элементов на диаграмме
   ✓ Не более 3-4 уровней вложенности
   ✗ Не пытайтесь показать все сразу

2. ЯСНОСТЬ:
   ✓ Используйте понятные имена ("Validate email" не "Val_em")
   ✓ Одна диаграмма = один процесс
   ✗ Не смешивайте много процессов

3. СТАНДАРТЫ:
   ✓ Используйте стандартные символы
   ✓ Сверху вниз или слева направо
   ✗ Не придумывайте новые символы

4. ГРАНИЧНЫЕ СЛУЧАИ:
   ✓ Покажите error paths
   ✓ Что если ошибка? Что если timeout?
   ✗ Не показывайте только "happy path"

5. ВАЛИДАЦИЯ:
   ✓ Все пути должны привести к END
   ✓ Не должно быть "зависших" путей
   ✗ Проверьте логику
```

## Сравнение нотаций

| Нотация | Сложность | Аудитория | Когда |
|---|---|---|---|
| **Flowchart** | Низкая | Все | Алгоритмы, простая логика |
| **BPMN** | Средняя | Бизнес + разработчики | Бизнес-процессы |
| **UML Activity** | Высокая | Разработчики | Технические взаимодействия |
| **Pseudocode** | Средняя | Разработчики | До кодирования |
