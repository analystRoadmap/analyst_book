> **UML — это язык для архитекторов и разработчиков**. Показывает структуру системы.

## Введение в UML

**UML (Unified Modeling Language)** — это стандартный язык визуализации архитектуры систем. Разработан Object Management Group (OMG).

```
UML используется для:
✓ Визуализации архитектуры
✓ Проектирования классов
✓ Описания взаимодействия компонентов
✓ Документирования системы

UML НЕ используется для:
✗ Описания бизнес-процессов (для этого BPMN)
✗ Описания потоков данных (для этого DFD)
```

## 1. Use Case Diagram

**Определение:** Показывает, кто (актеры) выполняет какие функции (use cases) в системе.

### Символы

```
┌─────────────────────────────────────────┐
│   [Actor / Пользователь]                │ ← Актер (палка)
│          (стик-фигура)                  │
└─────────────────────────────────────────┘

┌────────────────────────────┐
│  Use Case / Функция        │ ← Use Case (эллипс)
│  Например: "Login User"    │
└────────────────────────────┘

System Boundary: прямоугольник, внутри use cases
```

### Пример: Электронный магазин

```
                      ┌─────────────────────────────┐
                      │    E-commerce System        │
        ┌─────────┐   ├─────────────────────────────┤
        │ Customer│───┤ Browse Products             │
        └─────────┘   │ Add to Cart                 │
                      │ Checkout                    │
                      │ Pay (uses Payment Gateway)  │
            ┌──────┐  ├─────────────────────────────┤
            │Admin │──┤ Manage Products             │
            └──────┘  │ View Orders                 │
                      │ Send Invoice                │
                      └─────────────────────────────┘

         ┌──────────────────┐
         │ Payment Gateway  │ ← Внешняя система
         └──────────────────┘
```

### Виды связей

```
Association (→):
Customer → Login: Customer выполняет Login

Include (→→→):
Checkout → Pay: Checkout всегда включает Pay
(Pay — обязательная часть Checkout)

Extend (←---):
Pay ← Pay with Loyalty: Pay with Loyalty расширяет Pay
(опциональный вариант Pay)
```

## 2. Class Diagram

**Определение:** Показывает классы, их атрибуты, методы и связи.

### Синтаксис класса

```
┌──────────────────────┐
│ ClassName           │ ← Имя класса
├──────────────────────┤
│ -private_attr       │ ← Атрибуты (private -)
│ #protected_attr     │    protected (#)
│ +public_attr        │    public (+)
├──────────────────────┤
│ +public_method()    │ ← Методы
│ -private_method()   │
└──────────────────────┘
```

### Пример: Заказ в e-commerce

```
┌──────────────────────────┐      ┌──────────────────┐
│ Order                    │      │ Customer         │
├──────────────────────────┤      ├──────────────────┤
│ -id: int                 │      │ -id: int         │
│ -total_price: float      │      │ -name: string    │
│ -status: OrderStatus     │  owns    -email: string │
│ -created_date: datetime  │ ├─────┤ -phone: string  │
│ -items: OrderItem[]      │      │ +getProfile()   │
├──────────────────────────┤      └──────────────────┘
│ +create()                │
│ +cancel()                │      ┌──────────────────┐
│ +getTotal()              │      │ OrderItem        │
│ +ship()                  │  has │ -product_id      │
└──────────────────────────┘──────┤ -quantity        │
         ↑                        │ -price           │
         │ inherits               └──────────────────┘
         │
┌──────────────────────────┐
│ ExpressOrder            │
├──────────────────────────┤
│ -express_fee: float     │
│ -delivery_hours: int    │
└──────────────────────────┘
```

### Типы связей

```
Наследование (↑):
ExpressOrder ↑ Order: ExpressOrder наследует Order

Композиция (◇):
Order ◇ OrderItem: OrderItem существует только в контексте Order
(если Order удалена, OrderItem тоже)

Агрегация (○):
Company ○ Employee: Employee может существовать без Company

Ассоциация (→):
Order → Customer: обычная связь "один-ко-многим"
```

## 3. Sequence Diagram

**Определение:** Показывает порядок сообщений между объектами во времени.

### Синтаксис

```
Участник1 → Участник2 : сообщение
Участник2 → Участник1 : ответ

Время идет вниз
↓
```

### Пример: Процесс оформления заказа

```
Customer   OrderUI   OrderService   DB   PaymentGateway
   │          │            │         │          │
   │──────────→│  click()   │         │          │
   │           │────────────→ find() │          │
   │           │            │────────→          │
   │           │            │←────────          │
   │           │            │ save() │          │
   │           │            │────────→          │
   │           │            │        │ process  │
   │           │            │────────┼─────────→│
   │           │            │        │          │ (обработка)
   │           │            │        │←─────────│
   │           │ success()  │        │          │
   │←──────────│            │        │          │
   │ "Order #1"             │        │          │
```

### Фрагменты (Fragments)

```
opt [условие]: опциональный блок
  → выполняется если условие истинно

alt [условие1] / [условие2]: альтернативы
  → выполняется один из блоков

loop [count]: повтор
  → выполняется несколько раз

par: параллельное выполнение
  → несколько участников действуют одновременно
```

## 4. State Diagram (Statechart)

**Определение:** Показывает состояния объекта и переходы между ними.

### Символы

```
○ → (○) : начальное состояние
(○) : состояние
[условие] : guard condition (когда происходит переход)
action / : что делать при переходе
(⊗) : финальное состояние
```

### Пример: Жизненный цикл заказа

```
        ┌──────────────────────────────────────┐
        │                                      │
        ↓                                      │
    ○ → [New] ──submit──→ [Pending] ──pay──→ [Paid]
                    │           │
        cancel /     │ cancel /  │ deliver
        refund       ↓           ↓
                 [Cancelled]  [Shipped]
                     ↓           │
                     └───→ [Completed]

[New]:      Заказ создан, деньги не взяты
[Pending]:  Ожидание оплаты
[Paid]:     Деньги получены, готовим отправку
[Shipped]:  Товар отправлен
[Completed]: Заказ завершен
[Cancelled]: Заказ отменен
```

## 5. Component Diagram

**Определение:** Показывает компоненты системы, их интерфейсы и зависимости.

### Пример: Архитектура веб-приложения

```
┌────────────────────────────────────────────┐
│           Web Application                  │
├────────────────────────────────────────────┤
│                                            │
│ ┌─────────┐      ┌──────────┐             │
│ │ UI Comp │ ────→│ API Comp │             │
│ └─────────┘      └──────────┘             │
│                        │                  │
│                        ↓                  │
│                 ┌──────────────┐          │
│                 │ Service Comp │          │
│                 └──────────────┘          │
│                        │                  │
│                        ↓                  │
└────────────────────────────────────────────┘
         ↓
┌────────────────────────────────────────────┐
│           Database Component               │
├────────────────────────────────────────────┤
│ - User Table                               │
│ - Order Table                              │
│ - Product Table                            │
└────────────────────────────────────────────┘
```

## 6. Deployment Diagram

**Определение:** Показывает физическую архитектуру системы (серверы, сети, устройства).

### Пример: Развертывание приложения

```
┌──────────────────────────────────────────┐
│         Web Browser                      │
│         (User Device)                    │
└──────────────────┬───────────────────────┘
                   │ HTTP
                   ↓
┌──────────────────────────────────────────┐
│         Web Server                       │
│         (Nginx / Apache)                 │
│         - Static files                   │
│         - Reverse proxy                  │
└──────────────────┬───────────────────────┘
                   │
     ┌─────────────┴────────────┐
     ↓                          ↓
┌──────────┐            ┌──────────┐
│ App      │            │ App      │
│ Server 1 │            │ Server 2 │
│(Node.js) │            │(Node.js) │
└──────┬───┘            └───┬──────┘
       └────────┬───────────┘
                ↓
         ┌────────────────────┐
         │  Database Server   │
         │  (PostgreSQL)      │
         │  - Master          │
         │  - Replication     │
         └────────────────────┘
```

## Когда использовать какую диаграмму?

| Диаграмма | Когда | Для кого |
|---|---|---|
| **Use Case** | Вначале: что система делает? | Все (бизнес + разработчики) |
| **Class** | Проектирование: классы и структура | Разработчики, архитекторы |
| **Sequence** | Деталь: как объекты взаимодействуют | Разработчики |
| **State** | Логика: состояния и переходы | Разработчики, тестеры |
| **Component** | Архитектура: основные части системы | Архитекторы, tech leads |
| **Deployment** | Infrastructure: где работает система | DevOps, архитекторы |

```

```
