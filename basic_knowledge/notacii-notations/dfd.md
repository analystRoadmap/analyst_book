# DFD (Data Flow Diagram)

| Грейд | Роль |
|---|---|
| 🔵 Middle | SA |

> **DFD — это legacy нотация** для моделирования потоков данных. В новых проектах используйте C4 Model вместо этого.

## Историческая справка

**DFD (Data Flow Diagram)** — это нотация, разработанная в 1970-1980х годах для систем анализа (structured analysis). Была стандартом для документирования систем до 2010х годов.

```
Статус DFD:
⚠️ LEGACY - в новых проектах используйте C4 Model
✓ Все еще актуальна для:
  - Анализа legacy систем
  - Документирования потоков данных
  - Обучения основам системного анализа
```

## Основные элементы DFD

### 1. Процесс (Process)

```
Нотация Yourdon/DeMarco:
    ⭕ ← Круг/овал
    │ Process Name
    │ (может быть номер)

Пример:
    ⭕
    1.0
  Validate
   Order
```

Альтернативная нотация Gane/Sarson:
```
┌───────────────────┐
│   Process Name    │
│        1.0        │ ← Прямоугольник с волн. краями
└───────────────────┘
```

### 2. Хранилище данных (Data Store)

```
Нотация Yourdon/DeMarco:
  ┌─────────────┐
  │  Database   │
  │    Name     │
  └─────────────┘

Нотация Gane/Sarson:
  ═════════════════
  │  Database   │
  ═════════════════
  (параллельные линии - как полка)

Пример:
  ┌──────────┐
  │ Customer │
  │ Database │
  └──────────┘
```

### 3. Внешняя сущность (External Entity)

```
Нотация Yourdon/DeMarco:
  ◻ ← Квадрат
  External Actor
  (или другая система)

Нотация Gane/Sarson:
  ┌─────────────┐
  │   Actor     │ ← Просто квадрат
  └─────────────┘

Примеры:
  ◻
  Customer

  ◻
  Payment Gateway
```

### 4. Поток данных (Data Flow)

```
→ стрелка
  "Data Name"

Примеры:
Process A ──"Order"──→ Process B
             ↓
         (данные "Order"
         передаются от A к B)

External Entity ──"Customer Info"──→ Database
```

## Уровни детализации в DFD

### Context Diagram (Уровень 0)

Самый высокий уровень - система как черный ящик:

```
┌──────────────┐                 ┌──────────────┐
│  Customer    │                 │  Warehouse   │
└──────┬───────┘                 └──────┬───────┘
       │                                │
       │ Order Info                     │ Shipping Info
       ↓                                ↑
      ┌────────────────────────────────┐
      │   E-commerce System            │
      │                                │
      │                                │
      └────────────────────────────────┘
       ↑                                ↓
       │ Confirmation                   │ Stock Update
       │                                │
  ┌────┴───────┐                  ┌─────┴────────┐
  │  Email     │                  │ Supplier     │
  │ Service    │                  │ System       │
  └────────────┘                  └──────────────┘
```

### DFD Level 1 (детализация первого уровня)

Разбиваем систему на подсистемы:

```
┌────────────────────────────────────────┐
│  E-commerce System                     │
├────────────────────────────────────────┤
│                                        │
│  ┌──────────────┐   ┌──────────────┐ │
│  │ 1.0          │   │ 2.0          │ │
│  │ Manage       │   │ Process      │ │
│  │ Products     │   │ Orders       │ │
│  └──────┬───────┘   └──────┬───────┘ │
│         │                  │         │
│         └────┬─────────────┘         │
│              │                       │
│         ┌────▼─────┐                 │
│         │           │                │
│      Database      │                 │
│      Products      │                 │
│         │           │                │
└─────────┼───────────┼─────────────────┘
          │           │
          └───────────┘
```

### DFD Level 2, 3, ... (дальнейшая детализация)

Каждый процесс можно разбить еще подробнее.

## Обозначения потоков данных

### Простой поток

```
Source ──"Data Name"──→ Destination

Пример:
┌──────────┐           ┌──────────┐
│ Customer │──"Order"─→│ Process  │
└──────────┘           └──────────┘
```

### Сложный поток (множество элементов)

```
Source ──┬─"Data1"──→ Dest1
         │
         ├─"Data2"──→ Dest2
         │
         └─"Data3"──→ Dest3

Пример:
┌──────────┐       ┌──────────┐
│ Customer │──────→│ Process  │
└──────────┘       └──────────┘
                        ↓
                   ┌─────┬─────┐
                   │     │     │
                ┌──┴──┐ ┌┴──┐ ┌┴──┐
                │Email│ │SMS│ │DB │
                └─────┘ └───┘ └───┘
```

## DFD vs другие нотации

| Нотация | Фокус | Сложность | Статус |
|---|---|---|---|
| **DFD** | Потоки данных | Средняя | Legacy |
| **C4 Model** | Архитектура, контекст | Высокая | Современный |
| **BPMN** | Бизнес-процессы | Средняя | Современный |
| **UML** | Классы и объекты | Высокая | Современный |

## DFD vs C4 Model

```
DFD (старый подход):
╔════════════════════════════════════════╗
║        E-commerce System               ║
║  ⭕ Process 1    ⭕ Process 2          ║
║     ↕              ↕                   ║
║  [Database]   [Database]              ║
╚════════════════════════════════════════╝

C4 Model (новый подход):
System Context:
[Customers] ──→ [E-commerce App] ←── [Warehouse]
                       ↓
                 [Email Service]

Container:
┌──────────────────────────────┐
│ Web App │ API │ Mobile App   │
│         └─────┘              │
│          Database            │
└──────────────────────────────┘

Component:
[Product Service] ────┐
[Order Service]   ────┼→ [Data Layer]
[Payment Service] ────┘
```

## Правила и Best Practices

```
1. ИЕРАРХИЯ:
   ✓ Context (Level 0) → System (Level 1) → Components (Level 2)
   ✓ Каждый процесс может быть детализирован
   ✗ Не больше 6-7 процессов на одной диаграмме

2. НУМЕРАЦИЯ:
   ✓ Level 0: 0
   ✓ Level 1: 1.0, 2.0, 3.0
   ✓ Level 2: 1.1, 1.2, 1.3 (внутри 1.0)
   ✗ Не пропускайте номера

3. ПОТОКИ:
   ✓ Нужно указать имя потока ("Order Info", не "Data")
   ✓ Один поток = одна стрелка
   ✓ Можно связать несколько стрелок для удобства
   ✗ Не пересекайте потоки без необходимости

4. ХРАНИЛИЩА:
   ✓ Бизнес БД должны быть на диаграмме
   ✓ Может быть несколько БД
   ✗ Не смешивайте БД с системой

5. ВАЛИДАЦИЯ:
   ✓ Каждый поток должен иметь источник и целевое назначение
   ✓ Процессы не должны быть изолированы
   ✗ Не оставляйте "плавающие" стрелки
```

## Когда еще используется DFD?

```
1. Legacy системы:
   - Старые проекты (до 2010) документированы в DFD
   - Нужно понимать для maintenance/модификации
   - Помочь мигрировать на современные нотации

2. Обучение:
   - DFD хороша для обучения основам
   - Помогает понять потоки данных
   - Простая нотация для начинающих

3. Документирование данных:
   - Если нужна подробная документация потоков данных
   - Не только процессов, но именно данных
   - Может быть дополнением к BPMN

4. Регуляторные требования:
   - Некоторые старые стандарты требуют DFD
   - В банках/страховках могут быть legacy процессы
```

## Миграция с DFD на C4 Model

```
DFD концепция → C4 эквивалент

DFD Context Diagram → C4 System Context Diagram
(внешние системы, пользователи)

DFD Level 1 → C4 Container Diagram
(основные компоненты: БД, API, UI)

DFD Level 2 → C4 Component Diagram
(внутренняя структура контейнера)

DFD Process → C4 Dynamic Diagram
(как компоненты взаимодействуют)
```
