> **Проектирование** — это создание архитектуры, компонентов и структуры системы, которая решает бизнес-проблемы, удовлетворяет требованиям пользователей и обеспечивает достижение целей организации.

## Что должен знать аналитик о проектировании?

Аналитик участвует в проектировании на всех этапах разработки. Это не означает, что он проектирует базы данных или архитектуру системы (это задача архитекторов и разработчиков), но он должен понимать, как его требования трансформируются в архитектурные решения.

### Основные аспекты проектирования для аналитика

**1. Функциональная архитектура**
- Взаимодействие между модулями системы
- Потоки данных и информации
- Ролевая модель и ответственности
- Интеграция с внешними системами

Пример: При описании процесса "Оформление заказа" аналитик должен понимать, как система будет:
- Валидировать данные в каждом шаге
- Взаимодействовать с системой инвентаризации
- Отправлять уведомления клиентам
- Логировать все действия для аудита

**2. Моделирование данных**
Аналитик определяет, какие данные нужны системе:

| Сущность | Атрибуты | Отношения |
|----------|----------|----------|
| Заказ | ID, дата, статус, сумма | Связь с Клиентом, Товарами |
| Товар | ID, название, цена, наличие | Связь с Категорией, Заказами |
| Клиент | ID, имя, контакты, адрес | Связь с Заказами, Платежами |

**3. Архитектурные шаблоны**
Понимание основных подходов помогает лучше описывать требования:
- **Monolithic** (монолитная) — одно приложение со всеми функциями
- **Microservices** (микросервисы) — независимые сервисы, взаимодействующие через API
- **Event-driven** (событийная) — система реагирует на события в реальном времени
- **Layered** (многоуровневая) — отделение бизнес-логики от представления

### Как аналитик влияет на проектирование?

1. **Точные требования** — четкое описание функциональности уменьшает неопределенность для архитекторов
2. **Диаграммы и модели** — использование BPMN, DFD, UML помогает визуализировать решение
3. **Нефункциональные требования** — указание производительности, масштабируемости, безопасности
4. **Валидация с командой** — обсуждение технических ограничений и возможностей

### Ошибки, которых следует избегать

- ❌ Требования без учета технических ограничений
- ❌ Забывать о интеграциях с существующими системами
- ❌ Описывать "как сделать" вместо "что нужно получить"
- ❌ Игнорировать non-functional requirements (безопасность, нагрузка, доступность)

## Ключевые инструменты для документирования проектирования

- **UML диаграммы** — диаграммы классов, последовательности, компонентов
- **BPMN** — для моделирования процессов
- **DFD (Data Flow Diagrams)** — потоки данных в системе
- **Архитектурные диаграммы** — взаимодействие компонентов
- **ER диаграммы** — моделирование данных

## Взаимодействие с другими ролями

| Роль | Взаимодействие |
|------|----------------|
| **Архитектор** | Согласование нефункциональных требований, ограничений системы |

| **Разработчик** | Уточнение технических деталей, ответы на вопросы по требованиям |
| **QA инженер** | Совместная разработка критериев приемки и сценариев тестирования |
