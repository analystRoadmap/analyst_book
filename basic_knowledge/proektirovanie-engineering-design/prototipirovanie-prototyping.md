# Прототипирование (Prototyping)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️🅰️ BA+SA |

> **Прототипирование** — создание упрощенной модели продукта или функции для валидации идей, получения обратной связи и снижения рисков до начала полномасштабной разработки.

## Почему прототипирование критично для аналитика?

Прототип — это инструмент коммуникации. Вместо 10 страниц текста можно показать интерактивную модель, которую все поймут одинаково:

- **Клиент видит** результат своих инвестиций
- **Разработчики уточняют** технические требования
- **QA подготавливает** тест-кейсы
- **Аналитик валидирует** правильность требований

## Виды прототипов

### 1. Low-Fidelity (Низкая детализация)

**Что это:**
- Бумажные эскизы, sketches
- Wireframes в Figma/Balsamiq
- Простые диаграммы потоков

**Примеры:**
```
┌─────────────────────────┐
│    МЕНЮ  |  ПОИСК       │
├─────────────────────────┤
│                         │
│  Товар 1    |  Товар 2  │
│  Товар 3    |  Товар 4  │
│                         │
│ [Фильтры]  [Сортировка]│
└─────────────────────────┘
```

**Преимущества:**
- Быстро создавать (часы вместо дней)
- Дешево (бумага, Figma бесплатен)
- Легко менять на основе feedback
- Не создают ложное впечатление готовности

**Когда использовать:**
- На ранних этапах (много неопределенностей)
- При выборе между несколькими концепциями
- На встречах с заказчиком для быстрого согласования

---

### 2. Mid-Fidelity (Средняя детализация)

**Что это:**
- Wireframes с реальными данными
- Интерактивные потоки в Figma/Adobe XD
- Базовая интерактивность (клики, переходы)

**Отличие от Low-Fi:**
- Более точные размеры и пропорции
- Реальный текст вместо "Lorem ipsum"
- Показаны основные интерактивные элементы

**Используется когда:**
- Требования стабилизировались
- Нужно показать user flows детальнее

---

### 3. High-Fidelity (Высокая детализация)

**Что это:**
- Близко к финальному дизайну
- Реальные шрифты, цвета, иконки
- Полная интерактивность (все клики работают)
- Микро-взаимодействия и анимации

**Примеры инструментов:**
- Figma + Prototyping
- Adobe XD
- Axure
- Marvel
- InVision

**Когда использовать:**
- Перед разработкой (финальная валидация)
- Для usability testing с реальными пользователями
- Когда дизайн финализирован

---

### 4. Кликабельный прототип (Clickable Prototype)

**Что это:**
- Статические изображения экранов
- Связаны горячими точками (hotspots)
- Пользователь может кликать и переходить

**Инструменты:**
- Figma (встроенный протotyping)
- Axure
- Penpot

**Пример структуры:**
```
Экран 1 (Вход)
├─ Кнопка "Вход" → Экран 2
└─ Ссылка "Забыли пароль?" → Экран 3

Экран 2 (Каталог)
├─ Фильтр → Экран 2.1
├─ Товар → Экран 4
└─ Корзина → Экран 5
```

## Процесс прототипирования

| Шаг | Деятельность | Ответственный |
|-----|--------------|---------------|
| 1 | Определить пользовательские сценарии | Аналитик |
| 2 | Создать low-fi wireframes | Дизайнер |
| 3 | Валидировать с заказчиком | Аналитик + Клиент |
| 4 | Создать mid-fi потоки | Дизайнер |
| 5 | Провести usability testing | Аналитик + Дизайнер |
| 6 | Собрать feedback | Аналитик |
| 7 | Создать high-fi макеты | Дизайнер |
| 8 | Финальная валидация | Все stakeholders |

## Типичные ошибки при прототипировании

❌ **Слишком детально на ранних этапах** — тратите время на детали, которые могут поменяться

❌ **Забывают про user research** — прототипируют, не поговорив с пользователями

❌ **Статичные экраны без потоков** — непонятно, как переходит между экранами

❌ **Игнорируют feedback** — показали прототип, но не внедрили замечания

❌ **Слишком красивый дизайн на ранних этапах** — людям кажется, что это готовый продукт

## Как аналитик готовит прототипирование?

1. **User scenarios** — описание пути пользователя через систему
2. **Wireframe brief** — описание что должно быть на каждом экране
3. **Content examples** — примеры реальных данных
4. **Interaction flows** — где нужны popup, modal, validation
5. **Edge cases** — как система ведет себя при ошибках, пустых данных
