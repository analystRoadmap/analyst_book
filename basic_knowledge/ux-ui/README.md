# UX/UI Design

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️🅰️ BA+SA |

> **UX/UI** — дисциплины, которые превращают требования и бизнес-процессы в интуитивно понятные, красивые и функциональные интерфейсы, обеспечивающие положительный пользовательский опыт.

## Зачем аналитику понимать UX/UI?

Аналитик определяет **ЧТО** должна делать система, дизайнер определяет **КАК** это должно выглядеть. Но эти две роли должны работать в тесном контакте:

- **Аналитик** → "Пользователь должен иметь возможность отфильтровать товары"
- **Дизайнер** → "Где разместить фильтры? Как они выглядят?"
- **Вместе** → "Фильтры слева, результаты справа, реагируют в реальном времени"

## Различие UX и UI

| UX (User Experience) | UI (User Interface) |
|---------------------|-------------------|
| **Ощущение** — как система работает | **Визуальное** — как система выглядит |
| Потоки пользователя, взаимодействия | Цвета, кнопки, шрифты, иконки |
| "Удобно ли пользователю?" | "Красиво ли выглядит?" |
| Wireframes, User Flows | Mockups, Design System |

**Хороший продукт = хороший UX + хороший UI**

## UI Design Components

### Layout (Расположение элементов)

Как располагаются элементы на экране:
- **Grid-based** — элементы выравнены по сетке (8px, 16px)
- **Responsive** — меняется в зависимости от размера экрана
- **Whitespace** — пространство вокруг элементов для дыхания

**Для аналитика:** При описании экрана указывайте, какие элементы видны и их приоритет.

---

### Colors (Цвета)

| Тип цвета | Назначение | Примеры |
|-----------|-----------|---------|
| **Primary** | Основной цвет бренда | Кнопка "Купить", основная CTA |
| **Secondary** | Вспомогательный цвет | Второстепенные кнопки |
| **Success** | Успешные действия | Зеленый для ✓ подтверждения |
| **Error** | Ошибки | Красный для ❌ ошибок |
| **Warning** | Предупреждение | Оранжевый для ⚠️ предупреждений |
| **Neutral** | Фон, текст | Серый для текста и фона |

**Для аналитика:** Обозначьте, где система должна показать успех, ошибку, предупреждение.

---

### Typography (Шрифты)

Система шрифтов для разных элементов:

| Элемент | Размер | Вес | Пример |
|---------|--------|-----|--------|
| **Заголовок (H1)** | 32px | Bold | # Главный заголовок |
| **Заголовок (H2)** | 24px | Bold | ## Подзаголовок |
| **Body Text** | 14-16px | Regular | Основной текст |
| **Подпись** | 12px | Regular | Вспомогательный текст |
| **Code** | 12px | Monospace | `var x = 5;` |

**Совет:** Используйте не более 2-3 шрифтов в дизайне.

---

### Icons (Иконки)

Визуальные символы для действий и статусов:
- **Functional** — вызывают действие (поиск 🔍, удалить ❌)
- **Status** — показывают состояние (загрузка ⏳, ошибка ❌)
- **Decorative** — улучшают внешний вид (лайк ❤️, звезда ⭐)

---

### Spacing (Интервалы)

Промежутки между элементами:

```
┌─────────────────────────┐
│ ↓ 16px (отступ сверху)  │
│                         │
│ ┌─────────────────────┐ │
│ │  Button (8px внутри)│ │
│ └─────────────────────┘ │
│ ↓ 24px (между элементами)
│ ┌─────────────────────┐ │
│ │  Button (8px внутри)│ │
│ └─────────────────────┘ │
│                         │
│ ↓ 16px (отступ снизу)   │
└─────────────────────────┘
```

**Правило:** Используйте шкалу 8px (8, 16, 24, 32, 48px).

---

### Design System (Система компонентов)

**Design System** — набор переиспользуемых компонентов с единой логикой:

Примеры компонентов:
- **Button** (Primary, Secondary, Disabled)
- **Input Field** (Text, Password, Error state)
- **Card** (Basic, Highlighted, Clickable)
- **Modal** (Confirm, Form, Alert)
- **Navigation** (Tabs, Sidebar, Breadcrumbs)

**Преимущества:**
✓ Консистентность во всем продукте
✓ Разработчикам быстрее (компоненты готовы)
✓ Дизайнерам быстрее (не рисуют с нуля)
✓ Новых фич быстрее (используют готовые кирпичики)

## UX Design (User Experience)

### User Research (Исследование пользователей)

**Вопросы для исследования:**
- Кто наши пользователи?
- Какие у них боли?
- Как они сейчас решают проблему?
- Что им нравится / не нравится?

**Методы:**
- Интервью (спрашиваем пользователей)
- Наблюдение (смотрим как они работают)
- Опросы (анкеты)
- Тестирование юзабилити (даем пользователю задачу)

---

### Personas (Образы пользователей)

Вымышленные, но основанные на исследованиях образы:

```
Имя: Олег, 42 года
Роль: HR менеджер в компании 200 человек
Боли: Слишком много бумажной работы, нужна автоматизация
Цели: Упростить управление отпусками и отсутствиями
Опыт: Средний с ПО, использует Excel и Word
```

---

### User Stories (Пользовательские истории)

Описание функции с точки зрения пользователя:

```
As a [role], I want to [action], so that [value]

Пример:
As a buyer, I want to filter products by price,
so that I can find products within my budget.
```

**Для разработчиков:** Дает контекст ценности, а не просто "сделать фильтр по цене".

---

### User Flows (Пути пользователя)

Визуальное представление пути пользователя через систему:

```
Вход
  ↓ [Клиент нажимает "Купить"]
Каталог товаров
  ↓ [Выбирает товар]
Страница товара
  ↓ [Добавляет в корзину]
Корзина
  ↓ [Идет к оплате]
Оформление заказа
  ↓ [Вводит адрес]
Способ доставки
  ↓ [Выбирает способ оплаты]
Платеж
  ├─ [Успех] → Подтверждение
  └─ [Ошибка] → Повторить платеж
```

---

### Usability Testing (Тестирование юзабилити)

**Как это работает:**

1. **Подготовка** — готовим задачи для пользователя
2. **Тестирование** — пользователь выполняет задачи
3. **Наблюдение** — смотрим как он работает, записываем
4. **Анализ** — что прошло хорошо, где были проблемы

**Метрики:**
- Task Completion Rate (выполнил ли пользователь задачу?)
- Time on Task (сколько времени потребовалось?)
- Error Rate (сколько ошибок сделал?)
- Satisfaction (понравилось ли пользователю?)

## Сотрудничество аналитика и дизайнера

| Этап | Аналитик | Дизайнер |
|------|----------|----------|
| **Анализ требований** | Собирает требования | - |
| **User Research** | Проводит интервью | Анализирует insights |
| **Wireframing** | Уточняет детали | Создает скелет |
| **Design** | Валидирует дизайн | Создает макеты |
| **Prototyping** | Готовит сценарии | Делает интерактивный прототип |
| **Testing** | Организует исследование | Наблюдает и записывает |
| **Итерации** | Собирает feedback | Улучшает дизайн |

## Лучшие практики

✓ **Начните с потребности, не с решения** — "Пользователю нужно быстро найти товар" вместо "Добавим поле поиска"

✓ **Протестируйте с пользователями** — не полагайтесь только на ваше мнение

✓ **Итеративный подход** — версия 1 → feedback → версия 2 → и т.д.

✓ **Документируйте решения** — почему выбрали именно этот дизайн?

✓ **Доступность** — интерфейс должен работать для всех (люди с ограничениями)
