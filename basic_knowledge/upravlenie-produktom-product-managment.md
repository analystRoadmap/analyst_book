# Управление продуктом (Product Management)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️ BA |

> **Product Management** — управление жизненным циклом продукта, включая планирование, приоритизацию функций, работу с заинтересованными сторонами и доставку ценности пользователям.

## Разница между Product Manager и Business Analyst

| Аспект | Product Manager (PM) | Business Analyst (BA) |
|--------|-------------------|-------------------|
| **Фокус** | Стратегия, vision, бизнес-модель | Требования, процессы, их реализация |
| **Горизонт планирования** | 6-12 месяцев | 1-3 месяца (спринт) |
| **Главный вопрос** | "Какой продукт нам нужно сделать?" | "Как реализовать этот продукт?" |
| **Взаимодействие** | С клиентами, в фокус-группах | С разработчиками, QA |
| **Документ** | Product Roadmap, PRD | Требования, User Stories |

## Роль BA в управлении продуктом

Хотя BA не обязательно Product Manager, он часто работает в тесном контакте с PM:

- **Помогает переводить** vision PM в требования
- **Проводит анализ** осуществимости функций
- **Готовит прототипы** для валидации с клиентами
- **Документирует** принятые решения
- **Отслеживает** реализацию требований

## Инструменты для управления продуктом

### 1. Jira

**Назначение:** Управление задачами, требованиями, прогрессом разработки

```
Структура Jira:
├─ Epic (большая фича)
│  ├─ Story (пользовательская история)
│  │  ├─ Task (техническая задача)
│  │  ├─ Task (техническая задача)
│  │  └─ Bug (ошибка)
│  └─ Story (еще одна история)
└─ Epic (еще одна большая фича)
```

**Для Product Manager'а:**
- Roadmap View (план на несколько месяцев)
- Release Notes (что вышло в релизе)
- KPI и метрики

**Для Business Analyst'а:**
- Требования в Description
- Acceptance Criteria для каждой User Story
- Linking между requirements и tasks

### 2. Confluence

**Назначение:** Документирование требований, процессов, решений

```
Структура:
├─ Product Documentation
│  ├─ Product Vision & Strategy
│  ├─ Requirements (по фичам)
│  ├─ Processes
│  └─ Design & Prototypes
├─ Project Plans
│  ├─ Release Plan v1.0
│  ├─ Sprint Plans
│  └─ Risk Register
└─ Meeting Notes
```

**Интеграция с Jira:**
- Ссылка на requirements в Epic description
- Страницы Confluence в Jira issues
- Встроенные таблицы Jira issues в Confluence

### 3. Trello / Monday / Asana

**Назначение:** Упрощенное управление задачами (для маленьких команд)

```
Доска Kanban:
To Do | In Progress | In Review | Done
```

### 4. Azure DevOps

**Назначение:** Интегрированное управление для Microsoft-ориентированных компаний

## Product Roadmap

**Roadmap** — план развития продукта на 6-12 месяцев

### Структура Roadmap'а

```
2024 Q1: Foundation
├─ Feature A (User login)
├─ Feature B (Payment integration)
└─ Feature C (Notifications)

2024 Q2: Growth
├─ Feature D (Search)
├─ Feature E (Recommendations)
└─ Feature F (Analytics)

2024 Q3: Scale
├─ Feature G (API)
├─ Feature H (Mobile app)
└─ Performance optimization
```

### На что обращают внимание

- **Timeline** — когда планируется релиз
- **Priority** — важность для бизнеса
- **Resources** — сколько людей нужно
- **Dependencies** — какие фичи зависят от других
- **Value** — какую ценность добавляет

## Demo (Демонстрация функциональности)

**Demo** — показ реализованной функциональности заинтересованным сторонам

### Когда проводится Demo?

- **После спринта** (Sprint Review)
- **Перед релизом** (Release Demo)
- **Для клиента** (Client Demo)
- **Для бизнеса** (Stakeholder Demo)

### Структура хорошей Demo

```
1. Контекст (1 мин)
   "Мы разработали функцию поиска по требованию клиента"

2. Проблема (1 мин)
   "До этого пользователи не могли быстро найти товар"

3. Решение (2-3 мин)
   [Показываем в действии]
   - Где находится поиск?
   - Какие фильтры доступны?
   - Как быстро он работает?

4. Результат (1 мин)
   "Время поиска товара упало с 5 минут до 30 секунд"

5. Q&A (1-2 мин)
   "Есть вопросы?"
```

### Советы для успешной Demo

✓ **Репетируйте** — не импровизируйте
✓ **Используйте реальные данные** — не "Lorem ipsum"
✓ **Покажите happy path и исключения** — что если что-то сломается?
✓ **Говорите о ценности** — не о технических деталях
✓ **Будьте готовы к feedback** — записывайте замечания
✓ **Заканчивайте рано** — не затягивайте

## Метрики успеха продукта

| Метрика | Что измеряет | Норма |
|---------|-------------|-------|
| **User Engagement** | Активные пользователи | > 70% DAU |
| **Retention Rate** | % пользователей через месяц | > 40% |
| **Churn Rate** | % уходящих пользователей | < 5% в месяц |
| **NPS (Net Promoter Score)** | Готовность рекомендовать (1-10) | > 40 |
| **CSAT** | Удовлетворенность (1-5) | > 4.0 |
| **Time to Value** | Как быстро пользователь получает пользу | < 5 минут |

## Взаимодействие BA и PM

| Ситуация | BA делает | PM делает |
|----------|-----------|----------|
| **Планирование новой фичи** | Детально описывает требования | Определяет когда и зачем |
| **Исследование рынка** | Анализирует возможности реализации | Беседует с клиентами |
| **Приоритизация** | Предлагает сложность реализации | Выбирает что первое |
| **Demo** | Может показывать техническую часть | Рассказывает о ценности |
| **Post-release анализ** | Проверяет соответствие требованиям | Смотрит на метрики успеха |

## Типичные ошибки

❌ **Разработка без требований** — "Давайте угадаем что нужно клиенту"
❌ **Вся феча в один спринт** — требует пилота, валидации, итераций
❌ **Демо только успешные сценарии** — клиент нужно видеть и ошибки
❌ **Игнорирование feedback** — показали, услышали замечания, но не внедрили
❌ **Фичи вместо решения проблем** — "Добавим кнопку" вместо "Решим проблему пользователя"
