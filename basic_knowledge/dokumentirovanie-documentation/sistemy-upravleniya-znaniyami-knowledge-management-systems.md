# Системы управления знаниями (Knowledge Management Systems)

| Грейд | Роль |
|-------|------|
| 🔵 Middle | 🅱️🅰️ BA+SA |

> **KMS (Knowledge Management Systems)** — платформы для сбора, структурирования, хранения и совместного использования информации, которые превращают разрозненные знания в организованные и доступные ресурсы.

## Зачем нужна KMS?

**Сценарий без KMS:**
```
Новый сотрудник пришел
  "Где документы?"
  "В Confluence странице 3... нет подожди 4... или может в общей папке?"
  "Кто это писал?" "Петр, но он в отпуске до октября"

Результат: неправильная информация, потеря времени, ошибки
```

**С хорошей KMS:**
```
Новый сотрудник входит в систему
  Видит структурированное содержимое
  Находит процесс за 30 секунд
  Видит дату обновления и контакт автора
  Может оставить вопрос в комментариях
```

## Популярные KMS платформы

### 1. Confluence (Atlassian)

**Лучше для:** Документация, требования, процессы

```
Возможности:
✓ Создание wiki-страниц с иерархией
✓ Интеграция с Jira (ссылки требований)
✓ Встроенный поиск и индексирование
✓ Версионирование и история изменений
✓ Комментарии и обсуждения
✓ Шаблоны (для требований, процессов)
✓ Макросы (вставить диаграмму, таблицу)
```

**Пример структуры:**
```
Проект "E-Commerce"
├─ Требования
│  ├─ Функциональные (50 страниц)
│  ├─ Нефункциональные (10 страниц)
│  └─ User Stories (100+ шаблонов)
├─ Процессы
│  ├─ Оформление заказа (BPMN)
│  └─ Возврат товара (BPMN)
├─ Дизайн
│  ├─ Прототипы (ссылки на Figma)
│  └─ Гайдлайны UI
└─ Встречи и решения
```

### 2. Notion

**Лучше для:** Интегрированная база знаний, управление проектами

```
Возможности:
✓ Таблицы базы данных с фильтрацией
✓ Множество типов контента (текст, таблицы, доски)
✓ Встроенная фильтрация и сортировка
✓ API для интеграций
✓ Шаблоны и воспроизводимые блоки
✓ Совместное редактирование в реальном времени
```

**Пример использования:**
```
Database: Требования
├─ Title: "Добавить фильтр в каталог"
├─ Статус: In Progress
├─ Приоритет: High
├─ Ответственный: @Alice
├─ Дата дедлайна: 2024-02-15
└─ Связанные: Epic #3, Task #45

Database: User Stories
├─ Title: "Как клиент, я хочу..."
├─ Acceptance Criteria: (ссылка на требование)
├─ Тест-кейсы: (встроенные таблицы)
└─ Статус: Reviewed
```

### 3. Git + Markdown (Requirements-as-Code)

**Лучше для:** Версионируемые требования, совместная разработка

```
Структура:
/requirements
├─ /features
│  ├─ feature-001-search.md
│  ├─ feature-002-filters.md
│  └─ feature-003-sorting.md
├─ /processes
│  ├─ process-01-ordering.md
│  └─ process-02-returns.md
└─ /nfr
   ├─ performance.md
   ├─ security.md
   └─ scalability.md

История изменений: git log показывает все версии
Обсуждение: Pull Request comments для обратной связи
Версионирование: git tags для релизов
```

**Преимущества:**
- Версионирование как в коде
- Pull requests для обзора
- Конфликты явные и разрешимые
- Можно коммитить вместе с кодом

### 4. Microsoft SharePoint

**Лучше для:** Корпоративное хранилище, бизнес-документы

```
Возможности:
✓ Интеграция с Office 365 (Word, Excel, PowerPoint)
✓ Версионирование документов
✓ Поиск по метаданным
✓ Workflow автоматизация
✓ Синхронизация локальной папки
```

## Сравнение инструментов

| Критерий | Confluence | Notion | Git+Markdown | SharePoint |
|----------|-----------|--------|-------------|-----------|
| **Простота обучения** | Средняя | Легкая | Сложная | Средняя |
| **Поиск** | Отличный | Хороший | Базовый | Хороший |
| **Версионирование** | +/- | Есть | Отличное | Есть |
| **Офлайн** | Нет | Нет | Да (Git) | Да |
| **Цена** | $$$ | $-$$ | Бесплатно | $$$ |
| **Для разработчиков** | Средне | Плохо | Отлично | Плохо |
| **Масштабируемость** | Хорошая | Хорошая | Отличная | Отличная |

## Структурирование знаний в KMS

### 1. По типам документов

```
├─ Требования
│  ├─ Функциональные
│  ├─ Нефункциональные
│  └─ User Stories
├─ Процессы
├─ Дизайн
├─ Тестирование
└─ Операции
```

### 2. По проектам

```
├─ Проект A
│  ├─ Требования
│  ├─ Процессы
│  └─ Дизайн
├─ Проект B
│  └─ [Аналогично]
└─ Общее
   ├─ Шаблоны
   ├─ Best practices
   └─ Стандарты компании
```

### 3. По ролям

```
├─ Для аналитиков
│  ├─ Шаблоны требований
│  ├─ Процессы анализа
│  └─ Best practices
├─ Для разработчиков
│  ├─ API документация
│  ├─ Architecture guide
│  └─ Code examples
└─ Для менеджмента
   └─ KPI и метрики
```

## Лучшие практики KMS

### Организация

✓ **Понятная структура** — новый человек сориентируется без помощи
✓ **Единственный источник истины** — одна версия документа, а не копии везде
✓ **Теги и метаданные** — легче находить через фильтры
✓ **Индекс и навигация** — оглавление на главной странице

### Поддержка

✓ **Дата обновления видна** — знаешь, насколько свежая информация
✓ **Автор и контакт** — можно задать вопрос
✓ **Версионирование** — можешь вернуться к старой версии
✓ **Archive** — старые документы не загромождают систему

### Поощрение использования

✓ **Интеграция с рабочим процессом** — KMS в месте, где люди работают
✓ **Шаблоны** — начало работы за 2 клика
✓ **Примеры** — показать, как правильно
✓ **Обучение** — научить новых сотрудников

## Типичные ошибки KMS

❌ **Пустая система** — создали, но никто не заполняет
❌ **Множество копий документов** — не один источник истины
❌ **Устаревшая информация** — "Обновлено 2 года назад"
❌ **Никакой структуры** — всё в одну кучу, невозможно найти
❌ **Слишком сложная система** — люди предпочитают свои папки
