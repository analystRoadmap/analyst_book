# Документирование (Documentation)

| Грейд | Роль |
|-------|------|
| 🟢 Junior | 🅱️🅰️ BA+SA |

> **Документирование** — это процесс создания и организации информации о требованиях, процессах и архитектуре системы, которая делает знания доступными, понятными и сохраняемыми во времени.

## Две стороны документирования

### 1. Артефакты (Документы, которые создает аналитик)

Это то, что аналитик пишет:

| Документ | Назначение | Аудитория |
|----------|-----------|----------|
| **Требования (Requirements)** | Что нужно сделать | Разработчики, QA |
| **Процессы (BPMN/диаграммы)** | Как работает бизнес | Команда, менеджмент |
| **Прототипы (Wireframes)** | Как выглядит интерфейс | Дизайнеры, клиент |
| **User Stories** | Ценность для пользователя | Вся команда |
| **Спецификации (Specs)** | Технические детали | Разработчики |
| **Чек-листы (Checklists)** | Как проверить качество | QA инженеры |

### 2. Системы организации знаний (Knowledge Management Systems)

Это как и где хранится информация:

- **Где:** Confluence, Notion, Wiki, Git, SharePoint
- **Как организовано:** структура папок, теги, поиск
- **Кто имеет доступ:** разные уровни прав доступа
- **Как обновляется:** версионирование, история изменений

## Виды документов и их назначение

### Business Documents

**Бизнес-требования (Business Requirements Document, BRD)**
- Что клиент хочет получить?
- На каком языке? Простой язык без технизма
- Кто пишет? Аналитик с клиентом
- Как выглядит? Описание проблемы, целей, результатов ожидания

---

**Процессы (Business Process Models)**
- Как работает бизнес сейчас и как должен работать?
- Формат: BPMN диаграмма или текстовое описание
- Для кого? Бизнес, разработчики, аналитики
- Включает: роли, шаги, условия, исключения

---

### Technical Documents

**Функциональные требования (Functional Requirements)**
- Что должна делать система?
- Описание функций, экранов, взаимодействий
- На кого направлено? На разработчиков
- Формат: подробное текстовое описание

---

**Нефункциональные требования (Non-Functional Requirements)**
- Как быстро работает? (Performance)
- Насколько надежна? (Reliability)
- Как защищена? (Security)
- На сколько пользователей рассчитана? (Scalability)

---

**Технические спецификации (Technical Specifications)**
- Как реализовать требования?
- Архитектура, базы данных, API
- Написано архитектором или разработчиком (не аналитиком)

---

### Design Documents

**Прототипы и макеты (Wireframes & Mockups)**
- Как выглядит интерфейс?
- Какие элементы на экране?
- Какие взаимодействия?

---

**Пользовательские потоки (User Flows)**
- По какому пути проходит пользователь?
- Какие экраны видит?
- Какие действия выполняет?

---

### Quality Documents

**Тест-кейсы (Test Cases)**
- Что тестировать?
- Как тестировать?
- Какие ожидаемые результаты?

---

**Критерии приемки (Acceptance Criteria)**
- Когда требование считается выполненным?
- Какие условия должны быть выполнены?

---

## Системы организации знаний

### Когда использовать какую систему?

| Система | Лучше для | Примеры |
|---------|----------|--------|
| **Confluence** | Документация, требования, процессы | Вики-страницы, шаблоны требований |
| **Notion** | Базы знаний, таблицы, интеграции | Каталог требований, управление проектом |
| **Git + Markdown** | Версионируемые требования | Requirements-as-code, совместная разработка |
| **Figma** | Прототипы и дизайн | Макеты, компоненты, design system |
| **Jira** | Управление задачами и требованиями | User stories, эпики, трэкинг статуса |
| **SharePoint** | Корпоративное хранилище | Документы, процедуры, политики |

### Принципы организации знаний

**1. Таксономия (Иерархическая структура)**

```
Проект X
├─ Требования
│  ├─ Функциональные
│  ├─ Нефункциональные
│  └─ Пользовательские истории
├─ Процессы
│  ├─ As-Is (текущие)
│  └─ To-Be (желаемые)
├─ Дизайн
│  ├─ Wireframes
│  └─ Макеты
└─ Тестирование
   ├─ Тест-кейсы
   └─ Результаты
```

**2. Метаданные и теги**

Каждый документ должен иметь:
- Версия
- Автор
- Дата создания/обновления
- Статус (Draft, Review, Approved, Archived)
- Теги (категория, проект, версия)

**3. Поиск и навигация**

- Полнотекстовый поиск
- Ссылки между документами
- Навигационное меню

## Лучшие практики документирования

### Написание

✓ **Ясный язык** — короткие предложения, без сложной терминологии
✓ **Структурированность** — заголовки, списки, таблицы
✓ **Примеры** — показать реальный случай использования
✓ **Визуализация** — диаграммы вместо текста где возможно
✓ **Версионирование** — отслеживать кто что менял

### Организация

✓ **Единая структура** — все документы структурированы одинаково
✓ **Единообразное именование** — понятные названия файлов
✓ **Индекс и оглавление** — как найти нужное
✓ **Перекрестные ссылки** — связь между документами

### Поддержка

✓ **Регулярное обновление** — устаревшие документы хуже чем их отсутствие
✓ **История версий** — можно вернуться к старой версии если нужно
✓ **Правила доступа** — кто может смотреть/редактировать
✓ **Архивация** — старые документы не смешиваются с актуальными

## Типичные ошибки

❌ **Документация "когда-нибудь потом"** — обычно "потом" никогда не наступает
❌ **Word-документы вместо системы управления** — невозможно искать, ссылаться
❌ **Один аналитик хранит всё в голове** — при смене человека теряется всё
❌ **Документы, которые никто не читает** — пустая трата времени
❌ **Слишком детальные документы** — никто не будет читать 50 страниц
