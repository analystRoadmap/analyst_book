# Трассировка требований (Requirements Traceability)

| Грейд | Роль |
|---|---|
| 🔵 Middle | SA |

> **Трассировка требований — это "страховка"** от потери требований между идеей и production.

## Что такое трассировка требований?

**Трассировка требований (RTM)** — это процесс документирования и отслеживания связей между:
- Бизнес-требованиями
- Функциональными требованиями
- Дизайном системы
- Кодом реализации
- Тест-кейсами

```
Пример цепочки:

BR-001: Увеличить конверсию на 15%
    ↓ (связь)
FR-001: Форма быстрого заказа
    ↓ (связь)
DESIGN: Макет формы в Figma
    ↓ (связь)
CODE: Компоненты Vue (QuickOrderForm.vue, validators.ts)
    ↓ (связь)
TEST: Тест-кейсы (TC-001 до TC-010)
    ↓ (связь)
PRODUCTION: Развернуто на боевом сервере
```

## Зачем нужна трассировка?

### Ответить на критичные вопросы

```
Вопрос 1: "Требование FR-001 реализовано правильно?"
→ Смотрим RTM → находим CODE-001 (QuickOrderForm.vue)
→ Находим TEST-001, TEST-002, TEST-003
→ Если все тесты passed, требование реализовано!

Вопрос 2: "Если изменим требование FR-002, что еще надо менять?"
→ Смотрим RTM → FR-002 связано с CODE-002, CODE-003
→ Связано с TEST-004, TEST-005
→ Нужно обновить эти компоненты и переписать тесты

Вопрос 3: "Все ли требования закрыты тестами?"
→ Смотрим RTM → есть ли тесты для каждого FR?
→ Если есть F R без тестов → риск, требование может сломаться!

Вопрос 4: "Есть ли лишний код, который никому не нужен?"
→ Смотрим RTM → CODE-099 не связан ни с одним требованием
→ Это потеря времени → удалить код? Или добавить требование?

Вопрос 5: "На сколько процентов требования реализованы?"
→ Смотрим RTM → 8 из 10 требований в статусе "DONE"
→ Реализация на 80% готова
```

### Когда особенно критично

| Сценарий | Почему RTM важна |
|---|---|
| **Критичные системы** (медицина, авиация, финансы) | DO-178C, IEC 61508 требуют трассировки |
| **Большие проекты** (50+ разработчиков) | Легко потерять требования в хаосе |
| **Долгие проекты** (1-3 года) | Люди уходят, знание теряется |
| **Много изменений** | Нужно отслеживать impact analysis |
| **Регуляторное воздействие** | Аудиторы требуют доказательства |

## Матрица трассировки требований (RTM)

### Структура простой RTM

```
| ID | Описание | Дизайн | Код | Тест | Статус | Комментарий |
|---|---|---|---|---|---|---|
| FR-001 | Форма заказа | Screen_01 | QuickOrder.vue | TC-001 | DONE | Завершено |
| FR-002 | Валидация ФИ | Screen_01 | validators.ts | TC-002 | IN PROGRESS | На тестировании |
| FR-003 | Сообщение успеха | Modal_01 | Modal.vue | TC-003 | PENDING | Не начиналось |
| FR-004 | Обработка ошибок | ErrorMsg_01 | ErrorHandler.ts | TC-004,TC-005 | DONE | Двойной тест |
| NFR-001 | Производительность | - | - | TC-006 | DONE | Load test прошёл |
```

### Структура детальной RTM (Excel)

```
Столбцы:

1. REQ_ID: FR-001, FR-002, ...
2. REQ_NAME: Описание требования
3. PRIORITY: CRITICAL, HIGH, MEDIUM, LOW
4. DESIGN_COMP: Figma компонент или экран
5. CODE_FILE: Путь к файлу (src/components/QuickOrder.vue)
6. CODE_CLASS: Класс или функция (QuickOrderForm, OrderValidator)
7. TEST_ID: TC-001, TC-002, ... (может быть несколько)
8. STATUS: PENDING, IN_PROGRESS, DONE, BLOCKED
9. COMPLETENESS: % готовности (0%, 25%, 50%, 75%, 100%)
10. NOTES: Комментарии, блокеры
11. LAST_UPDATE: Дата последнего обновления
12. OWNER: Кто отвечает (разработчик, тестер)
```

## Как составлять RTM?

### Пошаговый процесс

```
Шаг 1: СБОР ТРЕБОВАНИЙ
→ Собрали все требования (FR-001 до FR-020)
→ Создали SRS документ
→ Требования утверждены

Шаг 2: ПРИСВОЕНИЕ ID
→ Каждому требованию уникальный ID:
   FR-001 (Functional Requirement 1)
   NFR-001 (Non-Functional Requirement 1)
   BR-001 (Business Requirement 1)

Шаг 3: СОЗДАНИЕ RTM (начальный этап)
→ Открыли Excel или Google Sheets
→ Создали колонки: ID, Описание, ...
→ Переписали все требования

Шаг 4: СВЯЗЫВАНИЕ С ДИЗАЙНОМ
→ Дизайнер создал макеты в Figma
→ В RTM добавили ссылку на макет:
   FR-001 → Figma: Quick-Order-Form (Screen 1)

Шаг 5: СВЯЗЫВАНИЕ С КОДОМ (во время разработки)
→ Разработчик пишет компонент QuickOrder.vue
→ В RTM обновляем:
   FR-001 → Code: src/components/QuickOrder.vue
   FR-001 → Status: IN_PROGRESS (50%)

Шаг 6: СВЯЗЫВАНИЕ С ТЕСТАМИ (во время тестирования)
→ Тестер пишет тест-кейсы TC-001, TC-002
→ В RTM добавляем:
   FR-001 → Test: TC-001, TC-002
   FR-001 → Status: DONE (100%)

Шаг 7: АКТУАЛИЗАЦИЯ (постоянно)
→ Если требование изменилось → обновляем RTM
→ Если требование отклонено → отмечаем REJECTED
→ Еженедельно проверяем актуальность
```

## Пример: Полная трассировка функции "Быстрый заказ"

### Бизнес-требование

```
BR-001: Увеличить конверсию на 15%

Почему:
- Клиенты отказываются при длинном процессе оформления
- Конкуренты (Wildberries) имеют быстрый заказ
- Это даст 2-3% выручки больше

Как измерим:
- Отношение завершенных заказов / посещений
- Сейчас: 2%, цель: 2.3%
```

### Функциональные требования

```
FR-001: Форма быстрого заказа должна содержать 3 поля: Имя, Отчество, Телефон
  Design:     Figma file: Quick-Order-Form (Screen 1-2)
  Code:       src/components/QuickOrderForm.vue
  Code:       src/utils/validators.ts (NameValidator, PatronymicValidator, PhoneValidator)
  Test:       TC-001: Форма отображается
              TC-002: Все 3 поля видны
              TC-003: Поля валидируются в браузере
  Status:     DONE (100%)
  Owner:      John Developer

FR-002: При успешном оформлении показать сообщение с номером заказа
  Design:     Figma file: Quick-Order-Form (Screen 3 - Success Modal)
  Code:       src/components/SuccessModal.vue
              src/services/OrderService.ts (createOrder method)
  Test:       TC-004: Успешное оформление
              TC-005: Modal показывает правильный номер
  Status:     DONE (100%)
  Owner:      John Developer

FR-003: При ошибке показать сообщение об ошибке с контактом поддержки
  Design:     Figma file: Quick-Order-Form (Screen 4 - Error Modal)
  Code:       src/components/ErrorModal.vue
              src/services/ErrorHandler.ts
  Test:       TC-006: Ошибка сервера (500)
              TC-007: Validation error
              TC-008: Network timeout
  Status:     DONE (100%)
  Owner:      John Developer

NFR-001: Время загрузки формы < 500 мс
  Design:     -
  Code:       Оптимизирована загрузка компонента
  Test:       TC-009: Performance test (Load Time)
  Status:     DONE (100%)
              Фактический результат: 280 мс
  Owner:      Jane Tester

NFR-002: Система должна обрабатывать 1000 req/sec
  Design:     -
  Code:       Load-balanced servers (3x Nginx + 5x Node.js)
  Test:       TC-010: Load test (1000 req/sec)
  Status:     DONE (100%)
              Фактический результат: 1200 req/sec
  Owner:      Jane Tester
```

### Матрица в виде таблицы

```
| ID | Описание | Компонент | Файлы кода | Тесты | % | Статус | Замечания |
|---|---|---|---|---|---|---|---|
| FR-001 | 3 поля формы | Screen 1-2 | QuickOrder.vue | TC-001:3 | 100% | DONE | ✓ Merged |
| FR-002 | Success message | Screen 3 | SuccessModal.vue | TC-004:5 | 100% | DONE | ✓ Production |
| FR-003 | Error handling | Screen 4 | ErrorModal.vue | TC-006:8 | 100% | DONE | ✓ Tested |
| NFR-001 | Performance | - | *optimized | TC-009 | 100% | DONE | 280ms OK |
| NFR-002 | Scalability | - | Load balance | TC-010 | 100% | DONE | 1200req/s |
```

## Анализ RTM: Ответы на вопросы

### Анализ 1: Требование FR-002 потребует изменений

```
Q: Если изменим FR-002 на "показать SMS вместо модального окна",
   что нужно обновить?

A: Смотрим RTM:
   FR-002 → Code: SuccessModal.vue, OrderService.ts
   FR-002 → Test: TC-004, TC-005

   Действия:
   1. Обновить компонент: вместо Modal → SMS sending
   2. Переписать тесты TC-004, TC-005 (не для modal, а для SMS)
   3. Обновить Figma дизайн
   4. Обновить RTM: новые компоненты, новые тесты

   Без RTM мы бы забыли про тесты!
```

### Анализ 2: Нет тестов для некоторого требования

```
Q: В RTM видим:
   FR-003 → Test: (пусто) ← ПРОБЛЕМА!

A: FR-003 (обработка ошибок) не имеет тестов!
   Это критичный дефект — требование может сломаться
   без нашего ведома.

   Действия:
   1. Срочно написать тесты для FR-003
   2. Запустить их
   3. Обновить статус в RTM
```

### Анализ 3: Orphaned код (код без требования)

```
Q: В коде видим файл src/api/legacy-payment.ts,
   но в RTM нет требования, которое его использует.

A: Это orphaned код — потеря ресурсов.

   Действия:
   1. Узнать, для чего это существует
   2. Если не нужно → удалить
   3. Если нужно → добавить требование в RTM
```

## Инструменты для управления RTM

### Простые решения

```
Excel / Google Sheets:
✓ Легко начать
✓ Не нужно учиться
✗ Сложнее с большими проектами
✗ Версионирование неудобно

Примеры:
- Таблица RTM (ID, Name, Design, Code, Test, Status)
- Фильтры по статусу (DONE, IN_PROGRESS, PENDING)
- Conditional formatting (красный = BLOCKED, зеленый = DONE)
```

### Продвинутые решения

```
Jira + Confluence:
✓ User Stories связаны с Tasks (код) и Tests
✓ Статус отслеживается автоматически
✓ История изменений сохраняется
✗ Требует обучения

GitHub Issues + Projects:
✓ Интегрирован с кодом (автоматическая связь)
✓ PR связаны с Issues
✓ Open source

Azure DevOps:
✓ Полнофункциональный инструмент
✓ Связь с кодом, тестами, builds
✗ Платный

RequirementTraceability.com:
✓ Специализированный инструмент
✓ Beautiful UI
✗ Дорого

Markdown + Git (для tech-teams):
✓ Версионирование в Git
✓ Все в одном репозитории
✗ Требует технических навыков
```

## RTM как живой документ

```
НАЧАЛО ПРОЕКТА (Неделя 1):
RTM создана с требованиями (FR-001 до FR-020)
Статус: 80% требований одобрено

РАЗРАБОТКА (Неделя 2-4):
Каждый день разработчики обновляют RTM:
- Новый код написан → обновляют Code column
- Требование меняется → обновляют Description

ТЕСТИРОВАНИЕ (Неделя 5-6):
Тестеры добавляют тест-кейсы в RTM:
- TC-001, TC-002, ... привязаны к требованиям
- Если тест fallen → обновляют Status

PRODUCTION (Неделя 7):
RTM финальная:
- 100% требований имеют статус DONE
- Все связи заполнены
- Это证証idence базис (документация)

PRODUCTION + 3 МЕСЯЦА:
Если вопрос "почему мы реализовали X именно так?"
→ Смотрим RTM → видим требование → видим обсуждение
```

## Best Practices

```
1. АКТУАЛЬНОСТЬ:
   ✓ Обновляйте RTM в реальном времени (не в конце проекта)
   ✓ Еженедельно проверяйте актуальность
   ✗ Не отставайте от разработки

2. АВТОМАТИЗАЦИЯ:
   ✓ Используйте Jira / GitHub для автоматических связей
   ✓ Требование → код (через branch naming)
   ✗ Не обновляйте RTM вручную, если можно автоматизировать

3. ЯСНОСТЬ:
   ✓ Требование ID должны быть понятные (FR-001, не "R1")
   ✓ Описание требования должно быть однозначно
   ✗ Не усложняйте по неопределенность

4. ДОСТУПНОСТЬ:
   ✓ Все члены команды должны видеть RTM
   ✓ Ссылка в Confluence, README, Jira
   ✗ Не прячьте RTM в чужой компьютер

5. ДЕТАЛЬНОСТЬ:
   ✓ Для критичных проектов — детальная RTM
   ✓ Для MVPs — простая RTM
   ✗ Не переусложняйте для небольших проектов
```
