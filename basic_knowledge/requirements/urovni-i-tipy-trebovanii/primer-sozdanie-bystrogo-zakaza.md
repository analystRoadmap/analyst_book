# Пример: Создание быстрого заказа

| Грейд | Роль |
|---|---|
| 🟢 Junior | BA + SA |

> **Практический пример** демонстрирует, как требование проходит все три уровня: от бизнес-цели до функциональных спецификаций.

## Контекст проблемы

Интернет-магазин теряет клиентов, которые отказываются от покупки при оформлении заказа. Стандартный процесс занимает 5-10 минут (заполнение адреса, выбор доставки, привязка карты и т.д.). Бизнес решил создать упрощенный вариант для быстрого заказа.

## Уровень 1: Бизнес-требование

```
BR-001: Увеличить конверсию заказов на 15%

Бизнес-правила:
- Общение с клиентом по имени-отчеству (деловой этикет)
- Все платежи через авторизованный шлюз
- Данные клиентов защищены согласно 152-ФЗ
- Процесс заказа интегрирован с CRM для анализа
```

## Уровень 2: Пользовательские требования (User Stories)

### User Story 1: Клиент оформляет быстрый заказ

```
Как клиент интернет-магазина,
я хочу оформить заказ за 2-3 минуты,
чтобы не потерять интерес к товару.

Критерии приемки:
- На форме быстрого заказа только необходимые поля: Имя, Отчество, Телефон
- Есть кнопка "Оформить быстрый заказ"
- После нажатия кнопки появляется подтверждение
- Я могу вернуться в каталог и продолжить покупки
```

### User Story 2: Оператор получает заявку

```
Как сотрудник колл-центра,
я хочу сразу увидеть новую заявку в системе,
чтобы перезвонить клиенту в течение 5 минут.

Критерии приемки:
- Заявка появляется в очереди обработки в реал-тайме
- Видны данные: Имя, Отчество, Телефон, Время заявки
- Есть история всех заявок клиента
- Я могу отметить заявку как обработанную
```

## Уровень 3: Функциональные требования

### FR-001: Форма быстрого заказа

```
Требование: Система должна отобразить форму на странице товара
с полями для ввода Имени, Отчества и номера телефона.

Детали реализации:
- Форма расположена в правой колонке под кнопкой "В корзину"
- Поля заполняются в следующем порядке: Имя → Отчество → Телефон
- При нажатии Tab происходит переход к следующему полю
- Кнопка "Оформить заказ" активна только если все поля заполнены

Примеры:
1. Пользователь вводит "Иван" в Имя → включается фокус на Отчество
2. Попытка отправить пустую форму → сообщение об ошибке
3. Все поля заполнены → кнопка активна и доступна для клика
```

### FR-002: Валидация Имени

```
Требование: Поле "Имя" должно содержать от 2 до 24 символов.

Правила:
- Минимум 2 символа (исключить "А", "В")
- Максимум 24 символа (длинные имена вроде "Константин")
- Не разрешены цифры и спецсимволы
- Первый символ преобразуется в заглавный

Примеры:
✓ Иван (4 символа)
✓ Мария (5 символов)
✗ И (1 символ) → ошибка
✗ Иван123 (содержит цифры) → ошибка
✗ Иван-Петр (спецсимволы) → предупреждение
```

### FR-003: Валидация Отчества

```
Требование: Поле "Отчество" должно содержать от 3 до 48 символов.

Правила:
- Минимум 3 символа
- Максимум 48 символов
- Буквы только русского алфавита + дефис
- Первый символ заглавный

Примеры:
✓ Иванович (8 символов)
✓ Петровна (8 символов)
✗ Ив (2 символа) → ошибка
✗ Иванович123 → ошибка
```

### FR-004: Валидация Телефона

```
Требование: Поле телефона должно начинаться с +7 и содержать 11 цифр.

Реализация:
- По умолчанию заполнено "+7"
- Пользователь вводит только цифры после +7
- Маска ввода: +7 (XXX) XXX-XX-XX
- При нажатии кнопки проверяется формат

Примеры:
✓ +7 (900) 123-45-67
✓ +7 (812) 456-78-90
✗ +7 (900) 123-45 → ошибка (неполный номер)
✗ +8 (900) 123-45-67 → ошибка (не +7)
```

### FR-005: Сообщение об успехе

```
Требование: После успешного оформления заказа показать сообщение.

Текст сообщения:
"Спасибо за заказ, [Имя] [Отчество]!
Заказ №[номер заказа] оформлен.
Сотрудник нашей службы поддержки
свяжется с вами в течение 30 минут.
Спасибо, что пользуетесь нашим сервисом!"

Поведение:
- Сообщение отобразить на 5 секунд
- Затем вернуться к исходной форме
- История заявки сохраняется в профиль пользователя
```

### FR-006: Обработка ошибки

```
Требование: При ошибке оформления показать подробное сообщение.

Сценарии ошибок:

1. Ошибка сервера (500):
   "Произошла техническая ошибка. Попробуйте позже."

2. Телефон уже заказывал (дубль):
   "Клиент с таким телефоном уже оформил заказ сегодня.
   Пожалуйста, позвоните в колл-центр: +7 (800) 555 35 35"

3. Сетевая ошибка:
   "Нет подключения к интернету. Проверьте соединение."

Все ошибки записываются в лог для анализа.
```

## Нефункциональные требования (NFR)

```
NFR-001: Производительность
- Загрузка формы: < 500 мс
- Валидация полей: < 100 мс (в браузере)
- Отправка заявки: < 2 сек (до получения ответа)

NFR-002: Доступность
- Форма доступна 24/7
- Uptime: 99.9% (максимум 43 минуты простоя в месяц)

NFR-003: Безопасность
- HTTPS для всех запросов
- Телефон хранится в зашифрованном виде
- Соответствие 152-ФЗ о защите данных

NFR-004: Масштабируемость
- Система выдерживает 10,000 одновременных пользователей
- 1,000 заявок в минуту (пиковая нагрузка)

NFR-005: Удобство
- Форма адаптивная (работает на мобильных)
- Шрифт не менее 14px для читаемости
- Контрастность текста не менее 4.5:1
```
