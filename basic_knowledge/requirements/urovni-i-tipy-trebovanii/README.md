# Уровни и типы требований

| Грейд | Роль |
|---|---|
| 🟢 Junior | BA + SA |

> **Трёхуровневая иерархия** требований обеспечивает связь между стратегией бизнеса и техническими решениями.

## Иерархия требований

Требования организуются в три уровня, каждый из которых служит основой для следующего:

```
┌─────────────────────────────────────────────┐
│  1️⃣ БИЗНЕС-ТРЕБОВАНИЯ (Business Requirements) │
│  Зачем? Какую цель преследует система?     │
└────────────────┬────────────────────────────┘
                 ↓
┌─────────────────────────────────────────────┐
│  2️⃣ ПОЛЬЗОВАТЕЛЬСКИЕ ТРЕБОВАНИЯ (User Reqs) │
│  Что? Как пользователь будет это делать?   │
└────────────────┬────────────────────────────┘
                 ↓
┌─────────────────────────────────────────────┐
│  3️⃣ ФУНКЦИОНАЛЬНЫЕ/НЕФУНКЦИОНАЛЬНЫЕ ТРЕБОВАНИЯ │
│  Как? Какие именно функции реализовать?    │
└─────────────────────────────────────────────┘
```

## Уровень 1: Бизнес-требования

**Определение:** Высокоуровневые бизнес-цели организации, которые должна поддерживать система.

### Примеры бизнес-требований

| Тип | Пример |
|---|---|
| Доход | Увеличить выручку на 20% через онлайн-канал продаж |
| Эффективность | Сократить время обработки заказа с 2 часов до 15 минут |
| Риск | Соответствовать требованиям закона 152-ФЗ о защите данных |
| Конкурентность | Зарезервировать место в памяти клиента как инновационный сервис |

### Бизнес-правила (Business Rules)

Это набор ограничений и стандартов, определяющих работу системы:

- Все платежи должны проходить через платежный шлюз Sberbank
- Клиент обращается только по имени-отчеству (деловой этикет)
- Скидка для постоянных клиентов составляет не более 30%
- Данные клиентов хранятся на серверах в РФ

## Уровень 2: Пользовательские требования (User Requirements)

**Определение:** Задача, которую пользователь должен выполнить в системе, описанная с его точки зрения.

### Формат: User Story

```
Как [тип пользователя],
я хочу [действие],
чтобы [получить выгоду/результат].

Критерии приемки:
- Критерий 1
- Критерий 2
- Критерий 3
```

### Примеры пользовательских требований

**User Story 1:**
```
Как клиент электронного магазина,
я хочу быстро оформить заказ за 2 минуты,
чтобы не потерять интерес к покупке.

Критерии приемки:
- Заполнение формы занимает не более 2 минут
- Нужны только ФИО, email и телефон
- Есть кнопка "Оформить быстрый заказ"
```

**User Story 2:**
```
Как сотрудник поддержки,
я хочу видеть новые заявки в реал-тайме,
чтобы не пропустить срочные вопросы.

Критерии приемки:
- Список заявок обновляется каждые 5 секунд
- Новая заявка выделена цветом
- Есть звуковое уведомление
```

## Уровень 3: Функциональные требования

**Определение:** Точное описание того, что система должна делать в конкретной ситуации.

### Структура функционального требования

```
REQ-001: [ID требования]
Описание: [Краткое описание]
Тип: Функциональное требование
Приоритет: Высокий/Средний/Низкий
Сложность оценки: Относительные единицы

Требование:
Система должна [выполнять действие] [при условии] [с результатом].

Примеры сценариев:
1. Базовый сценарий: ...
2. Альтернативный сценарий: ...
3. Сценарий ошибки: ...
```

### Примеры функциональных требований

**Требование FR-001:**
```
Система должна позволить клиенту ввести «Имя», «Отчество»
и «номер телефона» и нажать кнопку «Оформить заказ».
```

**Требование FR-002:**
```
При успешном оформлении заказа система должна отобразить:
"Заказ №[номер] оформлен. Спасибо за покупку.
Сотрудник поддержки свяжется с вами в течение 30 минут."
```

**Требование FR-003:**
```
При ошибке оформления система должна показать:
"Произошла ошибка. Попробуйте позже или позвоните нам: +7 (800) 555 35 35"
```

## Уровень 3б: Нефункциональные требования (NFR)

**Определение:** Требования к качеству, производительности, надежности, безопасности системы.

Примеры NFR:
- Время загрузки: < 500 мс
- Доступность: 99.9% uptime
- Количество одновременных пользователей: 10,000
- Безопасность: HTTPS, шифрование паролей
- Локализация: русский и английский языки

## Атрибуты качества (Quality Attributes)

Подтип NFR, описывающий требуемые характеристики сервиса:

| Атрибут | Описание | Пример |
|---|---|---|
| Производительность | Скорость выполнения | Поиск результата за < 100 мс |
| Масштабируемость | Рост нагрузки | 10,000 пользователей одновременно |
| Безопасность | Защита данных | Двухфакторная аутентификация |
| Надежность | Устойчивость к сбоям | Восстановление за < 5 минут |
| Удобство | Простота использования | Не более 3 кликов для основной операции |

## Практический пример: Быстрый заказ

| Уровень | Требование |
|---|---|
| **Бизнес** | Увеличить конверсию на 15% благодаря упрощенному заказу |
| **Пользовательское** | Я как клиент хочу заказать товар за 2 минуты |
| **Функциональное** | Форма содержит поля: ФИО, email, телефон |
| **NFR** | Загрузка формы < 500 мс, доступна 99.9% времени |

## Документ (в контексте анализа)

В анализе под "документом" понимается **любой контейнер информации**:
- Текстовой документ (Word, Google Docs)
- Таблица (Excel, Sheets)
- Диаграмма (UML, BPMN, DFD)
- База данных
- Вики (Confluence)
- Презентация

Главное — четкое фиксирование и доступность информации для команды.
