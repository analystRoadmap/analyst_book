# Разработка и управление требований

| Грейд | Роль |
|---|---|
| 🔵 Middle | BA + SA |

> **Требования живут и развиваются** на протяжении всего жизненного цикла проекта.

## Этап 1: Выявление требований (Elicitation)

**Определение:** Процесс сбора информации о потребностях заинтересованных сторон.

### Кто участвует?

| Роль | Что знает |
|---|---|
| **Product Manager** | Видение, стратегия, рынок |
| **Бизнес-аналитик** | Процессы, боли бизнеса |
| **Пользователи** | Как реально они работают |
| **Разработчики** | Технические ограничения |
| **Тестеры** | Что может сломаться |
| **Дизайнеры** | Как должно выглядеть |

### Методы выявления

```
Интервью:
- 1-на-1 с ключевыми стейкхолдерами
- Вопросы: "Какие боли в текущем процессе?", "Идеальное решение?"
- Выход: Список требований в черновом виде

Наблюдение:
- Посидеть рядом с пользователем 2-3 часа
- Смотреть, как он реально работает (не рассказывает)
- Выход: Скрытые требования, "обходы" (хаки)

Совместные сессии:
- Собрать 5-10 человек вместе
- Мозговой штурм "что нужно изменить?"
- Выход: Быстро много идей, но может быть беспорядок

Анализ документов:
- Текущие процессы, регламенты, жалобы
- Изучить, как работает сейчас
- Выход: Контекст, предыстория
```

### Выход: Список требований (черновой)

```
Требование 1: Клиент должен оформить заказ за < 5 мин
Требование 2: Оператор должен получить уведомление о новом заказе
Требование 3: Система должна хранить историю заказов
Требование 4: ФИО клиента должно быть по имени-отчеству
Требование 5: ???
```

## Этап 2: Анализ требований (Analysis)

**Определение:** Глубокое изучение требований на полноту, непротиворечивость и проверяемость.

### Чек-лист анализа требования

```
□ Полнота: Требование достаточно детально?
  ✓ "Клиент может заполнить форму с ФИ и телефоном"
  ✗ "Клиент делает что-то с формой" (слишком расплывчато)

□ Однозначность: Все интерпретируют одинаково?
  ✓ "Время загрузки < 500 мс" (метрика четкая)
  ✗ "Система должна быть быстрой" (слово "быстрая" для каждого свое)

□ Проверяемость: Можно ли протестировать?
  ✓ "Форма показывается на странице товара"
  ✗ "Форма должна быть красивой" (как тестировать красоту?)

□ Осуществимость: Можно ли это реализовать за сроки/бюджет?
  ✓ "3 поля ввода"
  ✗ "Система распознает речь пользователя" (может быть слишком сложно)

□ Независимость: Требование не зависит от других?
  ✓ "Платеж обрабатывается через Sberbank" (четко описано)
  ✗ "Система платит..." (зависит от другого требования)

□ Приоритет: Что важнее?
  🔴 CRITICAL — без этого система не работает
  🟠 HIGH — важно, влияет на основные процессы
  🟡 MEDIUM — хорошо иметь, но можно без этого
  🔵 LOW — nice-to-have, для следующей версии
```

### Разрешение конфликтов

```
Конфликт 1: Требования противоречат друг другу
Требование А: "Система шифрует все данные (для безопасности)"
Требование Б: "Система обрабатывает 10,000 запросов/сек (для скорости)"

Решение:
→ Обсудить с заинтересованными сторонами
→ Найти компромисс (шифровать только личные данные)
→ Один требование может быть LOW приоритет

Конфликт 2: Требования конкурируют за ресурсы
Требование А: "Добавить 10 новых фич" (120 часов)
Требование Б: "Оптимизировать производительность" (80 часов)
Бюджет: 150 часов

Решение:
→ Выбрать приоритеты с PM
→ Отложить LOW приоритет требования
→ Может быть разбить на версии (v1.0, v1.1)
```

### Выход: Уточненный список требований

```
REQ-001: Форма быстрого заказа
  - Приоритет: CRITICAL
  - Статус: Одобрено
  - Конфликтов: Нет

REQ-002: Уведомление оператора
  - Приоритет: HIGH
  - Статус: На уточнение (время ответа?)
  - Конфликтов: Зависит от REQ-001

REQ-003: История заказов
  - Приоритет: MEDIUM
  - Статус: Одобрено
  - Конфликтов: Нет
```

## Этап 3: Документирование (Specification)

**Определение:** Запись требований в согласованном, понятном формате.

**Выходные документы:**
- Спецификация требований (SRS) в Confluence
- Диаграммы (UML, BPMN, DFD) в draw.io
- User Stories в Jira (если Agile)
- Матрица трассировки требований (RTM) в Excel

(Подробнее см. раздел "Документирование требований")

## Этап 4: Утверждение требований (Validation)

**Определение:** Убедиться, что требования соответствуют ожиданиям всех сторон.

### Методы валидации

**Обзор требований (Review):**
```
Участники: Аналитик, PM, Tech Lead, Бизнес-стейкхолдер
Длительность: 60-90 минут
Процесс:
1. Аналитик предоставляет SRS за 2-3 дня до встречи
2. Все читают и готовят вопросы
3. На встрече обсуждают по требованиям
4. Выносят решение: APPROVED / NEED CHANGES / REJECTED
5. Аналитик исправляет в течение 3 дней

Выход: Акт утверждения требований (signed-off)
```

**Прототипирование для валидации:**
```
Создать интерактивный прототип формы быстрого заказа
→ Показать реальным пользователям (5-10 человек)
→ Наблюдать, как они используют (usability test)
→ Собрать обратную связь: "Это правильно?", "Что не так?"
→ Исправить требования на основе feedback

Выход: Валидированные требования
```

**User Acceptance Testing (UAT) Criteria:**
```
Каждое требование должно иметь критерии приемки:

REQ-001: Форма быстрого заказа
✓ Форма содержит 3 поля: Имя, Отчество, Телефон
✓ Все поля обязательны
✓ При клике на "Оформить" отправляется на сервер
✓ Получаю подтверждение "Заказ принят"
✓ Форма появляется на странице товара без скролла

Выход: Список тестов для QA
```

## Этап 5: Трассировка требований (Traceability)

**Определение:** Отслеживание, как требование реализуется в коде и тестах.

(Подробнее см. раздел "Трассировка требований")

## Управление изменениями требований

Требования меняются! Это нормально. Но нужен процесс:

```
СЦЕНАРИЙ: Бизнес просит добавить 2FA в формы

1️⃣ CHANGE REQUEST:
   - Что? "Добавить двухфакторную аутентификацию"
   - Почему? "Усилить безопасность"
   - Когда? "Для следующего релиза"
   - Кто просит? "CTO + Security Team"

2️⃣ ОЦЕНКА ВЛИЯНИЯ:
   - Какие требования затронуты?
     • REQ-001: Аутентификация (измен.)
     • REQ-050: Integration с SMS (новое)
   - Сколько часов? 40 часов разработки
   - Что сломается? Может быть, старые API
   - Кто согласиться? PM, разработчики, тестеры

3️⃣ РЕШЕНИЕ:
   - ✓ APPROVED: Добавляем в требования
   - ✗ REJECTED: Откладываем на версию 2.0
   - 🟡 CONDITIONAL: Добавляем если вместится в спринт

4️⃣ РЕАЛИЗАЦИЯ:
   - Обновить SRS (версия 1.3)
   - Создать новые User Stories в Jira
   - Обновить диаграммы
   - Обновить матрицу трассировки

5️⃣ КОММУНИКАЦИЯ:
   - Email команде: "Требования изменились по причине X"
   - Встреча с разработчиками: Объяснить изменения
   - Документ "Change Log" на wiki
```

## Конфликты и их разрешение

### Типичные конфликты

```
Конфликт 1: Бизнес хочет много, а времени мало
✓ Решение: Приоритизировать (MoSCoW: Must/Should/Could/Won't)

Конфликт 2: Требования от разных пользователей противоречивы
✓ Решение: Выяснить, кто Primary actor, его требования приоритет

Конфликт 3: Технический долг vs новые фичи
✓ Решение: Баланс 80/20 (80% новое, 20% техдолг)

Конфликт 4: Требование невозможно реализовать технически
✓ Решение: Работать с дизайнером на более простое решение

Конфликт 5: Требование очень дорогое
✓ Решение: Версионировать (MVP в v1, улучшения в v2)
```

## Типичные метрики управления требований

| Метрика | Нормально | Плохо |
|---|---|---|
| % требований утверждено | > 95% | < 80% (много неопределенности) |
| Время от выявления до утверждения | 2-3 недели | > 1 месяца (застой) |
| Изменения требований на проекте | 15-25% | > 50% (нестабильность) |
| % требований с критериями приемки | 100% | < 80% (проблемы с тестированием) |
| % требований прослеживаемо | 100% | < 90% (потеря требований) |
