# Нефункциональные требования (NFR)

| Грейд | Роль |
|---|---|
| 🔵 Middle | BA + SA |

> **NFR определяют качество**, а не функциональность. Если система медленная, небезопасная или ненадежная — она не нужна (ISO 25010).

## Определение и отличие от функциональных требований

| Аспект | Функциональное | Нефункциональное |
|---|---|---|
| **Вопрос** | Что система должна делать? | Как хорошо система это делает? |
| **Пример** | Пользователь регистрируется | Регистрация занимает < 2 минут |
| **Измеримость** | Да/Нет | Числовые метрики |
| **Критичность** | Видно пользователю | Может быть незаметно, но стоит дорого |

### Почему NFR упускают?

1. **Невидимость** — когда система работает быстро, пользователь не замечает оптимизации
2. **Сложность** — требуют глубокого технического понимания
3. **Дороговизна** — улучшить производительность/безопасность дорого
4. **Конфликты** — требования конкурируют (безопасность замедляет систему)
5. **Недооценка бизнесом** — "это может подождать"

### Последствия игнорирования NFR

- **Production outages** — система не выдерживает нагрузку
- **Потеря пользователей** — медленная работа отгоняет клиентов
- **Кибератаки** — слабая безопасность подвергает риску компанию
- **Невозможность масштабироваться** — архитектура не поддерживает рост
- **Штрафы** — нарушение SLA, потеря репутации

## ISO 25010: Восемь характеристик качества

### 1. Производительность (Performance)

**Определение:** Степень, в которой система выполняет функции в приемлемое время с приемлемым использованием ресурсов.

#### Ключевые метрики

**Время отклика (Response Time)**
```
Требование: Поиск в каталоге должен вернуть результаты за < 500 мс

Нормативы:
- 0-200 мс:    идеальное (пользователь не чувствует задержку)
- 200-1000 мс: приемлемое (заметная, но не раздражающая)
- > 1000 мс:   неудовлетворительное (пользователь теряет терпение)

Пример для веб-приложения:
- GET /api/products?search=laptop:      < 100 мс на p99
- POST /api/orders:                     < 500 мс на p95
- Загрузка главной страницы:            < 2 сек на 3G
```

**Пропускная способность (Throughput)**
```
Требование: Система должна обрабатывать 1000 req/sec

Единицы:
- запросы в секунду (req/sec)
- транзакции в секунду (tps)
- операции в секунду (ops/sec)

Пример:
- Payment API: 500 платежей/сек при пике
- Upload Service: 100 файлов/сек
```

**Утилизация ресурсов (Resource Utilization)**
```
Требование: CPU < 75%, Memory < 80% при нормальной нагрузке

Примеры:
- CPU average: 45-65%
- Memory: < 75%
- Disk I/O: < 70%
- Network bandwidth: < 80% capacity
```

### 2. Масштабируемость (Scalability)

**Определение:** Способность системы увеличивать пропускную способность при расширении ресурсов.

#### Виды масштабирования

**Вертикальное (Scale Up):**
```
Добавление мощности одному серверу (больше CPU, RAM, SSD)

✓ Проще реализовать (нет синхронизации)
✓ Нет задержек между узлами
✗ Ограничено мощностью одного оборудования
✗ Дорого (новый сервер = простой)
✗ Точка отказа (падение = все падает)

Требование:
"При увеличении пользователей в 2x,
нужно добавить 1 сервер"
```

**Горизонтальное (Scale Out):**
```
Добавление дополнительных серверов (балансировщик нагрузки + кластер)

✓ Теоретически неограниченное расширение
✓ Лучше для облака (добавляем серверы по мере необходимости)
✓ Высокая доступность (падение одного = остальные работают)
✗ Сложность синхронизации между узлами
✗ Консистентность данных (кеш, сессии)
✗ Сетевые задержки

Требование:
"Система поддерживает горизонтальное масштабирование
на Kubernetes кластере. При увеличении нагрузки в 10x
производительность снижается не более чем на 20%"
```

### 3. Доступность (Availability)

**Определение:** Доля времени, в течение которого система работает без сбоев.

#### Метрики uptime

```
99.0%  = 3.65 дня простоя в год     (бытовые системы)
99.5%  = 1.83 дня простоя в год     (обычные сервисы)
99.9%  = 43 минуты простоя в год    (критичные системы)
99.99% = 4.3 минуты простоя в год   (банки, авиация)

Требование: 99.9% uptime
Максимум простоя = 365 * 24 * 60 * 0.1 / 100 = 52.6 минуты в год
```

#### Надежность (Reliability)

```
MTBF (Mean Time Between Failures) — среднее время между сбоями
Требование: MTBF > 720 часов (месяц без сбоев)

MTTR (Mean Time To Recovery) — среднее время восстановления
Требование: MTTR < 5 минут (система восстанавливается быстро)
```

### 4. Безопасность (Security)

**Определение:** Защита системы от несанкционированного доступа и утечек данных.

```
Требования по ГОСТ/ISO 27001:

Аутентификация:
- Двухфакторная аутентификация для критичных операций
- Пароли минимум 12 символов (буквы, цифры, спецсимволы)
- Сессии истекают через 30 минут неактивности

Авторизация:
- Пользователи видят только свои данные
- Администраторы имеют отдельный аккаунт для админ-функций
- Изменение доступов логируется и одобряется

Шифрование:
- HTTPS для всех каналов (TLS 1.3+)
- Данные в БД зашифрованы (AES-256)
- Резервные копии зашифрованы

Аудит:
- Все действия с данными логируются
- Логи хранятся 1 год в неизменяемом виде
- Еженедельное сканирование на уязвимости
```

### 5. Удобство использования (Usability)

**Определение:** Простота и интуитивность использования системы.

```
Метрики:
- Новый пользователь выполняет базовую операцию без помощи за < 5 минут
- 95% пользователей находят нужную функцию за < 3 клика
- Время обучения оператора: < 4 часа
- Количество ошибок пользователей: < 1% операций

Требования:
- Шрифт не менее 14px
- Контраст текста 4.5:1 (WCAG AA)
- Интерфейс поддерживает горячие клавиши (Ctrl+S, Ctrl+P)
- Все поля иеют подсказки (tooltip, placeholder, help text)
```

### 6. Надежность (Reliability)

**Определение:** Способность системы функционировать при сбоях и восстанавливаться.

```
Требования:
- Система продолжает работать при потере одного сервера
- При сбое БД система переходит на резервную копию за < 1 мин
- Все данные синхронизированы между 3 дата-центрами
- Есть план восстановления (RTO < 4 часов, RPO < 1 часа)

RTO (Recovery Time Objective) — максимальное приемлемое время восстановления
RPO (Recovery Point Objective) — максимальный приемлемый объем потери данных
```

### 7. Совместимость (Compatibility)

**Определение:** Способность работать с другими системами и стандартами.

```
Требования:
- Поддержка браузеров: Chrome, Firefox, Safari, Edge (последние 2 версии)
- Мобильная поддержка: iOS 14+, Android 10+
- API совместим с REST/JSON стандартами
- Поддержка import/export в CSV, Excel, JSON
- Соответствие ГОСТ Р 52653-2006 (электронный документооборот)
```

### 8. Сопровождаемость (Maintainability)

**Определение:** Простота внесения изменений и исправления багов.

```
Требования:
- Код покрыт юнит-тестами на 80%+
- Документация актуальна на 100%
- Любой баг исправляется за < 2 дней
- Новая фича добавляется за < 1 неделю
- Архитектура позволяет изменять компоненты независимо
```

## Практический пример NFR для e-commerce

```
Функциональное требование:
FR-001: Пользователь может добавить товар в корзину

Нефункциональные требования:

NFR-Performance:
- Добавление в корзину: < 100 мс
- Обновление цены корзины: < 50 мс
- Загрузка страницы товара: < 1 сек

NFR-Scalability:
- 100,000 одновременных пользователей
- 10,000 добавлений в корзину в секунду (пик)

NFR-Availability:
- 99.99% uptime
- Система работает 24/7, включая праздники

NFR-Security:
- Номер карты не хранится в памяти
- Платежи через PCI-DSS compliant шлюз
- HTTPS для всех операций

NFR-Usability:
- Кнопка "В корзину" видна без скролла
- Подтверждение добавления за < 2 сек
- Корзина доступна на всех страницах
```

## Типовые NFR для разных типов систем

| Система | Критичные NFR |
|---|---|
| **E-commerce** | Performance, Security, Availability |
| **Банк** | Security, Reliability, Compliance |
| **CMS** | Usability, Maintainability, Performance |
| **IoT** | Reliability, Power consumption, Network latency |
| **Analytics** | Scalability, Data consistency, Performance |
